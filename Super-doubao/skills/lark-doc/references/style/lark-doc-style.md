# 文档样式指南

创建或编辑文档时，必须遵循本指南，使用结构化 block 提升可读性和视觉层次。

## 一、核心原则

> 以下为**默认建议**，按文档体裁灵活取舍：信息型 / 方案型 / 汇报型可充分采用；议论、随笔、应用文、叙事等以文字为主的体裁不强求。

1. **结构优于文字**：**有结构化信息时**用结构化 block 表达；纯论证、叙事、抒情内容用纯文本即可，不视为缺陷
2. **Front-load 结论**：信息型 / 方案型文档建议以 `<callout>` 开头概括核心结论；其他体裁按需，不强制
3. **视觉节奏**：长段落适当拆分；不同主题章节间可用 `<hr/>` 分隔（不设连续纯文本段落数硬上限）
4. **最少惊讶**：同一文档内同类信息保持一致即可，不要求全篇统一、更不要求跨文档统一风格
5. **重要信息画板化**：核心流程、架构、对比、风险、路线图、指标趋势等信息，**在结构确实适合图形表达时**优先使用画板

## 二、元素选择指南（可选建议，非强制映射）

> 下表是**按内容特征给出的可选建议**，不是"内容类型 → 组件"的一对一硬映射。同类内容不必每次都用同一组件；信息本身不具结构化特征时，纯文本同样是合理选择。按当前文档的实际需要挑选即可。

> ### ⛔ 富 block 使用红线（优先级高于下表映射）
> 下表是"可选菜单"，不是"默认配方"。套用前先过以下红线，**纯文本永远是默认选项**：
>
> 1. **callout 红线**：`<callout>` 只用于"必须让读者一眼看到"的**一句话**关键警示、关键数据或安全提示。**禁止**用 callout 包裹：结论段、总结段、解读段、过渡段、普通陈述段。**全文 callout 建议 ≤ 2 个，宁缺毋滥。**
> 2. **结尾红线**：正文写完**自然收尾**。**禁止**在文末追加"总结 / 升华 / 点睛 / 温馨提示 / 潜台词解读 / 核心结论回顾"之类的 callout 或表格——这是最常见的同质化套路。
> 3. **table 红线**：`<table>` 仅用于**≥3 个属性维度、需要横向对照**的结构化数据。叙述性、流程性、解读性、清单性内容用**文字或有序/无序列表**，不要塞进表格凑结构。
> 4. **grid 红线**：`<grid>` 仅用于真正的"左右并置对比"。普通并列信息用列表即可。
>
> 一句话：**先问"这段不加 block 是否照样清楚"，若是，就不加。** block 是为信息服务，不是为"看起来丰富"服务。

涉及图表需求时，按类型选择插入方式：思维导图/时序图/类图/饼图/甘特图用 `<whiteboard type="mermaid">` 直接内嵌；其他新图表启动 SubAgent 插入 `<whiteboard type="svg">完整 SVG</whiteboard>`；只有编辑**已有**画板时才调用 **lark-whiteboard** skill。

| 场景                                         | 推荐方案                                  |
|--------------------------------------------|---------------------------------------|
| 核心结论 / 摘要 / 注意事项                           | `<callout>` + emoji + 背景色             |
| 重要方案对比 / 优劣势 / Before vs After             | `<grid>` 2 列分栏；SVG SubAgent           |
| 简短低风险对比                                    | `<grid>` 2 列分栏                        |
| 3+ 属性的结构化数据 / 指标表                          | `<table>` + 表头背景色                     |
| 任务清单 / 检查项                                 | `<checkbox>`                          |
| 代码片段                                       | `<pre lang="x" caption="说明">`         |
| 引用 / 公式                                    | `<blockquote>` / `<latex>`            |
| 操作入口 / 跳转链接                                | `<button>` / `<a type="url-preview">` |
| 流程图 / 时间线 / 示意图 / 自定义图形 / 架构图 / 数据图 /思维导图等 | 画板图表                                  |


### 画板意图识别

下表的「信号词」只用于**识别候选**，不是出图的触发条件。是否真的画板化要**同时满足两个条件**：①**重要度**——该内容是章节核心结论 / 关键决策 / 主要论据；②**适配度**——其信息结构确实更适合图形表达（多节点关系、流程、趋势、占比等）。两者**同时成立**才出图；仅仅命中信号词、但本质是文字论证或一次性说明的，保持文字：

| 内容特征 | 信号词 / 模式 | 推荐画板类型 |
|-|-|-|
| 多步骤的操作流程或决策路径 | "先…然后…最后"、"步骤 1/2/3"、"如果…则…否则" | 流程图 / 泳道图 |
| 系统或模块间的依赖与交互 | "调用"、"依赖"、"上游/下游"、"请求→响应" | 架构图 |
| 上下级或从属关系 | "汇报给"、"下属"、"隶属"、"团队结构" | 组织架构图 |
| 时间线或阶段演进 | "Q1/Q2"、"里程碑"、"阶段一→阶段二"、日期序列 | 时间线 / 里程碑 |
| 因果分析或问题归因 | "根因"、"原因"、"导致"、"影响因素" | 鱼骨图 |
| 两个及以上方案/对象的多维度对比 | "vs"、"方案 A/B"、"优劣"、"对比" | 对比图 |
| 层级递进或优先级排序 | "基础→进阶→高级"、"L1/L2/L3"、"核心→外围" | 金字塔图 |
| 数值趋势或周期变化 | 带数字的时间序列、"增长/下降"、百分比变化 | 折线图 / 柱状图 |
| 漏斗或转化率 | "转化率"、"漏斗"、"从…到…留存" | 漏斗图 |
| 发散或归纳的思维结构 | "要点"、"维度"、"分支"、多层嵌套列表 | 思维导图 |
| 循环或飞轮效应 | "正循环"、"飞轮"、"闭环"、"A 驱动 B 驱动 C" | 飞轮图 |
| 占比分布 | "占比"、"份额"、"分布"、百分比加总 ≈100% | 饼图 / 树状图 |

**判断规则：**
- 同时满足「重要度 + 适配度」时才图示；不要仅因命中信号词就把文字论证强行图形化
- 重要度低、或信息用文字 / 表格已足够清晰的，保持 `<table>` / `<grid>` / `<callout>` 或纯文本即可
- 确定需要插入哪些图表后，参照 [lark-doc-whiteboard.md](../lark-doc-whiteboard.md) 中的方式，插入图表画板。

## 三、颜色语义

下表为**可选参考配色**。在同一文档内，同一语义建议保持一致即可；不强制跨文档统一这套配色，也不要求每篇都用满所有语义色：

| 语义 | emoji 前缀 | callout 背景色 | 文字色 |
|-|-|-|-|
| 信息、说明 | ℹ️ "说明：" | `light-blue` | `blue` |
| 成功、推荐 | ✅ "推荐：" | `light-green` | `green` |
| 警告 / 错误 / 风险 | ⚠️❌ | `light-red` | `red` |
| 注意、待确认 | ❗"注意：" | `light-yellow` | `yellow` |
| 中性、辅助 | — | `light-gray` | — |

- 表头统一 `background-color="light-gray"`
- 关键指标用 `<span text-color="green/red">` 突出，**必须同时用 ↑↓ 或 +/- 标注方向**（色觉无障碍）

## 四、排版规范

- 标题层级 ≤ 4 层，段落单段 ≤ 5 行，列表嵌套 ≤ 2 层，Grid ≤ 3 列（版式上限，防止过深过宽）
- 信息型 / 方案型文档建议开头用 `<callout>` front-load 结论；其他体裁按需，不强制

## 五、丰富度自检（按体裁判断，避免过度结构化）

生成内容后做一次**轻量自检**：看是否有"该结构化却写成大段纯文本"的信息，以及是否有"为凑富 block 而硬加的无效图表"。以下为**参考区间，不是必须命中的硬门槛**——是否引入富 block，先按体裁判断：

- **信息型 / 方案型 / 汇报型**（说明、方案、评审、数据汇报等）：可参考下表上限，适度结构化。
- **议论 / 随笔 / 应用文 / 叙事**（观点文、感想、致辞、故事等）：以文字论证为主，可几乎不用富 block，**不视为不达标**。

| 指标 | 参考标准（上限封顶 + 按需触发） |
|-|-|
| 富 block 密度 | **不设下限**；信息型文档一般 ≤ ~60% 为宜，过高会淹没文字 |
| 元素多样性 | 内容确有多种结构需求时再用多种 block，不为凑数硬上 |
| callout 数量 | **全文 ≤ 2 个**；只在关键警示/数据处用，不包裹结论段、总结段 |
| 结尾形态 | **不得以"总结/升华/解读"类 callout 或表格收尾**，正文自然结束 |
| 视觉节奏 | 长文档不同主题章节间可用 `<hr/>` 分隔（非强制） |

> 触发原则：**只有当内容本身存在结构化信息（数据、对比、流程、风险链路、趋势等）时**，才引入对应富 block。纯论证、叙事、抒情段落保持纯文本即可——不强制开头 callout、不强制每节配块、不限制连续段落数。
>
> **自检追问（生成后逐条问自己）**：① 有没有把普通段落硬包成 callout？② 结尾是不是又挂了一个"总结/点睛"块？③ 有没有为凑结构把叙述塞进表格？④ 同一标题下、同级、同类信息的列表，有没有混用有序(1.2.3.)和无序(•)？任一为"是" → 删掉/统一，改回应有形态。
>
> **列表类型一致性（同层级）**：同级且同类信息的列表项，必须统一用有序或无序，不混用。有序(1.2.3.)仅用于有**先后/步骤/排名/数量**的内容；并列、无先后的条目用无序(•)。生成后扫描相邻同级小节，若列表类型不一致而内容同质 → 统一。<!-- doubao-extra: 列表一致性 -->
