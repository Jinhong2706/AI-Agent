# 断点恢复说明

## 进度状态记录

本技能执行过程中维护的进度状态，用于异常中断后恢复：

### 状态文件：`recovery-state.json`

```json
{
  "last_action": "add",
  "last_room_id": "12345",
  "last_nickname": "张大仙",
  "completed_steps": ["parse_intent", "validate_room", "save_to_list"],
  "pending_steps": ["notify_user"],
  "last_checkpoint": "2026-06-20 20:10:00"
}
```

---

## 断点场景与恢复策略

### 场景1：添加主播时中断

**中断点：** 已验证房间号，未保存到文件

**恢复策略：**
1. 检查 `recovery-state.json`
2. 若 `last_action=add` 且 `save_to_list` 未完成
3. 重新执行保存逻辑
4. 跳过已完成的验证步骤

---

### 场景2：批量添加时中断

**中断点：** 已添加2个，第3个失败

**恢复策略：**
1. 读取 `monitored_rooms.json` 确认已添加的
2. 从第3个继续
3. 跳过已存在的（去重）

---

### 场景3：轮询过程中中断

**中断点：** 正在检查第5个主播时失败

**恢复策略：**
1. 读取 `last_status.json` 确认已检查的
2. 从中断处继续
3. 不重复触发已处理的提醒

---

## 恢复触发条件

满足以下任一条件时触发恢复检查：

1. 技能启动时检测到 `recovery-state.json` 存在
2. 用户说"继续"、"恢复"、"刚才怎么了"
3. 检测到上次执行未完成（超时/异常退出）

---

## 恢复流程

```
检测 recovery-state.json 是否存在
   ↓
读取 last_action 和 completed_steps
   ↓
询问用户是否恢复：
  "检测到上次操作未完成，是否继续？"
   ↓
用户确认 → 从中断步骤继续
用户拒绝 → 清除状态文件，重新开始
```

---

## 状态清理

以下情况自动清除 `recovery-state.json`：

- [ ] 所有步骤完成
- [ ] 用户明确说"取消"、"不用了"
- [ ] 超过24小时未恢复（视为过期）
- [ ] 检测到 `monitored_rooms.json` 数据完整

---

## 手动恢复命令

用户可手动触发恢复：

```
恢复上次操作
继续刚才的任务
```

---

## 恢复限制

- 最大恢复尝试次数：**2次**
- 超过2次仍失败 → 放弃恢复，提示用户重新开始
- 恢复时不重复触发已发送的提醒
- 恢复时不重复添加已存在的主播

---

## 示例

### 示例1：正常恢复
```
[上次执行中断]
用户：继续
Agent：好的，恢复上次操作...
✅ 已完成：验证房间号
⏳ 继续执行：保存到监控列表
✅ 恢复完成！已添加监控：张大仙（12345）
```

### 示例2：恢复失败
```
[上次执行中断，状态文件损坏]
用户：继续
Agent：⚠️ 恢复状态异常，无法继续
建议：重新开始操作
```

### 示例3：状态过期
```
[24小时前中断]
用户：继续
Agent：检测到上次操作已超过24小时，状态已过期
建议：重新开始操作
[自动清除 recovery-state.json]
```
