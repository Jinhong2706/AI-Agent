# 错误处理说明

## 错误分级

| 等级 | 场景 | 处理 | 是否自动修复 |
|------|------|------|------------|
| E1 | 文件缺失、格式错误 | 自动创建/修复 | ✅ |
| E2 | 昵称模糊、数据不完整 | 用合理假设，标注待确认 | ✅ |
| E3 | 房间号冲突、用户需决策 | 暂停，给出选项 | ❌ |
| E4 | API失败、权限不足 | 停止，说明原因和替代方案 | ❌ |
| E5 | 非法请求、安全违规 | 拒绝，引导安全替代方案 | ❌ |

---

## 常见错误场景与处理

### E1：可自动修复的错误

#### 场景1：`monitored_rooms.json` 不存在
**错误提示：**
```
⚠️ 监控列表文件不存在，已自动创建
```

**修复动作：**
```bash
echo '{"rooms": []}' > monitored_rooms.json
```

---

#### 场景2：`last_status.json` 格式错误
**错误提示：**
```
⚠️ 状态文件格式错误，已重置
```

**修复动作：**
```bash
echo '{}' > last_status.json
```

---

#### 场景3：房间号格式错误（含字母/特殊字符）
**错误提示：**
```
❌ 房间号应为纯数字，已自动过滤
原输入：abc123
过滤后：123
```

**修复动作：**
从输入中提取纯数字部分作为房间号

---

### E2：可降级完成的错误

#### 场景1：昵称无法精确匹配房间号
**错误提示：**
```
⚠️ 未找到"{{nickname}}"的精确匹配
可能的匹配结果：
1. 张大仙（12345）
2. 张大仙爱吃梨（23456）
请选择序号，或提供房间号
```

**降级方案：**
列出模糊匹配结果让用户选择

---

#### 场景2：API返回数据不完整
**错误提示：**
```
⚠️ 部分数据获取失败，显示可用信息
```

**降级方案：**
仅显示能获取到的字段，跳过缺失字段

---

### E3：需要用户决策的错误

#### 场景1：房间号同时匹配多个昵称
**错误提示：**
```
❓ 房间号 12345 匹配到多个主播：
1. 张大仙
2. 张大仙的直播间
请选择要监控的主播序号（或输入"全部"）
```

**等待用户选择后再继续**

---

#### 场景2：用户要求覆盖已有监控
**错误提示：**
```
❓ 张大仙（12345）已在监控列表中
是否覆盖？（输入"确认"继续）
```

**等待用户确认后再继续**

---

### E4：不可执行的错误

#### 场景1：斗鱼API请求失败
**错误提示：**
```
❌ 查询失败

可能原因：
• 网络连接问题
• 斗鱼API暂时不可用
• 请求频率过高被限流

建议：
• 检查网络连接
• 稍后重试（建议间隔 ≥ 60秒）
• 确认房间号是否正确
```

**停止执行，不触发提醒**

---

#### 场景2：房间号不存在（404）
**错误提示：**
```
❌ 房间号 99999 不存在

可能原因：
• 房间号输入错误
• 该主播已改名/换房间
• 该房间已被封禁

建议：
• 前往斗鱼官网确认房间号
• 尝试用主播昵称搜索
```

**停止执行**

---

#### 场景3：文件权限不足
**错误提示：**
```
❌ 无法写入监控列表

可能原因：
• 文件权限不足
• 磁盘空间不足

建议：
• 检查文件权限
• 清理磁盘空间后重试
```

**停止执行**

---

### E5：不合规的错误

#### 场景1：用户要求监控非斗鱼平台主播
**错误提示：**
```
❌ 当前仅支持斗鱼平台的主播监控

你提到的主播似乎在 {{其他平台}} 直播
建议：等待后续版本支持其他平台 😊
```

**拒绝执行**

---

#### 场景2：用户要求短时间内高频轮询（<10秒）
**错误提示：**
```
❌ 轮询间隔过短，可能触发API限流

建议：
• 最小间隔：30秒
• 推荐间隔：60秒
```

**拒绝过短间隔，使用最小值替代**

---

## 错误日志格式

所有错误应记录到 `error.log`（可选）：
```
[2026-06-20 20:00:00] [E4] API请求失败 - room_id=12345, error=timeout
[2026-06-20 20:05:00] [E1] 监控列表文件不存在，已自动创建
[2026-06-20 20:10:00] [E3] 房间号匹配多个主播，等待用户选择
```

---

## 用户友好提示原则

1. **明确**：告诉用户发生了什么
2. **具体**：给出具体原因，不只是"出错了"
3. **建设性**：提供解决建议或替代方案
4. **不过度技术化**：不用API、JSON等技术术语
5. **有行动指引**：告诉用户接下来该做什么
