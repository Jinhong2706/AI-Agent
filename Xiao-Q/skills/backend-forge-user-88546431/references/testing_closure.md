# Testing Closure

测试闭环是 Backend Forge 的完成依据。

## 通用规则

1. 每个实现任务必须有测试策略。
2. 每个关键路径必须有可运行验证。
3. 无法运行测试时，必须说明阻塞原因和降级验证。
4. 不得用“应该可以”代替测试结果。
5. 测试失败不得宣称完成。

## 新项目

最低覆盖：

- 核心 API 正常路径。
- 数据持久化。
- 输入校验失败。
- 资源不存在。
- 至少一个异常路径。

## API/功能开发

最低覆盖：

- Service/业务单元测试。
- Controller/Router 接口测试。
- 关键错误路径。
- 数据库读写验证。

## 数据模型与迁移

最低覆盖：

- 迁移能执行。
- ORM model/entity 与表结构一致。
- 关键查询能运行。
- 新约束有正反向测试。
- 兼容风险已说明。

## 认证鉴权

最低覆盖：

- 未登录。
- 权限不足。
- 越权访问。
- 正常授权。
- Token/session 过期或无效，按项目能力覆盖。

## Bug 修复

优先顺序：

1. 写出失败测试或复现脚本。
2. 确认失败对应用户问题。
3. 修复。
4. 确认失败测试转绿。
5. 跑相关回归。

无法复现时，必须说明已尝试路径和剩余假设。

## 性能/并发

最低覆盖：

- 优化前基线。
- 优化后对比。
- 数据规模或并发条件说明。
- 不牺牲正确性的验证。

## 降级验证

允许降级的情况：

- 项目没有测试框架。
- 依赖无法安装。
- 外部服务不可用。
- 测试环境需要用户凭据。
- 运行成本超出本轮任务。

降级输出必须包含：

- 原计划测试。
- 实际完成验证。
- 未完成原因。
- 测试债务。
- 后续补齐建议。
