# Quality Gates

质量门禁用于判断是否可以进入实现、是否可以进入收口。

## 实现前门禁

必须明确：

- 目标。
- 范围。
- 非目标。
- 架构决策。
- 数据影响。
- 权限影响。
- 测试闭环。
- 风险和回滚。

轻量任务可以压缩表达，但不能跳过实质判断。

## 实现中门禁

发现以下情况必须停下：

- 需要架构决策但尚未形成架构简报或 ADR。
- 需要新增数据库迁移但计划未包含。
- 需要改变权限模型。
- 需要删除或重写数据。
- 需要引入缓存、队列、异步任务。
- 影响生产配置。
- 触及架构会签项。
- 现有测试失败且无法判断是否与本次改动相关。

## 收口门禁

收口必须包含：

- 修改内容。
- 实际验证命令和结果。
- 是否满足验收。
- 架构假设是否已验证，若适用。
- 降级验证和测试债务，若存在。
- 残余风险。

## 代码质量

后端实现应优先保证：

- 业务语义清晰。
- 分层合理。
- 输入校验明确。
- 错误语义稳定。
- 事务边界清楚。
- 数据访问可测试。
- 日志有排障价值。
- 测试覆盖关键路径。

## 解释机制

每轮最多输出 3 条后端要点，只解释影响以下维度的取舍：

- 正确性。
- 可维护性。
- 测试。
- 性能。
- 安全。
