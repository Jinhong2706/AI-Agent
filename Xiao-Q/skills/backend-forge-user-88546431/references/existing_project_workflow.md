# Existing Project Workflow

已有项目流程用于 Spring Boot、FastAPI、Django 以及可有限介入的其他后端项目。

## S0 项目扫描

先读：

- 依赖文件：`pom.xml`、`build.gradle`、`requirements.txt`、`pyproject.toml`、`Pipfile`。
- 入口文件：Spring Boot application class、FastAPI app、Django `manage.py` 和 settings。
- 目录结构。
- 测试目录和测试命令。
- 数据库配置和迁移目录。
- README、AGENTS、项目规范。

扫描产出：

- 框架和版本线索。
- 数据库和迁移工具。
- 测试框架与运行命令。
- 分层结构。
- 当前任务可能影响的模块。

## S1 需求确认

必须确认：

- 要改什么。
- 不改什么。
- API 输入输出。
- 数据影响。
- 权限边界。
- 验收标准。

若用户只给报错，先进入排查路径，不直接写修复。

## S2 方案确认

输出改动契约：

- 涉及文件/模块。
- 业务逻辑变更。
- 数据库/迁移影响。
- 事务和一致性影响。
- 权限影响。
- 测试闭环。
- 回滚或兼容风险。

标准 `bf-` 下，改动契约未确认不得实现。`bf-a` 可采用安全默认，但高风险必须停下。

## S3 子单元冻结

仅大任务需要。拆分维度：

- API 契约。
- 数据迁移。
- 业务逻辑。
- 鉴权。
- 测试。
- 配置。

每个子单元必须有验证方式。

## S4 实现

遵守现有项目风格：

- 不替换框架。
- 不擅自换 ORM。
- 不擅自换迁移工具。
- 不做无关重构。
- 不绕过现有测试工具。

## S5 验证

按任务选择：

- 单元测试。
- 集成/接口测试。
- 迁移验证。
- 鉴权测试。
- 回归测试。
- 性能/并发验证。

测试失败时优先修复本次引入的问题。若失败来自既有问题，必须说明证据。

## S6 收口

输出：

- 做了什么。
- 测试/验证结果。
- 是否符合验收。
- 残余风险。
- 本轮后端要点，最多 3 条。
