# New Project Cocreation

新项目共创用于空目录或从 0 到 1 创建 Java/Python 单服务后端。第一版重点是可运行、可测试、可维护，不追求架构平台化。

## 默认交付

- 可运行单服务。
- 真实业务 API。
- 数据库模型和迁移/建表方案。
- 单元测试和接口/集成测试。
- 基础异常处理、日志、配置分层。
- 本地启动、测试、配置说明。

## 非默认项

以下能力按需求触发，不进入默认骨架：

- Docker。
- CI/CD。
- 消息队列。
- 缓存。
- 认证鉴权。
- 复杂后台管理。

## C0 目标收口

必须确认：

- 服务要解决的问题。
- 核心资源名词。
- 首批 API。
- 是否有用户/角色/权限。
- 数据是否需要持久化。
- 明确不做什么。

推荐问法：一次只问当前最大分叉。例如先确认业务资源，再追 API 和数据关系。

## C1 技术定型

默认方案：

- Java：Spring Boot。
- Python：FastAPI。
- 正式数据库：PostgreSQL。
- Spring Boot 快速验证：H2。
- FastAPI 快速验证：SQLite。
- Spring Boot 迁移：Flyway 或 Liquibase，优先 Flyway，除非用户或项目约束更适合 Liquibase。
- FastAPI 迁移：Alembic。

需要确认：

- Java 还是 Python。
- 是否正式使用 PostgreSQL。
- 是否需要认证鉴权。
- 本地运行方式和包管理工具。

## C2 领域与数据设计

产出：

- 实体和字段。
- 实体关系。
- 唯一约束和索引。
- 必填/可选规则。
- 事务边界。
- 错误语义。

必须检查：

- 是否有软删除或状态流转。
- 是否有并发写入风险。
- 是否有越权访问风险。
- 是否有审计字段需求。

## C3 实现范围冻结

冻结内容：

- 目录结构。
- API 契约。
- 数据模型。
- 迁移计划。
- 测试清单。
- 本地命令。

C3 前禁止写实现代码。

## C4 实现与测试闭环

实现顺序：

1. 项目骨架和依赖。
2. 配置分层。
3. 数据模型和迁移。
4. Repository/DAO。
5. Service。
6. Controller/Router。
7. 异常处理。
8. 测试。
9. 本地说明。

收口前必须运行测试。无法运行时，必须说明阻塞原因、已完成的静态检查和测试债务。
