# Task Routing

Backend Forge 先判断项目状态，再判断执行策略，最后进入任务模式。

## 项目状态

| 状态 | 识别依据 | 动作 |
|---|---|---|
| `empty_new` | 空目录或仅隐藏文件 | 进入新项目共创 |
| `medium_backend_architecture` | 用户要求独立负责中型后端架构、系统方案、模块边界、数据/权限/运行架构 | 进入架构负责人轨道 |
| `spring_boot_existing` | `pom.xml`/`build.gradle` + Spring Boot 依赖或启动类 | 进入已有项目流程 |
| `fastapi_existing` | FastAPI 依赖、`main.py`/`app/`、路由声明 | 进入已有项目流程 |
| `django_existing` | `manage.py`、Django settings、apps | 进入 Django 维护流程 |
| `backend_other` | 后端项目但非主线框架 | 先咨询，再决定是否有限介入 |
| `non_backend` | 前端、移动端、文档或其他 | 退出或转为通用协助 |

脚本缺失时，使用文件扫描和依赖文件手动判断。

## 执行策略

- 标准 `bf-`：需求和方案确认后执行。
- 快速 `bf-fast`：轻量优先，发现风险升级。
- 全自动 `bf-a` / `bf a`：采用安全默认方案，遇到高风险或架构会签项停下。

## 任务模式

### 后端架构负责人轨道

命中条件：

- 用户要求独立负责中型项目后端架构。
- 用户要求从前端负责人视角主导后端技术方案。
- 任务涉及模块划分、API 策略、数据所有权、权限模型、运行架构或关键依赖选择。
- 用户需要架构简报、ADR、风险清单或实现分片计划。

### 新项目共创

命中条件：

- 空目录。
- 用户要求从 0 到 1 新建后端服务。
- 用户要求先设计服务骨架、接口、数据库和测试。

### API/功能开发

命中条件：

- 新增或修改业务 API。
- 修改服务层业务逻辑。
- 跨 Controller/Router、Service、Repository 的完整闭环。

### 数据模型与迁移

命中条件：

- 新增/修改表、字段、索引、约束。
- 修改 ORM model/entity。
- 需要 Flyway、Liquibase、Alembic 或 Django migrations。

### 认证鉴权

命中条件：

- 登录、注册、会话、JWT、角色、权限、资源访问控制。
- 用户数据隔离或越权风险。

### Bug/生产问题排查

命中条件：

- 报错、异常日志、接口 500、数据错误、偶发并发问题。
- 用户明确说“排查”“修复”“定位”。

### 测试补强

命中条件：

- 用户要求补测试。
- 当前变更缺少测试闭环。
- 代码审查发现关键路径无测试。

### 性能/并发优化

命中条件：

- 慢查询、接口响应慢、锁竞争、重复请求、并发写入异常。
- 中型项目内的性能和并发基线、瓶颈定位、局部优化。

### 缓存/异步/队列

命中条件：

- Redis、Spring Cache、异步任务、定时任务、Celery、RabbitMQ、Kafka。
- 需要幂等、重试、失败处理、缓存失效策略。

### 配置/部署排查

命中条件：

- 本地启动失败、环境变量、数据库连接、打包运行、容器配置排查。
- 不默认扩展到完整 DevOps 平台建设。

### 代码审查/重构

命中条件：

- 用户要求 review、重构、整理结构。
- 默认代码审查先列风险，不直接改。

## 等待态

只问当前最关键问题。优先级：

1. 业务目标。
2. 架构负责人授权范围。
3. 数据影响。
4. 权限边界。
5. 验收方式。
6. 运行和测试环境。
