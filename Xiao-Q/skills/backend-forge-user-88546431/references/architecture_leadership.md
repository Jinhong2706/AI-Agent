# Architecture Leadership

架构负责人轨道用于让有经验的前端工程师独立承担中型项目后端架构负责人。目标不是替代企业级架构委员会，而是把中型项目后端架构拆成可决策、可验证、可交付的阶段。

配套参考：

- 架构选择先查 [architecture_decision_rules.md](architecture_decision_rules.md)。
- 前端视角输入先用 [frontend_to_backend_mental_model.md](frontend_to_backend_mental_model.md) 转成后端问题。

## 适用范围

适合：

- 单服务或少量服务的中型业务系统。
- 新项目后端架构从 0 到 1。
- 已有项目后端重整、模块划分、数据边界和权限模型梳理。
- 前端负责人需要主导 API、数据、权限、测试和交付节奏。

不直接覆盖：

- 组织级技术治理。
- 多团队平台化治理。
- 企业级 IAM/SSO 平台。
- 跨业务线数据治理。
- 平台级多租户权限模型。
- 大规模云平台或 DevOps 平台建设。

这些不是禁止讨论，而是必须形成架构会签项，不能由 `bf-a` 自动决定。

## A0 负责人上下文

必须确认：

- 业务目标和成功标准。
- 用户规模、数据规模和增长预期。
- 团队能力：前端、后端、测试、运维分别由谁承担。
- 交付期限和阶段目标。
- 风险承受度：可接受的技术债、性能余量和安全要求。
- 前端/后端接口协作方式。

输出：负责人上下文摘要。

## A1 架构范围

必须确定：

- 系统边界和非目标。
- 模块划分。
- API 风格：REST、GraphQL 或混合。
- 同步接口和异步任务边界。
- 外部依赖和第三方服务。
- 单服务、模块化单体或少量服务的选择理由。

输出：架构范围说明。

若涉及模块化单体 vs 微服务，必须引用架构决策规则库并产出 ADR。

## A2 数据架构

必须确定：

- 核心实体和关系。
- 数据所有权。
- 读写路径。
- 事务边界。
- 幂等策略。
- 索引、分页、归档和增长预期。
- 迁移、回滚和历史数据兼容策略。

输出：数据架构说明和迁移策略。

若涉及 PostgreSQL vs MySQL 或 Redis，必须引用架构决策规则库并说明默认选择、触发条件和回退方案。

## A3 安全与权限

必须确定：

- 认证方式。
- 授权模型。
- 资源归属规则。
- 401/403/404 的错误语义。
- 敏感字段和日志脱敏。
- 审计日志需求。
- 前端可见状态和后端真实权限的分离。

输出：权限模型和安全测试清单。

若涉及 JWT vs session，必须引用架构决策规则库并说明撤销、过期、存储和前端处理策略。

## A4 运行架构

必须确定：

- 本地、测试、生产环境差异。
- 配置和密钥来源。
- 日志、指标、错误追踪。
- 数据库连接池、超时、重试。
- 缓存、队列、异步任务是否作为系统级基础设施。
- 部署和回滚的最低可行方案。

输出：运行架构说明和环境清单。

若涉及同步接口 vs 异步任务，必须引用架构决策规则库并说明用户等待时间、任务状态、幂等和失败反馈。

## A5 决策冻结

冻结前必须产出：

- 架构简报。
- 关键 ADR。
- 风险清单。
- 回滚策略。
- 测试策略。
- 实现分片计划。

A5 前禁止大规模实现。若高风险选择没有 ADR，不得把它包装成“默认方案”继续执行。

## A6 分片交付

按以下维度拆分：

- API 契约。
- 数据模型和迁移。
- 认证鉴权。
- 业务逻辑。
- 外部依赖。
- 缓存/队列/异步。
- 测试和回归。

每个分片必须有输入、输出、验收和回滚判断。

## A7 架构验证

最低验证：

- API 契约验证。
- 数据迁移验证。
- 权限路径测试。
- 核心业务集成测试。
- 关键错误路径。
- 性能或并发基线，若有增长风险。
- 架构风险复盘。

收口输出必须说明哪些架构假设已经验证，哪些仍是风险或测试债务。
