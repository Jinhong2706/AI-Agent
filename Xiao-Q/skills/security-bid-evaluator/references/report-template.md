# 评估报告模板

本文件定义评估报告的完整 Markdown 结构、表格规范和证据引用格式。进入 SKILL.md 第 5 步（生成报告）前**必读**。

## 1. 整体结构

报告按以下章节组织（严格按此顺序）：

```
# <项目名>招投标技术指标评估报告
（报告头信息）
## 一、执行摘要
## 二、满足度统计
## 三、重大风险清单
## 四、指标逐条评估
## 五、附录
```

## 2. 每章详细模板

### 2.1 报告头

顶部固定信息，放在一级标题下、一级章节之前：

```markdown
# <项目名>招投标技术指标评估报告

| 项目 | 内容 |
|---|---|
| 项目名称 | XXX 项目 |
| 评估日期 | 2025-XX-XX |
| 评估对象 | <我方产品名 + 版本> |
| 招标指标来源 | `<文件路径或对话来源>` |
| 参考文档数量 | X 份（详见附录） |
| 指标总数 | X 条（含 ★ X 条、▲ X 条、必须项 X 条） |
| 整体响应度 | **XX%**（完全满足条数 / 总条数） |

---
```

### 2.2 第一章：执行摘要

4-6 句话，内部决策者一分钟能看明白：

```markdown
## 一、执行摘要

本次评估基于 <参考文档数量> 份产品参考材料，对招标文件中的 <指标总数> 条技术指标进行逐条符合性评估。

整体结论：**<定调：完全可以响应 / 大部分响应、少量风险 / 关键短板较多，建议审慎投标 / 存在废标风险，不建议直接投标>**。

关键发现：
- <1 句话：最强项 / 正偏离亮点>
- <1 句话：最大风险点（不满足的必须项或 ★ 项）>
- <1 句话：需要后续动作（补材料 / 找定制 / 确认商务）>
```

### 2.3 第二章：满足度统计

两张小表，展示总体分布和重点分布：

```markdown
## 二、满足度统计

### 2.1 按评定等级分布

| 评定等级 | 条数 | 占比 |
|---|---:|---:|
| 完全满足 | X | XX.X% |
| 基本满足 | X | XX.X% |
| 部分满足 | X | XX.X% |
| 不满足 | X | XX.X% |
| **合计** | **X** | **100%** |

### 2.2 按要求级别分布

| 要求级别 | 总数 | 完全满足 | 基本满足 | 部分满足 | 不满足 | 满足率* |
|---|---:|---:|---:|---:|---:|---:|
| 必须项 | X | X | X | X | X | XX% |
| ★ 重要项 | X | X | X | X | X | XX% |
| ▲ 加分项 | X | X | X | X | X | XX% |
| 一般项 | X | X | X | X | X | XX% |

\* 满足率 = (完全满足 + 基本满足) / 总数
```

### 2.4 第三章：重大风险清单

所有"不满足"和"必须/★ 项的部分满足"按严重度降序列出。这是决策层最先要看的部分：

```markdown
## 三、重大风险清单

以下条目可能影响投标结果，建议优先处理。严重度从高到低排列。

### 🔴 高风险（必须项 / ★ 项的"不满足"）

| 序号 | 指标摘要 | 评定 | 风险说明 | 建议动作 |
|---|---|---|---|---|
| 3.2 | 支持国密算法并通过商密认证 | 不满足 | 认证证书缺失，可能构成实质性不响应 | 立即向资质部门核实，或评估放弃该项 |
| 5.1 | 提供 7×24 原厂驻场服务（★） | 不满足 | 服务模式不匹配 | 商务沟通是否接受外包驻场 / 或调整应答策略 |

### 🟡 中风险（必须项 / ★ 项的"部分满足"，或一般项的"不满足"）

| 序号 | 指标摘要 | 评定 | 风险说明 | 建议动作 |
|---|---|---|---|---|
| 2.4 | 单节点处理能力 ≥ 1000Mbps | 部分满足 | 实测 750Mbps，差距 25% | 考虑集群应答 / 申请复测 |

### 🟢 低风险（加分项的"部分满足"或"不满足"）

（酌情列出，不关键的可省略）

> 如无重大风险，此章写："本次评估未发现高风险项。"
```

### 2.5 第四章：指标逐条评估（报告主体）

**两种展现方式二选一**，根据指标数量决定：

#### 方式 A：指标数量 ≤ 30 条 → 表格式

```markdown
## 四、指标逐条评估

> 证据引用格式：`[文档名 > 章节/页码]："原文摘录"`
> 评定等级使用以下图标：✅ 完全满足 🟢 基本满足 🟡 部分满足 🔴 不满足

| 序号 | 指标描述 | 级别 | 评定 | 证据 | 应答策略 |
|---|---|---|---|---|---|
| 1.1 | 支持分布式部署，单节点 ≥ 500Mbps | 必须 | ✅ | [产品手册 > §2.1]："支持分布式集群部署，单节点吞吐 1Gbps" | 完全响应，应答时突出"单节点吞吐超出要求 1 倍"的技术优势 |
| 1.2 | ... | ... | ... | ... | ... |
```

#### 方式 B：指标数量 > 30 条 或 每条信息较多 → 逐条分节

```markdown
## 四、指标逐条评估

> 证据引用格式：`[文档名 > 章节/页码]："原文摘录"`

### 指标 1.1 支持分布式部署，单节点处理能力不低于 500Mbps

- **类别**：性能
- **要求级别**：必须
- **重要标记**：★
- **关键要点**：
  1. 支持分布式部署
  2. 单节点处理能力 ≥ 500Mbps
- **评定**：✅ **完全满足**
- **证据**：
  - 要点 1：[产品手册 v3.2 > §2.1 部署模式]："产品支持单机、双机热备、分布式集群三种部署模式，集群规模最大支持 32 节点"
  - 要点 2：[性能测试报告 > §3.4]："单节点吞吐量实测 1.2Gbps（64 字节小包 800Mbps）"
- **应答策略**：完全响应（正偏离）。
  1. 应答文档中明确"完全满足"
  2. 援引性能测试报告第 3.4 节的实测数据
  3. 重点突出：单节点吞吐实测 1.2Gbps，超出招标要求 140%，作为技术优势点
  4. 作为应答附件提交：性能测试报告 PDF、集群部署白皮书

---

### 指标 1.2 ...
```

**选择哪种方式**：
- 指标 < 15 条 → 方式 A（表格）
- 指标 15-30 条 → 方式 A（表格）但证据/应答列可换行，留出空间
- 指标 > 30 条 → 方式 B（分节）
- 任意数量下，如果每条证据/应答都很长 → 方式 B

### 2.6 第五章：附录

```markdown
## 五、附录

### 5.1 参考文档清单

| 编号 | 文档名 | 类型 | 版本 | 主要用途 | 引用次数 |
|---|---|---|---|---|---:|
| D1 | 产品手册 | docx | v3.2 | 功能能力 | 12 |
| D2 | 性能测试报告 | pdf | 2025-03 | 性能指标 | 6 |
| D3 | ... | ... | ... | ... | ... |

### 5.2 未使用的参考文档

（如有提供但未引用到的文档，列在这里，便于用户核查是否漏看）

### 5.3 评定规则说明

- 采用四级评定：完全满足 / 基本满足 / 部分满足 / 不满足
- 判定详细规则见 `security-bid-evaluator` skill 的 `references/evaluation-rules.md`
- 量化指标严格按数值对比，不做"接近"之类的模糊处理
- 复合指标按最弱处定级

### 5.4 术语说明

（只列报告中实际用到的术语，避免堆砌）
- **必须项**：招标文件中使用"必须"、"应当"等措辞的条款
- **★ 项**：招标文件中明确标星的重要指标
- **正偏离 / 负偏离**：应答指标优于 / 低于招标要求，详见招投标通用术语
```

## 3. 表格规范

- 一律使用标准 Markdown 表格（`| ... | ... |`），不用 HTML `<table>`
- 列宽合理（摘要列 < 40 字，证据列 < 80 字；超出换行或进方式 B）
- 对齐约定：数字列右对齐（`---:`），文字列左对齐（`:---` 或默认）
- 表格前加一句引导文字，让读者知道表在讲什么

## 4. 证据引用格式

**标准格式**：

```
[文档名 > 章节或页码]："原文摘录，不超过 2 句话"
```

**示例**：

```
[产品手册 v3.2 > §2.1]："产品支持单机、双机热备、分布式集群三种部署模式"
[性能测试报告.pdf > p.15]："在 64 字节小包场景下实测吞吐量 800Mbps"
[白皮书.pdf > 章节未知, 关键词"国密"]："产品全面支持国密 SM2/SM3/SM4 算法"
```

**特殊情况**：

- 章节号找不到 → 用 `关键词"xxx"` 定位
- pdf 没页码 → 用相对位置 `p.X / 前 1/3 处 / 最后一节`
- 多条证据 → 用 bullet list 分列
- 证据超过 2 句话 → 只摘录最关键的一两句，加省略号；完整内容用户翻原文

**禁止**：
- 改写证据（必须原文）
- 融合多处证据成一句（会失去溯源）
- 用"参考文档中提到"这种不指名的写法

## 5. 语言风格

- 内部报告，务实客观，**不用应答文档那种修辞和营销口吻**
- 中文全角标点（，。：；），代码/命令/英文术语用反引号
- 中英文和数字之间加空格（"支持 1000 TPS" 不是"支持1000TPS"）
- 术语前后一致（一开始用"参考文档"就不要换"材料"、"资料"、"依据")
- 不滥用加粗，**仅强调评定结论、风险等级、关键数字**

## 6. 报告长度参考

- 执行摘要：150-300 字
- 满足度统计：两张表 + 少量说明文字
- 重大风险清单：按条目数决定，无重大风险时一句话带过
- 指标逐条：主体，占全文 60-80%
- 附录：简洁，不要塞大段废话

整份报告理想长度：5000-15000 字，超过 15000 字考虑是否有信息冗余或指标可合并。
