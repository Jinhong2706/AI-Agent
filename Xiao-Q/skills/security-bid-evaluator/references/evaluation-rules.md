# 评定规则与应答策略

本文件定义指标评估的判定标准和应答策略模板。进入 SKILL.md 第 4 步（逐条评估）前**必读**。

## 1. 四级评定规则

评定时按下列顺序逐条核查，满足前一条才考虑后一条。

### 1.1 完全满足

**判定条件**（全部成立）：
1. 参考文档中存在**直接证据**（直接点名指标所述功能/能力）
2. 证据明确表示"支持 / 具备 / 提供 / 满足"，**无任何模糊或条件性措辞**
3. 对于量化指标：参考文档给出的数值 **完全达到或优于**招标要求
4. 复合指标的每一个关键要点都有证据支撑

**典型证据形态**：
- 产品手册明确列出该功能
- 测试报告给出达标或超标的性能数据
- 资质证书原件截图 / 证书编号
- 官方发布的兼容性列表

**示例**：
> 招标要求：支持 SQL 注入、XSS、CSRF 等常见 Web 攻击防护
> 证据：产品手册 §3.2 "WAF 模块支持 SQL 注入、XSS、CSRF、命令注入等 12 类 OWASP Top 10 攻击防护"
> → **完全满足**（三类要求全部直接命中，且覆盖面更广）

### 1.2 基本满足

**判定条件**（满足任一条）：
1. 主要要求点都有直接证据，**少量次要要求点**只有间接证据或未明确提及
2. 量化指标**略低于要求**但差距在可接受范围（差距 < 10% 且非硬性门槛）
3. 能力存在但**实现路径或术语与招标要求略有差异**（比如要求"主备部署"，产品支持"双机热备"，实质等价）

**典型场景**：
- 招标列出 5 个功能点，产品支持其中 4 个明确命中，第 5 个在白皮书里用略不同的名词提及
- 性能要求 1000 TPS，产品测试报告显示 950 TPS
- 要求"支持 RADIUS 认证"，产品支持"RADIUS、LDAP、AD 等多种认证方式"但没单独强调 RADIUS 的细节

**示例**：
> 招标要求：告警信息支持通过邮件、短信、Syslog、SNMP Trap 四种方式推送
> 证据：产品手册 §5.4 "支持邮件、短信、Syslog 告警推送"；管理员手册配置章节提到 SNMP 功能
> → **基本满足**（3/4 功能明确命中，SNMP 推送从另一章节可推断支持但未明确写告警场景）

### 1.3 部分满足

**判定条件**（满足任一条）：
1. 只满足**部分要求点**，另一部分在参考文档中无证据或明确不满足
2. 量化指标**明显低于要求**（差距 10%-30%）但不至于完全不满足
3. 能力部分具备，但**关键限制使实际效果打折**（比如要求"实时阻断"，产品只支持"记录+延迟处置"）

**典型场景**：
- 复合指标 4 个要点，2 个完全满足、2 个不满足
- 性能要求 1000 TPS，产品能力 700 TPS
- 要求"支持国产化全栈适配"，产品只适配了部分国产 CPU 和操作系统

**示例**：
> 招标要求：支持国密 SM2、SM3、SM4 算法，并通过国家密码管理局商用密码认证
> 证据：产品手册 §7.1 "支持 SM2/SM3/SM4 算法"；未找到国密认证证书或认证编号
> → **部分满足**（算法支持已证明，但认证资质证据缺失）

### 1.4 不满足

**判定条件**（满足任一条）：
1. 参考文档中**完全找不到**相关证据
2. 证据**明确表明不能满足**（比如产品手册写"不支持 xxx"）
3. 量化指标差距 > 30% 或低于硬性门槛
4. 关键要求点缺失（比如招标明确 ★ 项未找到证据）

**关键纪律**：
- **不要因为"同类产品一般都支持"就判满足** —— 这是最容易出错的地方
- **不要因为"白皮书大方向提到了"就判部分满足** —— 没有具体证据就是不满足
- 宁可判"不满足"让用户去找补充材料，**不要**虚高评定误导投标决策

**示例**：
> 招标要求：提供 7×24 小时原厂驻场服务，驻场工程师不少于 2 人
> 证据：参考文档中只有标准服务承诺（8×5 远程支持），未提及驻场服务
> → **不满足**（无相关证据）

## 2. 证据强度分级

同一条指标可能找到多条证据，按强度排序取最强的作为主证据，其余作为佐证。

| 强度 | 类型 | 举例 |
|---|---|---|
| ★★★ 直接证据 | 官方出具的点名证明 | 产品手册列出该功能、第三方测试报告给出达标数据、国家认证机构颁发的证书 |
| ★★ 半直接证据 | 产品材料覆盖但未点名 | 白皮书提到具备该能力但未展开、方案文档描述相似功能 |
| ★ 间接证据 | 从相关描述推断 | 架构图上有相关模块、产品截图显示配置项、竞品对比表暗示具备 |
| ○ 旁证 | 不能直接证明但可辅助 | 同类客户案例、合作伙伴名单、过往项目经验 |

**评定时的证据要求**：
- "完全满足" → 必须有 ★★★ 级证据
- "基本满足" → 主要要点 ★★★，次要要点 ≥ ★★
- "部分满足" → 部分要点 ≥ ★★，允许混用
- "不满足" → 无 ★★ 及以上证据

**只有旁证（○）不足以支撑任何满足级别的评定**，应判"不满足"或请用户补充材料。

## 3. 复合指标拆分

一条招标指标里如果包含多个独立要求点，**必须**拆分成关键要点清单分别评估。

**识别信号**：
- 逗号/顿号/分号连接的功能清单
- "并"、"且"、"同时"等连词
- "支持 X 且满足 Y"的复合条款
- 一句话里出现 2 个以上量化指标

**拆分示例**：

原始指标：
> 支持分布式部署，单节点处理能力不低于 500Mbps，集群规模不少于 8 节点，提供 Web 管理界面和 API 接口

拆分为 4 个要点：
1. 支持分布式部署
2. 单节点处理能力 ≥ 500Mbps
3. 集群规模 ≥ 8 节点
4. 提供 Web 管理界面 + API 接口（此项还可细拆）

每个要点分别找证据、分别评定，然后**按最弱处定整条等级**（3 项完全满足 + 1 项部分满足 → 整条部分满足）。

## 4. 应答策略模板

**每条指标都要给应答策略建议**（不论评定等级）。按以下模板生成，结合具体证据调整措辞。

### 4.1 完全满足 → 正偏离应答

**策略核心**：不只声明满足，还要"亮点化"包装证据，争取加分项。

**模板**：
> **应答策略**：完全响应。建议在应答文档中：
> 1. 明确写"完全满足"/"正偏离"
> 2. 援引 [证据出处] 的具体数据/功能描述
> 3. 可适度扩展，强调**超出要求**的部分（如招标要 500、我方 1000；招标要 3 类、我方 12 类），作为技术优势点
> 4. 如果有对应的第三方测试报告或认证证书，建议作为应答附件

### 4.2 基本满足 → 有条件正偏离应答

**策略核心**：整体声明满足，但针对"差一点"的地方要写明补充说明，避免被评委抠细节。

**模板**：
> **应答策略**：基本响应，建议声明"满足"。具体撰写时注意：
> 1. 主要要点援引 [证据出处] 的直接证据
> 2. 对于 [差异点]，补充说明："[具体差异，如 SNMP 推送方式未在告警章节单独描述，但产品管理协议章节已支持 SNMP，可通过 XX 配置实现]"
> 3. 如数值差距很小（<10%），可附承诺："实际部署时可通过 [优化手段] 达到或超过招标要求"
> 4. 建议准备 [相关补充材料]，在答疑或澄清阶段提交

### 4.3 部分满足 → 负偏离应答（可接受）

**策略核心**：坦率声明部分偏离，同时给出解决路径，避免直接不响应。

**模板**：
> **应答策略**：部分响应，声明"负偏离"或"部分满足"。具体撰写时：
> 1. 对**满足的要点**：正常援引证据进行正偏离应答
> 2. 对**不满足的要点 [列出具体点]**：建议采取以下路径之一：
>    - **路径 A（定制开发）**：承诺在项目实施期内完成开发，给出时间节点和验收方式
>    - **路径 B（合作交付）**：通过集成第三方产品/合作伙伴能力补足
>    - **路径 C（替代方案）**：说明替代实现虽在形式上不同，但功能等价，请评委接受
> 3. 如果该指标不是 ★ 项或必须项，适度应答即可；如果是 ★ 项，必须给出明确可信的补足方案
> 4. 提醒用户：此条可能影响评分，建议与销售/方案经理讨论是否值得投入定制成本

### 4.4 不满足 → 负偏离应答（需决策）

**策略核心**：根据要求级别决定是"坦诚负偏离"还是"考虑弃标/策略性投标"。

**模板**：
> **应答策略**：无法响应。建议分情况处理：
> 1. **若为"必须"或 ★ 项**：
>    - 该条款可能构成**废标风险**（招标文件通常规定"必须项不满足即视为不实质响应"）
>    - 强烈建议在投标前与**甲方沟通澄清**，或评估是否**定制开发 / OEM / 联合投标**可行
>    - 若决定仍应答，需在负偏离说明中**如实标注**，并附可信的补足承诺和商务让步
> 2. **若为"应"或非关键项**：
>    - 如实填写"负偏离"或"不响应"
>    - 在偏离说明中简述原因（产品定位/技术路线不同）
>    - 商务条款上适度让步争取综合得分
> 3. **不建议**虚假应答或模糊措辞 —— 被核查发现会直接废标或进黑名单

## 5. 特殊情况处理

### 5.1 证据矛盾

不同参考文档对同一能力的描述不一致（比如白皮书说支持、测试报告未测试）：
- 取**最新**、**最权威**（测试报告 > 产品手册 > 白皮书 > PPT）的那份
- 在证据摘录中**同时列出矛盾方**，并注明已采信哪份
- 提醒用户"材料间存在表述差异，建议更新参考文档"

### 5.2 参考文档过时

参考文档版本明显老于招标关注的产品版本：
- 用现有材料照评，但在"证据"末尾标注 `⚠️ 参考文档版本 vX.X，可能不反映最新能力`
- 在报告末尾附"建议更新的参考资料"清单

### 5.3 招标指标模糊

指标描述含糊（如"具备良好的可扩展性"、"性能业界领先"）：
- 在"关键要点"里标注 🔍 注明这是主观/模糊条款
- 评定时以"能否找到相关功能描述和典型案例"为准
- 应答策略里提醒用户"此条依赖评委主观判断，建议在应答时用多维度证据（性能数据 + 客户案例 + 架构说明）组合论证"

### 5.4 强制性资质缺失

招标要求的资质证书（等保测评、商密认证、CCRC 等）在参考材料中找不到：
- 直接判 **不满足**，不要推断"应该有"
- 在应答策略中特别提醒："这类资质类条款通常是**实质性响应**的硬门槛，建议立即向资质管理部门核实并补充证书扫描件"

### 5.5 隐含要求

招标条款里有隐含要求容易漏掉（比如"提供 3 年原厂质保" → 隐含"原厂"+"3 年"两个要点）：
- 在"关键要点"里**显式拆出**隐含要求
- 评估时分别找证据

## 6. 评定时的自检清单

每条评完后，快速核对：

- [ ] 有没有对每个关键要点都找过证据（没漏要点）？
- [ ] 证据是不是原文摘录（没改写）？
- [ ] 证据出处有没有写清楚（能让用户翻回去）？
- [ ] 量化指标有没有对数字（没用"接近"之类模糊判断）？
- [ ] 应答策略是不是针对本条指标的具体情况（不是套话）？
- [ ] 评定是不是基于证据而不是常识推断（没硬凑）？

六项都过关，才能进入下一条。
