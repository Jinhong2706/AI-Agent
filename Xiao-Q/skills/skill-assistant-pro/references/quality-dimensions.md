# 质检维度详细规则


## D0：知识增量（15%）— 核心维度

> **Good Skill = 专家知识 − LLM 已知知识**

评估 Skill 中每个段落是否带来了 LLM 训练数据中不存在的知识增量。

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D0.1 Expert 占比 | **必须** | 抽样 5-10 个段落，Expert 知识占比 ≥ 60% |
| D0.2 无基础解释 | **必须** | 不含"什么是 X"式基础概念解释（LLM 已知） |
| D0.3 决策树 | **必须** | 有非显而易见的决策判断（"当 X 失败时尝试 Y，因为 Z"） |
| D0.4 权衡与取舍 | **建议** | 有只有专家才知道的 trade-off（"A 更快但 B 能处理边界情况 C"） |
| D0.5 真实边界情况 | **建议** | 来自实战经验的边界情况 |
| D0.6 无标准教程 | **建议** | 不含标准操作的分步教程（"Step 1: 打开文件, Step 2: 编辑"） |
| D0.7 无通用最佳实践 | **建议** | 不含泛泛的建议（"写干净的代码""处理错误"） |

评分标准：

| 分数 | Expert 知识占比 | 特征 |
|------|----------------|------|
| 0-30 | < 30% | 大量基础解释和标准教程，LLM 已知内容为主 |
| 31-50 | 30-50% | 有专家知识但被冗余内容稀释 |
| 51-70 | 50-70% | 以专家知识为主，少量激活性内容 |
| 71-90 | 70-90% | 几乎全是知识增量，激活知识简短有效 |
| 91-100 | > 90% | 纯知识增量——每段都值得占用 Token |

**红旗（即判 ≤30）**：
- "什么是 [基础概念]" 的解释段落
- 标准库/框架用法的分步教程
- 行业标准术语的定义
- 泛泛的最佳实践（"写可维护的代码"）

**绿旗（高知识增量信号）**：
- "NEVER do X because [非显而易见的原因]"
- 只有踩过坑才知道的决策树
- 领域特定的思维框架
- 对 LLM 常见错误的精准纠正

## D1：元数据规范（10%）

检查 Front Matter + **Description 质量专项**（对齐 [anthropic-best-practices.md §Description 三件事原则](anthropic-best-practices.md#description-三件事原则)）。

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D1.1 `name` | **必须** | 存在且非空；kebab-case 格式；≤ 64 字符；与目录名一致；不含保留词 `anthropic`/`claude` |
| D1.2 `description` — WHAT | **必须** | 清晰说明 Skill 做什么（核心功能） |
| D1.3 `description` — WHEN | **必须** | 说明何时使用（"Use when..."触发场景） |
| D1.4 `description` — KEYWORDS | **必须** | 包含 Agent 可搜索的关键词（工具名/动作词/领域词） |
| D1.5 `version` | **建议** | SemVer 格式（`x.y.z`），**放在 `metadata:` 下**（spec 顶层只有 name/description；version 属于"非功能元数据"应归 metadata 自由 map） |
| D1.6 `tags` | **建议** | 数组 ≥ 2 个标签（顶层 `tags:` 或 `metadata.tags` 均可） |
| D1.7 `keywords`/`triggers` | **建议** | 触发关键词数组 |
| D1.8 触发信息位置 | **必须** | "When to use" 信息在 description 中而非 body 中（三层加载原则） |
| **D1.9 description 长度** | **必须** | description 字符数 ≥ 80 且 ≤ 1024（Anthropic 官方硬上限，超出会被截断） |
| **D1.10 Negative Boundaries** | **必须** | description 含显式 "Do NOT use for..." / "不要用于..." 子句划反向边界（防误触发） |
| **D1.11 命名规范** | **建议** | 优先动名词（gerund，如 `processing-pdfs`）；可接受名词短语 / 动作导向；避免模糊（`helper`/`utils`/`tools`） |
| **D1.12 trust signal 字段** | **建议** | 含 `compatibility` / `allowed-tools` 至少 1 项（开源场景再加 `license`；内部 / 项目级 skill `license` 可省）|
| **D1.13 第三人称** | **必须** | description 不含 "I can..."、"You can..."、"Help you..."（错位人称会破坏发现） |
| **D1.14 语言匹配** ⭐ | **必须** | description 主体语言匹配用户主流提问语言；多语言场景含中英双语 trigger words 兜底；技术术语保留英文原文（避免"中文用户 + 全英文 description"导致召回 ↓ 30-50%） |
| **D1.15 Trigger Words 边界** ⭐ | **建议** | description 只列**主入口动作**（用户冷启动会单独说的话），不列**派生菜单动作**（已进入 skill 后才会触发的子动作如"收藏"、"导出"、"重搜"）；派生动作放路由表/模块菜单 |

> **Description 是决定 Skill 存亡的字段**：Agent 只看 description 决定是否加载 Skill。
> Description 差 = Skill 永远不会被激活，无论 body 多优秀。
>
> ⚠️ **D1.9 + D1.10 + D1.13 + D1.14 是 best practices 自检的高频失分项**——很多老 skill 写得太长 / 没 negative boundaries / 第一人称腔 / 全英文 description 给中文用户用，量化给分时容易整体掉到 D1 ≤ 50。
>
> 💡 **D1.15 是常见过度堆砌**——把 skill 内部所有功能都列进 trigger words，反而拉低路由准确率。判断方法：问自己"用户冷启动对话时会不会单独说这句话"。

评分：D1.2-D1.4 + D1.8-D1.10 + D1.13-D1.14 八项必须各 8 = 64 分 | D1.15 建议 8 分 | D1.1 + 其余建议项分 28 分 | 满分 100

### Negative Boundaries 示例

✅ Good：
```yaml
description: >-
  Convert a Markdown resume into a professionally formatted PDF...
  Use when the user asks to create a resume, format a resume, or convert
  a resume to PDF. Triggers include: 'resume', 'CV', 'curriculum vitae'.
  Do NOT use for: cover letters, job searching, interview preparation,
  LinkedIn profile optimization, or portfolio websites.
```

❌ Bad（缺 negative boundaries 时，"准备面试"等含 career 关键词的请求会误触发）：
```yaml
description: Convert markdown resume to PDF. Use when working with resumes.
```

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D3.1 步骤编号 | **必须** | 清晰的步骤编号 |
| D3.2 步骤完整性 | **必须** | 从输入到输出完整链路无断裂 |

| D3.5 步骤间衔接 | **建议** | Step N 输出 = Step N+1 输入 |
| D3.6 异常路径 | **建议** | 失败时的处理方式 |
| D3.7 效率约束 | **建议** | 性能/效率限制说明 |
| D3.8 工具/命令 | **建议** | 使用的工具有明确指定 |
| D3.9 架构图 | **建议** | 复杂 SKILL 有流程图 |

评分：4 必须项各 15 = 60 分 | 5 建议项各 8 = 40 分 | 满分 100

## D5：反模式清单（10%）

> 专家知识的一半是知道**不该做什么**。好的 NEVER 列表来自踩坑经验，不是泛泛的警告。

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D5.1 NEVER 列表存在 | **必须** | 有明确的"不要做"列表 |
| D5.2 具体性 | **必须** | 每条反模式指向具体行为而非泛泛警告 |
| D5.3 原因说明 | **必须** | 每条反模式附带 WHY（非显而易见的原因） |
| D5.4 专家经验 | **建议** | 反模式来自实战踩坑，非常识 |
| D5.5 示例对比 | **建议** | 有"Bad → Good"对比示例 |

评分标准：

| 分数 | 特征 |
|------|------|
| 0-30 | 无反模式，或只有"避免错误""注意边界"式模糊警告 |
| 31-50 | 有具体 NEVER 列表但缺少原因 |
| 51-70 | 具体 NEVER 列表 + 部分原因说明 |
| 71-90 | 专家级反模式 + 详细原因 + 部分示例 |
| 91-100 | 每条都是"踩过坑才知道"的经验，附 Bad→Good 对比 |

**专家级反模式示例**：
```
NEVER use Inter/Roboto/Arial — these are AI 生成内容的签名字体
NEVER 在编辑后 pack 之前跳过 validate — OOXML 损坏后 PowerPoint 会静默修复并丢失数据
```

**弱反模式示例**：
```
避免犯错误。注意边界情况。不要写坏代码。
```

## D7：渐进式披露（10%）

> 三层加载系统的核心：SKILL.md body 放路由和决策框架（≤ 300 行理想），详细内容按需从 references/ 加载。

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D7.1 Body 行数 | **必须** | SKILL.md body ≤ 500 行（理想 ≤ 300 行）；> 500 行必须拆分 |
| D7.2 References 拆分 | **必须** | 低频/详细内容在 references/ 中而非全堆在 body |
| D7.3 加载触发器 | **必须** | body 中有明确的加载指令（"MANDATORY — READ ENTIRE FILE" 等） |
| D7.4 防过载标记 | **建议** | 有"Do NOT Load"标记防止不必要的加载 |
| D7.5 文档自洽 | **建议** | 交叉引用的文件确实存在 |
| D7.6 配套脚本 | **建议** | CPU 擅长的逻辑在 scripts/ 中 |
| **D7.7 引用深度 = 1** | **必须** | references/ 文件只从 SKILL.md 直接链接，不嵌套引用（SKILL.md → ref ✅；ref → detail ❌）。Anthropic 官方明示：嵌套引用时 Claude 可能用 `head -100` 部分读取导致信息丢失 |
| **D7.8 长 ref 含 TOC** | **建议** | 任何 > 100 行的 reference 文件首部含 `## Contents` 段落 |

评分标准：

| 分数 | 特征 |
|------|------|
| 0-30 | body 800+ 行无拆分，或 references/ 存在但从未被引用（孤儿引用） |
| 31-50 | body > 500 行，部分内容可拆分但未拆 |
| 51-70 | body ≤ 500 行，有 references/ 但缺加载触发器 |
| 71-90 | body ≤ 300 行，references/ 有加载触发器 |
| 91-100 | 完美的三层架构：精简 body + 按需加载 + 防过载标记 |

## D8：安全审计（10% — OWASP LLM Top 10 + 声明-行为一致性）


### D8-A：传统安全审计

| 检查项 | 级别 | 规则 | 参考标准 |
|--------|------|------|---------|
| D8.1 Prompt 注入防护 | **必须** | 用户输入边界隔离；不信任外部内容；校验工具参数 | OWASP LLM01 |
| D8.2 敏感信息泄露防护 | **必须** | 输出无 API Key/密码/内网地址；有脱敏规则 | OWASP LLM02 |
| D8.3 过度权限控制 | **必须** | 最小权限原则；文件操作有范围限制 | OWASP LLM08 |
| D8.4 供应链安全 | **建议** | 外部脚本有来源说明；依赖版本锁定 | OWASP LLM03 |
| D8.5 输出内容安全 | **建议** | 不含可直接执行的危险指令 | OWASP LLM05 |
| D8.6 内网信息保护 | **建议** | 不硬编码内网 IP/域名；示例用占位符 | 内网安全 |
| D8.7 数据边界与隔离 | **建议** | 声明数据访问范围；上下文隔离 | OWASP LLM06 |

### D8-B：声明-行为一致性检查

比较 SKILL.md 声明的功能与 `scripts/` 中代码的实际行为：

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D8.8 声明-行为匹配 | **必须** | description 声明的功能 ≈ 代码实际行为（不匹配 = 高风险） |
| D8.9 权限-功能匹配 | **必须** | 使用的权限（文件读写/网络/Shell）与声明功能匹配 |
| D8.10 LLM 越狱检测 | **必须** | 无嵌入的 prompt override 尝试（含 Base64/Unicode/ROT13 编码） |
| D8.11 隐蔽行为检测 | **建议** | 无零宽字符、Unicode bidi 控制符、注释中的隐藏指令 |
| D8.12 凭证硬编码 | **建议** | 无硬编码的真实 Token/密码/密钥 |

**声明-行为不匹配示例**：
- "代码格式化" Skill 却请求 `network` + `shell` → 🔴 严重不匹配
- "API 调用" Skill 请求 `network` → ✅ 合理
- Skill 声称只读取文件但 scripts/ 中有 `curl | bash` → 🔴 声明-行为矛盾

**LLM 越狱检测项**：
- "Ignore previous instructions" / "Forget everything above"
- "You are now..." / "Your new role is"
- "[SYSTEM]" / "[ADMIN]" / "[ROOT]"（伪造角色标签）
- Base64 编码的越狱指令（需解码后分析）
- Unicode 零宽字符中嵌入的隐藏指令

评分：D8-A 必须项各 10 = 30 分 | D8-A 建议项各 5 = 15 分 | D8-B 必须项各 12 = 36 分 | D8-B 建议项各 9.5 = 19 分 | 满分 100

### D8-C：detector 自身的金标回归

`scripts/skill_audit.py` 的 14 个 detector 实现本身的回归测试——**不**是用来评某个具体 skill 是否安全，而是当 detector 改动时确保不退化。

```bash
python3 .cursor/skills/skill-assistant/scripts/run_golden_baseline.py
```

> 跑 `tests/golden-dataset/`（workspace 根）下的 4 个反面教材 + 2 个正面样本，机械验证：
>
> - **bad-skills 召回率** ≥ 95%（detector 漏报会阻断）
> - **good-skills 误报率** ≤ 5%（detector 过敏会阻断）
> - **`known_gaps`**：当前 detector 已知的覆盖盲点（不计入失败，但显式提示，作为补强 backlog）

**触发条件**：`skill_audit.py` 修改 / 新增 detector / 调阈值 / 改 IOC 库时**必须**跑一次。

**当前 known gaps**：

| Detector | 盲点 |
|---|---|
| `PostInstallHookDetector` | 仅扫 `package.json`，不扫 `.md` 内联代码块中的 npm 钩子 |
| `ExfiltrationDetector` | `curl -X POST -d $VAR` 上下文匹配规则未覆盖通用变量名 |
| `CredentialTheftDetector` | 仅检测具体凭证窃取手法（osascript/Keychain/SSH）；不检测硬编码 API key 字面值 |
| `EntropyDetector` | 在 `KEY=long_string` 上下文阈值偏保守，未触发 sk-prod-... 这类含可读子串的场景 |

> 这些 gap 会在专项 detector 改进 PR 中逐个修复，每修复一个 known_gap 转 FIXED 后就从本表删除。

## D9：自由度校准（5%）

> 任务的脆弱程度决定 Skill 应给 LLM 多大自由度。创意任务需要原则而非步骤，脆弱操作需要精确脚本而非模糊指导。

| 检查项 | 级别 | 规则 |
|--------|------|------|
| D9.1 任务风险识别 | **必须** | Skill 明确了任务的风险/脆弱程度 |
| D9.2 自由度匹配 | **必须** | 高风险操作→低自由（精确脚本）/ 创意任务→高自由（原则框架） |
| D9.3 参数化 | **建议** | 脆弱操作中变量用参数而非让 LLM 自由发挥 |

评分标准：

| 分数 | 特征 |
|------|------|
| 0-30 | 创意任务用死板步骤，或脆弱操作只给模糊指导 |
| 31-60 | 自由度部分匹配但不一致 |
| 61-100 | 自由度与任务风险完美匹配 |

## Skill 模式权重调整

| Skill 模式 | 核心权重调整 |
|------------|-------------|
| 思维型 (Mindset) | D0→20%, D5→12%（知识增量和反模式最重要） |
| 导航型 (Navigation) | D7→15%（渐进式披露最重要） |
| 流程型 (Process) | D3→20%（执行流程最重要） |
| 工具型 (Tool) | D8→15%, D6→10%（安全和工程最重要） |
| 涉及外部调用 | D8→15%, D8.1/D8.4/D8.5 必检 |
