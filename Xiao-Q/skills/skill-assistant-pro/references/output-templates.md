# 搜索输出模板

搜索模块 Step 6 使用此文件。**搜索时无需加载此文件**——模板结构已内化为 Agent 行为，仅在输出格式有疑问时查阅。

> **🚨 展示形式硬约束（MUST 级红线）**
>
> **本文件里的每一个代码块都是强制模板，不是可选建议**。诊断头的关键词表 / 渠道漏斗表 / 环境快照、每个来源分区的 Top N、跨源比较、已安装对比、推荐方案四档——**全部必须用 markdown 表格**。
>
> ❌ **NEVER** 用列表 / 卡片 / 每条一个 `###` 小标题 / 段落文字替代 Top N 表格
> ❌ **NEVER** 把诊断头写成散文式（"本次搜索使用了 xxx 关键词，在 yyy 渠道找到 zzz 条..."）
> ❌ **NEVER** 即使只有 1 条结果也不能省表头——至少 1 行表头 + 1 行数据或"_暂无结果_"斜体占位
>
> 违反上述任一规则会在 Agent 自检环节触发 🚨 阻断级异常 `LAYOUT_NON_TABLE`，**必须回退重写输出**，不得交付用户。检测依据详见 [search-transparency.md#异常检测规则](search-transparency.md#三异常检测规则)。

> 「搜索诊断头」是输出的第一部分，展示关键词、候选数、漏斗收敛、异常提示，让用户看见黑盒。
> 诊断头的数据模型、异常检测规则、diff 展示详见 [search-transparency.md](search-transparency.md)。
> Refine 菜单 7-11 的处理规则详见 [refine-search.md](refine-search.md)。

---

## 输出结构

完整输出顺序（全部必须，除非标注「条件」）：

1. 🔍 **搜索诊断头**（本文件「搜索诊断头」章节）
2. 按来源分区展示（含「命中」列）
3. 跨源比较表（条件：有相似 Skill 时）
4. 已安装对比（条件：有功能重叠时）
5. 推荐方案（四档）
6. 提案式交互菜单（1-6 原菜单 + 7-11 refine 菜单）

### 第 4 分区：WebSearch 兜底·参考

**只在 `diagnostics.channels.websearch.status` 为 `triggered` 或 `strategy_forced` 时输出此分区**。该分区结果**不参与前 3 分区的推荐指数排序**，独立展示作为参考。

```markdown
## 🌐 来自 WebSearch 兜底 — 参考（触发原因：{trigger_reason}）

> ⚠️ 本分区结果来自真正的网页搜索（博客 / Awesome List / 社区文章），**不是可直接 `install_skill.sh` 安装的 Skill**，需要 Agent 识别其中的仓库链接二次处理。

### 🔍 网页中发现的潜在 Skill 仓库（已去重，排除前 3 分区已出现的）

| # | 仓库 / 链接 | 发现来源 | ⭐ Stars | 说明 |
|---|-----------|---------|:-:|------|
| 9 | [{owner/repo}](链接) 📝 | 博客 "{博客标题}" | ⭐ {N} | {描述} — 输入编号安装 |
| 10 | [{owner/repo}](链接) 📝 | Awesome List "{列表名}" | ⭐ {N} | {描述} |

### 📚 Awesome List / 博客索引（价值高但非单个 Skill）

| # | 来源 | 类型 | 说明 |
|---|------|:-:|------|
| 11 | [{标题}](URL) | Awesome | 含 {N} 个相关 Skill 索引 — 输入"展开 {标题}" 可提取全部 |
| 12 | [{博客标题}](URL) | 博客 | 介绍了 {话题}，作者 {@handle}，可作为能力缺口分析参考 |
```

**输出触发规则**：

| WebSearch 渠道状态 | 第 4 分区输出 |
|---|---|
| `not_triggered`（speed 策略 / balanced 未触发） | **整区省略** |
| `triggered`（balanced 条件触发） | 输出，标题含触发原因 |
| `strategy_forced`（thorough 默认常驻） | 输出，标题标 `✅ 默认常驻` |
| `disabled_by_strategy`（speed 硬规则） | **整区省略** |

**与前 3 分区的去重规则**：

1. 抽取出的 GitHub 仓库如果已出现在前 3 分区——**不出现在第 4 分区**，但在原分区该条增加"🌐 多源加分"标记，推荐指数 +0.3
2. 抽取出的平台链接（skills.sh / skillhub）如果已出现——同上，`diagnostics.channels.websearch.multi_source_boost` 计数 +1
3. Awesome List 和博客索引**从不参与去重**，始终独立作为「📚 索引类」输出

**命中列速查**：

| 标记 | 含义 |
|---|---|
| 🎯 L3 + L2-X | 宽窄都命中，最强匹配 |
| ✅ L2-X | 精确词命中 |
| ⚠️ L3 only | 仅宽路命中——需用户二次判断 |
| ➕ L4 | 扩展词命中（通常权重低） |
| 🈯 中文 | 中文变体命中 |
| 🏷️ #tag | `#tag` 搜索模式命中 |

无结果的来源用斜体标注 _该渠道无匹配结果_。

## 已安装 Skill 对比表

Step 0 扫描到的已安装 Skill 与搜索需求功能重叠时使用：

```markdown
### 🏠 已安装对比：你的 {已安装 Skill 名称} vs 搜索结果

| 能力 | 你的 {已安装名称} | #{A} {名称} | #{B} {名称} |
|------|:-:|:-:|:-:|
| {能力维度 1} | ✅ / ❌ | ✅ / ❌ | ✅ / ❌ |
| {能力维度 2} | ✅ / ❌ | ✅ / ❌ | ✅ / ❌ |

**决策建议**：

| 方案 | 适用场景 | 操作 |
|------|---------|------|
| **吸收融合**（推荐） | 已安装覆盖大部分能力，搜索结果有少量独有特性 | 抓取 #{X} 的 SKILL.md，融合独有特性 |
| **替换** | 搜索结果全面优于已安装 | 卸载已安装，安装 #{X} |
| **并行安装** | 两者定位不同，互补而非替代 | 保留已安装，额外安装 #{X} |
```

**决策逻辑**：

| 已安装 vs 搜索结果 | 推荐决策 |
|-------------------|---------|
| 已安装能力 ⊃ 搜索结果 | 无需安装 |
| 已安装能力 ≈ 搜索结果 | 吸收融合 |
| 已安装能力 ⊂ 搜索结果 | 替换 |
| 功能正交 | 并行安装 |
| 本地无相关 | 跳过对比 |

## 推荐方案（四档）

```markdown
### 💡 推荐方案

| 方案 | 适用场景 | 组合 |
|------|---------|------|
| **精简** | 只需核心能力 | 安装 #{最高推荐指数编号} |
| **完整** | 核心 + 管理能力 | 安装 #{A} + #{B}（互补组合） |
| **增强** | 全面覆盖 | 安装 #{A} + #{B} + #{C} |
| **融合** | 集大成自定义 Skill | 融合 #{A} + #{B} + #{C} 的特性（引导到创建模块） |

📊 评分说明：推荐指数 = 热度(40%) + 权威(35%) + 鲜度(25%)
⚠️ 安全提醒：安装前将自动执行三引擎安全扫描
```

每个方案必须给出具体编号和理由，融合方案需说明融合哪些特性。

## 提案式交互菜单（每次搜索后必须附加）

共 11 项：1-6 为结果处理，7-11 为 refine 分支。完整的 refine 处理规则见 [refine-search.md](refine-search.md)。

```markdown
---
**下一步？**

— 结果处理 —
1. 📦 安装？（告诉我编号，如"安装 1"或"安装 1 和 2"）
2. 🔍 查看详情？（如"看 3 的 SKILL.md"）
3. 🔀 融合到已安装 Skill？（如"把 #2 的备份能力融合到 skill-assistant"）
4. 🔄 搜索相似 Skill？
5. ⭐ 收藏到喜爱列表？
6. 🌐 完全重搜？（丢弃当前结果，用新关键词重来）

— 或者搜索不对劲？—
7. 🔧 修改关键词（如"L2-A 改成 ATS resume"）
8. 🔎 深搜某渠道（如"skills.sh 深搜 limit=50"）
9. 📡 升级策略（当前 {current} → {next}，预计多 {pct}% 候选）
10. 🈯 补充关键词（如"加 career builder"或"加中文：求职"）
11. 🗑️ 排除关键词（如"排除 template，我要 AI agent 型的"）
```

**菜单 6 vs 菜单 7 的区别**：
- 菜单 6 = 完全重搜，丢弃诊断历史
- 菜单 7 = 保留诊断状态，只改指定层级，输出带 diff

**Refine 上限**：单次会话同一搜索的 refine 次数上限 5 次。超限后菜单 7-11 仍可用但会警告。

如果用户是为了参考学习（创建/修改 Skill），额外说明结构亮点和值得借鉴的写法。
