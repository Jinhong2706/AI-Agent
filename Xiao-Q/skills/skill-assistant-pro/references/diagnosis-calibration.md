# 诊断校准参考手册

诊断模块 Step 1 执行时参阅本文件，获取评分校准锚点和 Token 分类示例。

---

## 1. 评分校准表

### 指令 (Directive) 评分校准

| 等级 | 分数 | 典型特征 |
|------|------|----------|
| 强 | 0.7-1.0 | 行为锚点明确（祈使动词）、有 Few-Shot 示例、设计原则已声明、隐喻/直觉已建立 |
| 中 | 0.4-0.69 | 有基本指令和工作流引导，但缺少示例或设计原则 |
| 弱 | 0-0.39 | 缺少核心锚点，加载后 LLM 行为几乎没变化 |

**指令检查清单：**
- [ ] 有明确的行为锚点（核心指令使用祈使动词）
- [ ] Few-Shot 示例充分
- [ ] 隐喻/直觉已建立（帮助 LLM 形成正确推理框架）
- [ ] 设计原则已声明（为临场判断提供决策框架）
- [ ] 工作流引导清晰

### 约束 (Constraint) 评分校准

| 等级 | 分数 | 典型特征 |
|------|------|----------|
| 强 | 0.7-1.0 | 明确禁止已知错误 + 边界条件覆盖 + 反模式列举 + 错误处理指导 |
| 中 | 0.4-0.69 | 有基本禁止但边界覆盖不全 |
| 弱 | 0-0.39 | 缺少禁止声明，LLM 可能犯已知错误 |

**渐进式披露评估**：

| 层级 | 加载时机 | 应包含的内容 | 常见违规 |
|------|----------|-------------|----------|
| **发现层** | Agent 启动 | 仅 name + description | description 与实际功能不一致 |
| **激活层** | 任务匹配后 | SKILL.md 正文 | 核心指令藏在 references/ 中 |
| **执行层** | 按需加载 | references/、scripts/ | 输出模板内联在 SKILL.md 中 |

### 冗余 (Redundancy) 评分校准

| 等级 | 冗余率 | 典型特征 |
|------|--------|----------|
| 干净 | <15% | 几乎每个 Token 都在服务指令或约束 |
| 轻微冗余 | 15-30% | 有少量可优化空间 |
| 中等冗余 | 30-50% | 核心指令被稀释 |
| 严重冗余 | >50% | 核心指令被淹没 |

### 效能分校准锚点

| 分数段 | 典型 Skill 特征 |
|--------|----------------|
| 85+ (A) | 行为偏移大且精准；有 Few-Shot；约束覆盖；冗余 <15% |
| 70-84 (B) | 核心框架清晰但缺示例或部分约束缺失；冗余 15-30% |
| 50-69 (C) | 有基本指令但缺设计原则；约束薄弱；冗余 30-50% |
| 30-49 (D) | 指令模糊，LLM 需大量猜测；约束几乎缺失 |
| 0-29 (F) | 加载后 LLM 行为几乎没变化；大量过程性描述 |

## 2. 冗余判定矩阵（因 Skill 类型而异）

| Token 类型 | 执行型 | 流程型 | 规范型 | 认知型 |
|-----------|--------|--------|--------|--------|
| 角色扮演/礼貌用语 | 冗余 | 冗余 | 冗余 | 冗余 |
| 过程性逻辑描述 | **冗余** | 部分冗余 | 可能是指令 | 可能是指令 |
| 背景知识注入 | 冗余 | 视情况定 | 指令 | **指令** |
| 设计哲学/原则 | 冗余 | 视情况定 | 指令 | **指令** |
| 与脚本重复的描述 | 冗余 | 冗余 | 冗余 | 冗余 |
| SKILL.md 与 references 重复 | 冗余 | 冗余 | 冗余 | 冗余 |

**关键区分法**：删掉这段 Token 后，LLM 的任务表现是否下降？下降 → 不是冗余；不变 → 是冗余。

## 3. Token 分类示例

### 示例 A：代码审查类 Skill

> "你是一个专业的代码审查员，拥有 10 年开发经验。请仔细检查代码中的安全漏洞，特别关注 SQL 注入和 XSS。不要给出模糊的建议，每个问题必须指出具体的代码行号。"

- "你是一个专业的代码审查员，拥有 10 年开发经验" → **冗余**（角色扮演）
- "请仔细检查代码中的安全漏洞" → **指令-激活**
- "特别关注 SQL 注入和 XSS" → **指令-锚定**
- "不要给出模糊的建议" → **约束**
- "每个问题必须指出具体的代码行号" → **约束**

### 示例 B：认知型 Skill 的知识注入

> "在 Tica 框架中，组件的生命周期与 Flutter Widget 不同：Tica 使用 attach/detach 而非 mount/unmount。不能简单地将 initState 映射为 onAttach。"

- "在 Tica 框架中...与 Flutter Widget 不同" → **指令-注入**（私有框架知识）
- "Tica 使用 attach/detach 而非 mount/unmount" → **指令-注入**（API 映射）
- "不能简单地将 initState 映射为 onAttach" → **约束**（禁止错误映射）

> 整段都不是冗余。删掉后 LLM 会犯错误映射。

### 示例 C：执行型 Skill 的冗余识别

> "PDF 旋转工具用于旋转 PDF 文件的页面。PDF 是一种广泛使用的文档格式，由 Adobe 于 1993 年创建。本工具使用 PyPDF2 库。具体步骤：1. 读取 2. 旋转 3. 保存。"

- "PDF 旋转工具用于旋转 PDF 文件的页面" → **冗余**（同义反复）
- "PDF...由 Adobe 于 1993 年创建" → **冗余**（LLM 已知常识）
- "本工具使用 PyPDF2 库" → **指令-注入**（指定依赖库）
- "具体步骤：1...2...3..." → **冗余**（执行型应在脚本中实现）
