# 扩展模式

## 容量阈值

| 规模 | 条目数 | 策略 |
|-------|---------|----------|
| 小型 | <100 | 单个 memory.md，无需命名空间 |
| 中型 | 100-500 | 拆分为 domains/，基本索引 |
| 大型 | 500-2000 | 完整命名空间层次，积极压缩 |
| 超大型 | >2000 | 每年归档，仅热层摘要 |

## 何时拆分

创建新命名空间文件时：
- 单个文件超过200行
- 主题有10+个不同纠错
- 用户明确分离上下文（"对于工作..."，"在这个项目中..."）

## 压缩规则

### 合并相似纠错
```
之前（3条）：
- [02-01] 使用Tab而非空格
- [02-03] 用Tab缩进
- [02-05] Tab缩进，谢谢

之后（1条）：
- 缩进：Tab（3次确认，02-01 到 02-05）
```

### 总结冗长模式
```
之前：
- 给Marcus写邮件时，使用要点，不超过5项，
  不使用术语，首先说结论，他更喜欢早上发送

之后：
- Marcus邮件：要点≤5，无术语，BLUF，偏好AM
```

### 带上下文归档
移动到冷层时：
```
## 已归档 2026-02

### 项目：old-app（自2025-08以来非活动）
- 使用Vue 2模式
- 更喜欢Vuex而非Pinia
- Jenkins上的CI（已弃用）

原因：项目完成，模式不太可能适用
```

## 索引维护

`index.md` 跟踪所有命名空间：
```markdown
# 记忆索引

## 热层（始终加载）
- memory.md：87行，更新于2026-02-15

## 温层（按匹配加载）
- projects/current-app.md：45行
- projects/side-project.md：23行
- domains/code.md：112行
- domains/writing.md：34行

## 冷层（归档）
- archive/2025.md：234行
- archive/2024.md：189行

上次压缩：2026-02-01
下次计划：2026-03-01
```

## 多项目模式

### 继承链
```
global（全局）(memory.md)
  └── domain（领域）(domains/code.md)
       └── project（项目）(projects/app.md)
```

### 覆盖语法
在项目文件中：
```markdown
## 覆盖
- 缩进：空格（覆盖全局Tab）
- 原因：项目ESLint配置要求空格
```

### 冲突检测
加载时检查冲突：
1. 构建继承链
2. 检测矛盾
3. 最具体的胜出
4. 记录冲突以供后续审查

## 用户类型适配

| 用户类型 | 记忆策略 |
|-----------|-----------------|
| 高级用户 | 积极学习，最少确认 |
| 普通用户 | 保守学习，频繁确认 |
| 团队共享 | 每用户命名空间，共享项目空间 |
| 注重隐私 | 仅本地，每类别明确同意 |

## 恢复模式

### 上下文丢失
如果代理在会话中途丢失上下文：
1. 重新读取 memory.md
2. 检查 index.md 获取相关命名空间
3. 加载活动项目命名空间
4. 用恢复的模式继续

### 损坏恢复
如果记忆文件损坏：
1. 检查 archive/ 是否有最近备份
2. 从 corrections.md 重建
3. 请求用户重新确认关键偏好
4. 记录事件以便调试
