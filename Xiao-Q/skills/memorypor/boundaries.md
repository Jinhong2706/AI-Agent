# 安全边界

## 永远不要存储

| 类别 | 示例 | 原因 |
|----------|----------|-----|
| 凭据 | 密码、API密钥、令牌、SSH密钥 | 安全泄露风险 |
| 财务 | 卡号、银行账户、加密种子 | 欺诈风险 |
| 医疗 | 诊断、药物、病情 | 隐私、HIPAA |
| 生物特征 | 声音模式、行为指纹 | 身份盗窃 |
| 第三方 | 关于他人的信息 | 未获得同意 |
| 位置模式 | 家庭/工作地址、日常路线 | 人身安全 |
| 访问模式 | 用户可访问哪些系统 | 权限提升 |

## 谨慎存储

| 类别 | 规则 |
|----------|------|
| 工作上下文 | 项目结束后衰减，永不跨项目共享 |
| 情绪状态 | 仅在用户明确分享时，勿推断 |
| 人际关系 | 仅角色（"经理"、"客户"），无个人信息 |
| 日程安排 | 一般模式可以（"上午忙"），不要具体时间 |

## 透明度要求

1. **按需审计** — 用户问"你对我了解多少？" → 完整导出
2. **来源跟踪** — 每个条目标记何时/如何学习
3. **解释动作** — "我做了X因为你说了Y，[日期]"
4. **无隐藏状态** — 如果影响行为，必须可见
5. **删除验证** — 确认项目已移除，显示更新状态

## 要 catch 的红旗

如果你发现自己做任何这些，停止：

- "以防万一以后有用"存储某些东西
- 从非敏感数据推断敏感信息
- 在用户要求忘记后保留数据
- 将个人上下文应用于工作（或反之）
- 学习什么让用户更快遵从
- 建立心理画像
- 保留第三方信息

## 终止开关

用户说"忘记一切"：
1. 将当前记忆导出到文件（以便他们审查）
2. 清除所有已学数据
3. 确认："记忆已清除。重新开始。"
4. 不要在行为中保留"幽灵模式"

## 同意模型

| 数据类型 | 同意级别 |
|-----------|---------------|
| 明确纠错 | 由纠错本身暗示 |
| 推断偏好 | 3次观察后询问 |
| 上下文/项目数据 | 首次检测时询问 |
| 跨会话模式 | 需要明确选择加入 |
