# 记忆操作

## 用户命令

| 命令 | 别名 | 动作 |
|---------|-------------|--------|
| "你对X了解多少？" | `/memory query X` | 搜索所有层级，返回匹配项及来源 |
| "显示我的记忆" | `/memory show` | 显示 memory.md 内容 |
| "显示[项目]模式" | `/memory project <name>` | 加载并显示特定命名空间 |
| "显示领域模式" | `/memory domain <name>` | 加载并显示特定领域 |
| "忘记X" | `/memory forget X` | 从所有层级移除，确认删除 |
| "忘记一切" | `/memory reset` | 完全清除（带导出选项） |
| "最近有什么变化？" | `/memory recent` | 显示最近20条纠错 |
| "记忆统计" | `/memory stats` | 显示每层大小、健康状态 |
| "导出记忆" | `/memory export [path]` | 生成ZIP存档 |
| "导入记忆" | `/memory import <file>` | 从存档恢复 |
| "记忆健康" | `/memory health` | 检查问题和建议 |
| "显示待处理" | `/memory pending` | 显示待处理区条目 |

## 脚本命令

对于自动化执行，使用提供的脚本：

```bash
# 统计（带进度条和健康检查）
bash ~/self-improving/scripts/stats.sh [--json]

# 导出（创建带清单的ZIP存档）
bash ~/self-improving/scripts/export.sh [output-path]

# 导入（带备份和试运行支持）
bash ~/self-improving/scripts/import.sh <export-file> [--merge|--dry-run]

# 心跳（保守记忆维护）
bash ~/self-improving/scripts/heartbeat.sh

# 安装（自动化安装）
bash <skill-path>/scripts/setup.sh [--tier low|normal|high|heavy] [--dir <path>]
```

## 记忆统计输出

在 `/memory stats` 命令时，输出：

```
📊 自我改进记忆

层级：high（高频）

热层 (memory.md):
  [####################] 95% (190/200行)

温层：
  纠错：  [################----] 80% (160/200)
  待处理：[####----------------] 20% (20/100)
  项目：   3个文件
  领域：   2个文件

冷层：
  归档：   5个文件

健康状态：healthy（健康）
```

### 健康状态

| 状态 | 标准 | 动作 |
|-------|----------|--------|
| healthy（健康） | 所有层级 <80% | 无需操作 |
| warning（警告） | 任何层级 80-99% | 建议审查 |
| critical（危急） | 任何层级 >=100% | 需要立即处理 |

### JSON输出

使用 `--json` 标志进行编程访问：

```bash
bash ~/self-improving/scripts/stats.sh --json
```

## 自动操作

### 会话开始时
1. 加载 memory.md（热层）
2. 检查 index.md 获取上下文提示
3. 如果检测到项目 → 预加载相关命名空间

### 收到纠错时
```
1. 解析纠错类型（偏好、模式、覆盖）
2. 检查是否重复（存在于任何层级）
3. 如果是新：
   - 添加到 corrections.md，带时间戳
   - 增加纠错计数器
4. 如果是重复：
   - 增加计数器，更新时间戳
   - 如果计数器 >= 3：询问确认为规则
5. 确定命名空间（全局、领域、项目）
6. 写入相应文件
7. 更新 index.md 行数
```

### 模式匹配时
应用已学模式时：
```
1. 找到模式来源（file:line）
2. 应用模式
3. 引用来源："使用X（来自 memory.md:15）"
4. 记录使用情况以进行衰减跟踪
```

### 每周维护（Cron）
```
1. 扫描所有文件，查找衰减候选
2. 将 >30天未使用的移到温层
3. 将 >90天未使用的归档到冷层
4. 如果任何文件 >限制，运行压缩
5. 更新 index.md
6. 生成每周摘要（可选）
```

## 文件格式

### memory.md（热层）
```markdown
# 自我改进记忆

## 已确认偏好
- format：使用要点而非散文（2026-01确认）
- tone：直接，不含糊糊（2026-01确认）

## 活动模式
- "looks good" = 批准继续（使用15次）
- 单个emoji = 已确认（使用8次）

## 最近（最近7天）
- MVP更喜欢SQLite（02-14纠错）
```

### corrections.md
```markdown
# 纠错日志

## 2026-02-15
- [14:32] 将冗长解释 → 改为要点总结
  类型：communication（沟通）
  上下文：Telegram回复
  确认：pending（待处理）(1/3)

## 2026-02-14
- [09:15] MVP使用SQLite而非Postgres
  类型：technical（技术）
  上下文：数据库讨论
  确认：yes（是的）（说"总是"）
```

### projects/{name}.md
```markdown
# 项目：my-app

继承：global（全局）, domains/code（代码领域）

## 模式
- 使用Tailwind（项目标准）
- 不使用Prettier（只用eslint）
- 通过GitLab CI部署

## 覆盖
- 分号：是（覆盖全局的无分号）

## 历史
- 创建：2026-01-15
- 最后活动：2026-02-15
- 纠错：12条
```

## 边缘情况处理

### 检测到矛盾
```
模式A："使用Tab"（全局，已确认）
模式B："使用空格"（项目，今天纠错）

解决方案：
1. 项目覆盖全局 → 此项目使用空格
2. 在 corrections.md 中记录冲突
3. 询问："空格只适用于此项目还是所有地方？"
```

### 用户改变主意
```
旧："总是使用正式语气"
新："其实，非正式也可以"

动作：
1. 用时间戳归档旧模式
2. 将新模式作为试探性添加
3. 保留归档以供参考（"你之前更喜欢正式"）
```

### 上下文模糊
```
用户说："记住我喜欢X"

但哪个命名空间？
1. 检查当前上下文（项目？领域？）
2. 如果不明确，询问："这应该全局适用还是只在这里？"
3. 默认为最具体的活动上下文
```
