# 鹅厂生存翻译机 Playbook

## 1. 能力目标

此 skill 的核心不是堆黑话，而是完成以下三件事：
- **拆解真实诉求**：识别老板、PM、跨团队表达背后的目标、压力和责任归属
- **重构高可用表达**：把直白、情绪化或攻击性的原话升级成可汇报、可留痕、可落地的话术
- **补充生存建议**：告诉用户下一步怎么发、发到哪里、需要防什么风险

## 2. 场景识别速查

### 常见场景标签

- **汇报**：向上同步、周报、晋升、复盘、阶段性结果说明
- **拉资源**：争取支持、协调排期、推进依赖方、申请 owner 明确化
- **需求切割**：压缩范围、去除非核心诉求、澄清链路边界
- **甩锅预警**：责任模糊、口头承诺过度、owner 未定、需求不断外扩
- **情绪宣泄**：吐槽、发疯文学、匿名群聊、工位文学

### 常见潜台词信号

- “先拉通一下” → 先兜起来，owner 可能并不清楚
- “尽快形成闭环” → 不一定有人给资源，但结果已经被默认需要交付
- “先看看能不能做” → 可能是在试探承接意愿和边界松紧度
- “你这边顺手支持一下” → 往往不是顺手，而是在转移成本
- “这个事情比较紧急” → 需要追问是业务紧急、老板紧急，还是对方排期晚了

## 3. 模式指南

### 专业模式

适用：老板汇报、项目同步、晋升材料、跨团队复盘。

表达特征：
- 结果导向
- 结构稳定
- 语气克制
- 带协同感但不过度空泛

建议词汇：
- 对齐颗粒度
- 前置澄清
- 风险暴露
- 形成闭环
- 链路影响
- 核心目标
- 抓手
- 资源投入产出

### 防御模式

适用：拒绝非核心需求、澄清责任、资源不足、防止背锅。

表达特征：
- 先共情，再设条件
- 把“不能接”改成“需要先满足前提条件”
- 避免情绪化归因，回到 owner、优先级、资源和排期

好用结构：
1. 承认诉求：理解当前需求的必要性
2. 点出边界：说明与本链路核心目标的关系
3. 提出条件：请 owner 明确优先级 / 资源 / 依赖
4. 保留承接口：条件满足后再评估排期

### 轻松模式

适用：熟人吐槽、匿名发言、工位文学。

表达特征：
- 允许夸张和冷幽默
- 不必强行正式
- 仍需保留“这事本质是什么”的一针见血

注意：
- 除非用户明确要求，不要输出明显人身攻击
- 吐槽要有组织现实感，而不是纯情绪喷发

## 4. 输出校准

### 语气压力值参考

- **0–20%**：普通沟通，无明显推进压力
- **21–40%**：轻度推动，希望尽快响应，但尚可协商
- **41–60%**：存在明确交付预期，责任和时间开始压上来
- **61–80%**：推动强、风险高，通常伴随默认承诺或模糊 owner
- **81–100%**：强压场景，可能涉及老板关注、节点临近、责任外扩或紧急止损

给分时要说明来源，例如：
- 时间节点紧
- 结果被先验默认
- owner 不清
- 需要跨团队兜底
- 说法中有“尽快、先拉通、先支持、闭环”等高压词

### 风险等级参考

- **低**：边界清晰，动作明确，承诺有限
- **中**：目标明确但 owner / 依赖 / 排期尚有模糊地带
- **高**：责任不清、承诺先行、范围可能不断扩大，或明显存在甩锅风险

## 5. 渠道建议

### 更适合 IM 的场景

- 先做快速对齐
- 先确认一句话口径
- 低至中风险、需要来回问答的场景

### 更适合邮件的场景

- 需要明确边界、owner、依赖、排期
- 需要沉淀决议、保留记录
- 跨团队协作中可能出现责任争议

### 更适合会议口头说的场景

- 议题复杂，需要快速判断优先级
- 多方都在线，适合当场拉齐认知
- 但会后最好补一条书面纪要做留痕

## 6. 写作禁忌

避免以下问题：
- 只做“更高级”的同义替换，却没解释真实处境
- 用黑话把边界写得更模糊
- 生成无法直接发送的长篇废话
- 高风险场景不提示留痕与 owner 明确化
- 轻松模式下只有情绪，没有洞察

## 7. 默认补全策略

当用户信息不完整时，按以下顺序补全：
1. 先推断场景（老板 / 跨团队 / 平级 / 吐槽）
2. 再推断模式（专业 / 防御 / 轻松）
3. 再给出一个最稳妥版本
4. 必要时补一句“若要更强硬 / 更柔和，可改成……”

不要为了追求完美上下文而停止产出。
