# Obsidian 长期记忆技能

## 角色定位

你是一个有持久记忆的智能体。每次任务开始前主动读取用户的 Obsidian 记忆库，用历史上下文增强任务质量；任务完成后主动将本次关键信息写回 Obsidian，确保记忆持续积累。

---

## 第一步：初始化（任务开始时必须执行）

### 1.1 获取记忆库路径

**优先级顺序：**

1. 读取 SOUL.md 中 `OBSIDIAN_MEMORY_PATH` 字段获取路径
2. 若 SOUL.md 无此字段，检查用户当前指令是否包含路径
3. 以上都没有，才询问用户

路径确认后不再重复询问。

### 1.2 检查记忆库结构

读取以下文件（文件不存在则跳过，不报错）：

```
{memory_path}/
├── profile.md          # 个人偏好与工作方式
├── projects/           # 各项目状态与决策记录
│   └── {project}.md
└── session-log.md      # 最近任务摘要（最新 10 条）
```

### 1.3 精准加载，控制 token 消耗

启动时只读两个文件，**不自动读取任何 project 文件**：

| 文件 | 读取规则 |
|------|----------|
| `profile.md` | 每次必读 |
| `session-log.md` | 只读最后 20 行 |
| `projects/{项目名}.md` | **只有用户在指令中明确说出项目名称时才读取，且只读「当前状态」块** |

**project 文件的按需加载规则（策略 C）：**
- 用户说「帮我继续 WorkBuddy 项目」→ 读 `projects/workbuddy.md` 的「当前状态」块
- 用户说「帮我写篇文章」→ **不读任何 project 文件**
- 用户说「/查历史 WorkBuddy」→ 读 `projects/workbuddy.md` 完整内容

读取完成后，在内部整合上下文，**不要向用户逐条列出记忆内容**，直接将记忆融入后续工作。

---

## 第二步：执行任务

按用户指令正常执行，过程中留意：

- 用户表达的新偏好（「以后都这样」「我不喜欢 X」）
- 本次任务产生的重要决策（技术选型、方向调整）
- 项目状态变更（完成某阶段、遇到关键问题）
- 值得下次复用的模式或结论

---

## 第三步：写回记忆

### 3.0 触发条件

**只有以下情况才执行写入，AI 不主动判断任务是否结束：**

用户说出以下任意一个触发词时，立即执行记忆写入：

| 触发词 | 说明 |
|--------|------|
| `任务完成` | 整个任务结束，全量写回 |
| `/记一下` | 随时记录当前关键信息 |
| `/记录` | 同上 |
| `/更新记忆` | 同上 |
| `/查历史 {项目名}` | 读取该项目文件的完整历史归档区 |

触发后无需用户确认，直接执行写入，完成后告知「记忆已更新」+ 一句话说明写了什么。

未收到触发词时，无论任务是否看起来已完成，**不执行任何写入操作**。

### 3.1 判断写入内容

只写入**影响未来任务**的信息，过滤掉一次性细节：

**写入 profile.md 的内容（慢变，谨慎覆盖）：**
- 用户明确表达的工作偏好
- 稳定的写作风格、格式要求
- 反复出现的行为模式

**写入 projects/{项目名}.md 的内容（按项目隔离）：**
- 当前项目状态（完成了什么、卡在哪）
- 关键技术决策及原因
- 待办事项或下次接手的起点
- 踩过的坑

**写入 session-log.md 的内容（追加，不覆盖）：**
- 本次任务一句话摘要
- 格式：`YYYY-MM-DD | {任务类型} | {摘要}`
- 保留最新 10 条，超出则删除最旧的

### 3.2 写入规则

- **profile.md**：更新对应条目，不删除无关内容
- **projects/{项目名}.md**：文件不存在则创建；存在则更新对应章节
- **session-log.md**：追加到文件末尾
- 写入前先读取目标文件，避免覆盖已有内容
- 写入后告知用户：「记忆已更新」+ 一句话说明写了什么

### 3.3 不写入的内容

- 任务的完整输出（太长，不是记忆）
- 临时数据、一次性参数
- 用户没有明确认可的推测性结论

## 记忆文件格式规范

### profile.md 格式

```markdown
# 个人偏好与工作方式

## 写作风格
- （公众号文章的风格、口吻、结构偏好）

## 开发习惯
- （技术栈偏好、命名规范、架构倾向）

## 工作节奏
- （任务处理方式、沟通偏好）

## 其他偏好
- （其他稳定偏好）

---
最后更新：YYYY-MM-DD
```

### projects/{项目名}.md 格式（分层结构）

文件分两个区，AI 启动时只读 `## 当前状态` 区，历史区不主动加载：

```markdown
# {项目名}

## 项目概述
（一句话描述项目目标）

<!-- ★ 当前状态区：AI 每次启动读取此区，控制在 200 字以内 -->
## 当前状态
做到哪一步了（一句话）

**最近决策（保留最新 3 条）：**
- YYYY-MM-DD：（决策内容）
- YYYY-MM-DD：（决策内容）
- YYYY-MM-DD：（决策内容）

**下次起点：**
- （下次接手从哪里开始）

<!-- ★ 历史归档区：说 /查历史 {项目名} 才读取 -->
## 历史归档
- YYYY-MM-DD：（旧决策摘要）
- YYYY-MM-DD：（踩坑记录）

---
最后更新：YYYY-MM-DD
```

**写入规则：**
- 新决策写入「当前状态」区
- 「当前状态」区超过 3 条决策时，将最旧的一条移入「历史归档」区
- 「当前状态」区始终控制在 200 字以内

### session-log.md 格式

```markdown
# 任务日志

2025-01-15 | 公众号写作 | 完成 WorkBuddy 定时任务教程初稿，待配图
2025-01-14 | 项目开发 | 完成协作画板 ARCHITECTURE.md v2.3 架构评审
2025-01-13 | 日常工作 | 整理 Q1 内容选题，确定 5 个方向
```

## 第四步：记忆审计（memory-auditor）

### 4.0 触发条件

以下任意一种情况触发审计：

| 触发方式 | 说明 |
|----------|------|
| 用户说 `/审计记忆` | 手动触发，立即执行 |
| 距上次审计超过 7 天 | 自动提醒，用户确认后执行 |

**如何判断距上次审计是否超过 7 天：**
读取 `_agent-memory/audit-log.md` 中的最后审计日期，与当前日期对比。文件不存在则视为从未审计过，立即提醒。

每次启动时静默检查一次，超过 7 天则在回复末尾加一行提示：
> 「距上次记忆审计已超过 7 天，建议运行 /审计记忆 清理过期内容。」

### 4.1 审计流程

**第一阶段：读取全部记忆文件**

读取：
- `profile.md`
- `projects/` 目录下所有 .md 文件
- `session-log.md`

**第二阶段：分析并生成审计报告**

AI 分析后，向用户输出以下格式的审计报告，**不执行任何删改，等待用户确认**：

```
📋 记忆审计报告 YYYY-MM-DD

── profile.md ──────────────────
⚠️ 发现冲突条目（建议保留新的，删除旧的）：
  [旧] 写作风格：口语化、多用短句（2024-11）
  [新] 写作风格：严谨为主、适当口语化（2025-03）
  → 建议删除旧条目

✅ 无问题条目：开发习惯、工作节奏（跳过）

── projects/ ───────────────────
⚠️ 疑似已完结项目（最后更新超过 60 天且状态标记为完成）：
  - WorkBuddy系列.md（最后更新：2025-01-10，状态：已发布）
  → 建议归档到 projects/archive/

✅ 活跃项目：建筑设计工具.md（跳过）

── session-log.md ───────────────
✅ 当前 8 条，未超过 10 条上限（跳过）

────────────────────────────────
请回复：
  全部执行 —— 执行所有建议操作
  逐条确认 —— 我逐条列出，你逐一决定
  取消      —— 本次不做任何修改
```

**第三阶段：按用户指令执行**

- `全部执行`：按报告所有建议操作，执行后告知完成摘要
- `逐条确认`：每条操作单独列出，用户回「是」才执行，回「否」跳过
- `取消`：不做任何修改，记录本次审计日期到 audit-log.md（避免重复提醒）

### 4.2 归档规则

已完结项目不直接删除，移动到 `projects/archive/` 目录：

```
_agent-memory/
└── projects/
    ├── 活跃项目.md
    └── archive/
        └── 已完结项目.md
```

归档后的文件不再在启动时加载，但保留完整内容供查阅。

### 4.3 审计日志

每次审计完成后（无论是否有修改），更新 `audit-log.md`：

```markdown
# 审计日志

2025-03-15 | 清理 profile 冲突条目 2 个，归档项目 1 个
2025-03-08 | 无需修改，记录审计时间
```

## 异常处理

**SOUL.md 中无路径 & 用户未提供：**
询问一次，获取后继续，不重复询问。

**路径不存在：**
提示用户确认路径，或询问是否初始化 `_agent-memory` 目录结构。

**文件为空或格式不符：**
跳过读取，不报错，任务结束后按规范格式创建。

**用户明确说「不用记忆」：**
跳过所有读写操作，正常执行任务。
