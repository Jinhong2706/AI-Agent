# 案例池构建（阶段 2 · Agent 1 角色）

立题对齐后，进入案例池构建。目标是横向扫一遍候选案例，建立结构化清单，为单案分析做准备。

## 工作方式

### 1. 检索来源（按优先级）

1. **行业奖项与榜单**：TGA 提名 / TGS / DICE Awards / PlayStation Partner Awards / Steam 年度榜 / SensorTower 月度榜
2. **行业媒体年度复盘**：GamesIndustry.biz / Eurogamer / The Verge / IGN 年度发行案例总结
3. **平台官方公告汇总**：PlayStation Blog / Xbox News / Nintendo Direct 历史回顾
4. **数据源横向比对**：SteamDB（PC）/ Newzoo / CIRP（主机）/ SensorTower（手游）
5. **垂直媒体 / 社区**：4Gamer / Famitsu（日本）/ 机核 / GameLook（中国）/ Reddit r/Games r/PS5

### 2. 案例筛选标准

每个备选案例需满足至少一项：
- 销量 / 活跃 显著（例如周销量 > 50 万 / 月活 > 100 万）
- 明显爆冷或翻车（反面教材也有研究价值）
- 有明显特殊发行动作（如突击上线 / 限定硬件 / GP Day1 / 全平台同步等）
- 在用户立题边界内有代表性（每个细分类别至少 1-2 个）

### 3. 案例池清单字段（必备）

| 字段 | 说明 |
|---|---|
| **编号** | F01 / C01 等（按主类别 + 序号） |
| **中文名** | 海外游戏强制中文名 |
| **英文/日文原名** | 副标题 |
| **厂商 / 发行方** | 研发方 vs 发行方分开写 |
| **品类** | 具体到子类（如 PvE 合作射击 / PvP 大逃杀） |
| **平台 / 首发时间** | 多平台分别列时间，标注独占性质 |
| **定价 + 商业模式** | F2P / 付费 / 订阅 / 实体版本 |
| **代表传播素材** | 关键梗 / 标语 / 节点（如"上海保卫战" / "打工人"） |
| **初判标签** | 第一次扫的时候打的临时标签——用于综合洞察阶段归纳分类 |
| **数据来源链接** | 至少 1 个权威源 |

### 4. 案例池产出物

输出 `01_case_pool/案例池清单.md`，结构：

```markdown
# 研究案例池清单

> 立题确认：[链接到研究立题确认]
> 案例池目标数量：X 个
> 实际收录：Y 个

## 案例总览表

| # | 中文名 | 厂商 | 品类 | 平台/首发 | 定价 | 关键词 | 初判标签 | 来源 |
|---|---|---|---|---|---|---|---|---|
| F01 | 案例游戏 A | 厂商 X | 开放世界 ARPG | PS4 YYYY.MM / PS5 YYYY.MM | F2P | 同步首发主机 / 平台 Awards / 启动套装 / Pro 增强 | 长跑深度合作 | [来源](...) |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

## 备选但不列入清单的案例（说明排除理由）

| 中文名 | 排除理由 |
|---|---|
| 某游戏 | 数据不足/无代表性/与研究边界不符 |
```

### 5. 关键检查

进入阶段 3 单案分析前，案例池必须满足：
- 每个细分类别至少 1-2 个
- 既有成功案例也有反面案例（反面只入池，不单独立章）
- 中文名都已统一（详见 `checklists/中文名规范对照表.md`）
- 海外游戏 + 国内游戏比例符合立题边界
- 数量在用户确认的范围内（不要过度扩张）

### 6. 用户复核

案例池清单完成后，**必须先让用户过一遍**，确认：
- 哪些案例他知道、哪些不知道（不知道的需补背景）
- 是否有遗漏的关键案例
- 是否有冗余可删除

确认后再进入阶段 3。
