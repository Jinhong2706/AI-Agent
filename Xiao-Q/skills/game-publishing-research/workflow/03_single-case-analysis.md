# 单案分析（阶段 3 · Agent 2 角色）

每个案例独立产出一份 MD。**单案分析不归纳跨案规律**——那是阶段 4 的事。本阶段重点是把单个案例的事实、动作、数据、来源沉淀成可复用的资产。

## 标准结构（与 templates/单案分析模板.md 对齐）

每份 MD 必须包含以下 6 个部分：

### 1. 元信息头
```
# 案例_{编号}_{中文名}

> 厂商 / 发行方：
> 品类：
> 平台 / 首发时间：
> 定价 / 商业模式：
> 在本研究中的初判标签：
```

### 2. 核心问题（一句话）
> 一句话讲清楚这款游戏在发行上做对/做错了什么、为什么值得研究。

### 3. 关键动作（按时间顺序或按维度）

按时间或维度展开发行团队**实际做了什么**：
- 上线节奏（多平台分别时间、独占性质）
- 平台资源（SoP / Direct / Showcase / 商店推荐位 / 限定硬件 / 会员包）
- KOL / 主播 / 社区运营
- 跨界 / IP 联动 / 文化叙事
- 首发后的长尾运营动作

每个动作必须能回答："**他们具体做了什么**？" 而不是只写"做了什么类型"。

### 4. 数据点（销量 / 活跃 / 评分 / 节点）

| 数据点 | 数据 | 来源 |
|---|---|---|
| 首日销量 | 250 万 | [Insomniac 官方公告](...) |
| 首月销量 | 500 万 | ... |
| MC 评分 | 96 | ... |

### 5. 对发行方的参考意义（必填）

每个案例最后必须落到这一节，回答两个子问题：
- **可借鉴**：什么动作是其他发行方可以套用的？
- **不可复制**：什么是平台 / IP / 时机决定的，不能直接抄？

### 6. 数据来源与参考资料（必填）

```
## 📚 数据来源与参考资料
- Wikipedia: [链接]
- 官方公告: [链接]
- 行业媒体: [链接]
- 数据源（SteamDB / SensorTower 等）: [链接]
```

来源至少 3 条；如果是关键数据点（销量、合作时间、独占期），来源必须是官方或权威数据源，不能只有 Wikipedia。

## 写作硬规则（贯穿单案分析）

### 中文名优先
所有海外游戏在标题、正文都用中文名。英文名只在第一次出现时作为副标题或括号注释（参考 `checklists/中文名规范对照表.md`）。

### 真实性降级
对官方未确认的归因 / 动机：
- ❌ 错：「Apex 突击上线是因为怕媒体提前曝免费模式」
- ✅ 对：「行业普遍说法是 Apex 突击上线为了避开 Anthem/Black Out 节奏 + 担心常规营销被先入为主拉低预期；EA/Respawn 没有公开材料确认核心动机」

### 关键传播素材原话保留
对真实在社区流传的素材，必须保留原话——这些是案例的关键资产：
- 绝地潜兵 2 的「Cyber 淞沪保卫战 / 上海不能丢」
- 幻兽帕鲁的「打工人 / 牛马 / 996 牛马 / 奴隶模拟器」
- 天国：拯救 2 的「24 小时回本」

HTML 转化时用 `<span class="highlight">` 黄底高亮。

### 不预设结论
单案分析阶段不要去归纳"它属于路径 X"——分类是阶段 4 的事。本阶段只描述发生了什么、有什么数据、来源是什么。

## 单案分析的标准产物

每个案例输出一份独立 MD，文件命名规则：

```
02_case_analysis/
├── 案例_{编号}_{中文名}.md
├── 案例_F01_案例游戏A.md
├── 案例_F03_案例游戏C.md
├── 案例_C02_绝地潜兵2.md
└── ...
```

如果案例使用了内部一手资料 / 不可外发的报告：

```
shared/internal_reports/
├── （内部报告文件）
└── ⚠️ 这些禁止外发，引用时只参考方法论不直接引用具体数字
```

## 用户复核节点

每完成 5-10 份单案 MD，让用户过一遍。重点确认：
- 中文名是否统一
- 关键事实是否准确（用户作为行业内人可能直接看出错处）
- 来源链接是否权威
- 写作风格是否符合期望

不要等全部 30 份写完再让用户看——迭代成本太高。
