# 交付物视图（阶段 5 · 可选）

如果用户要求把研究结果做成可视化交付物（HTML 集合页 / PPT / 智能表），按本节执行。

## HTML 集合页规范

### 文件结构

单文件 HTML，**不依赖外部 CSS / JS / 图片资源**——所有样式内嵌、所有图片用 emoji / 几何 CSS 替代。

理由：
- 单文件可以双击直接打开
- 可以发邮件 / 微信 / 企业内部网盘传输
- 不用部署服务器
- 任何浏览器无网络都能看

### 整体布局

```
┌──────────────────────────────────────────┐
│ [左侧 sidebar 暗色固定]│ [主内容区浅灰背景]│
│  ━━━━━━━━━━━━━━━     │  ┌─────────────┐ │
│  研究输出              │  │ Hero 渐变区  │ │
│  ● 综述                │  └─────────────┘ │
│  ▸ 主洞察 + 子锚点     │  正文章节         │
│  ▸ 主洞察 + 子锚点     │  路径卡 / 表格    │
│  ━━━━━━━━━━━━━━━     │  结论框           │
│  案例库（N 个游戏）    │  数据来源区块     │
│  🎮 路径 1 · 案例 A    │                   │
│  🎮 路径 1 · 案例 B    │                   │
│  ...                   │                   │
└──────────────────────────────────────────┘
```

### 视觉规范

| 元素 | 规范 |
|---|---|
| **字体** | 微软雅黑 / Arial 优先 |
| **背景** | `#f8f9fa` 浅灰 + 微弱几何纹理 |
| **侧边栏** | `#1f2937` 暗色背景 + 白色文字 |
| **Hero 区** | 每个视图不同渐变色（综述蓝 / F2P 绿 / 付费 3A 橙 / 案例库各自色） |
| **路径卡** | `border-left:4px solid {彩色}` 区分每条路径 |
| **结论框** | 绿底 `#d1fae5` + 💡 |
| **警告框** | 红底 `#fee2e2` + ⚠️ |
| **关键素材高亮** | `<span class="highlight">` 黄底 `#fef08a` |
| **表格** | hover 行高亮 + 表头深色 |
| **导航 emoji** | 🎮 F2P 案例 / 🕹️ 付费 3A 案例 / 🎯 应用层建议 |

### 路由 JS

侧边栏 nav-item `data-view` 属性 → 主区 `id="view-{view}"`。点击切换 view，子锚点 `data-anchor` 平滑滚动到 `id="{anchor}"`。

参考 `templates/HTML集合页模板.html`。

### 必备视图

每个集合页至少包含：
1. **综述**——研究方法说明 + 覆盖范围 + 整体结论速览
2. **主洞察视图**（综合洞察 MD 转化）—— 每个独立路径都要有锚点
3. **应用层视图**（如有）—— 用研究结论给具体产品做策略建议
4. **案例库视图**（每个案例独立页）—— 命名"路径 N · 中文名"

### 案例视图独立页结构

每个案例独立 view section，统一结构：
- Hero 标题区（彩色渐变）
- 核心问题
- 策略总结表
- 关键动作详解（按 ①②③… 编号）
- 对发行方的启示
- **📚 数据来源与参考资料**（每个具体数字附可点开链接）

### 关键传播素材黄底高亮

凡是真实在社区流传的传播素材原话，必须用 `<span class="highlight">` 高亮。这是案例 HTML 最有传播价值的部分。

## PPT 交付规范（如果用户要求）

PPT 不是 HTML 的替代品——它适用于汇报场景。规范：
- 每页一个核心论点（一页一句话）
- 标题避免口语化表达
- 数据用图表 + 数字双层呈现
- 关键传播素材原话保留
- 文末附数据来源 1 页

不主动建议用户做 PPT——除非用户明确说要汇报。

---

## 智能表格交付规范（如果用户要求）

如果用户要求把案例池或综合洞察转成智能表格（如腾讯文档智能表 / 飞书多维表 / Notion / Airtable）：
- 字段必须结构化、可筛选、可排序
- 每行一个案例，列必须包含来源链接
- 不要把综合洞察的"路径分类"当成固定字段——它会变化

---

## 交付物用户复核

每完成一份交付物（HTML / PPT / 智能表），让用户过一遍：
- 视觉是否符合期望
- 是否需要更深 / 更浅
- 哪些章节顺序需要调整
- 是否要继续延伸（例如增加新视图、新案例）

不要一次性把所有内容塞进交付物——分阶段产出 + 迭代反馈是更稳的节奏。
