# 写作硬规则清单

每次产出 MD / HTML 之前必跑此清单。这些规则来自一次完整研究项目的反复纠正经验，写进 skill 后就一次性固化。

## 1. 一句话先行
每个路径、每个案例、每个章节都先一句话讲清楚是什么，再展开细节。
- ❌ 错：直接开始堆数据 / 堆动作清单
- ✅ 对：先一句话核心逻辑，再展开

## 2. 中文名优先
所有海外游戏名一律使用中文名。详见 `checklists/中文名规范对照表.md`。
- ❌ 错：Helldivers 2 / Marvel Rivals / Black Myth: Wukong
- ✅ 对：绝地潜兵 2 / 漫威争锋 / 黑神话：悟空
- 例外：英文名在第一次出现时可作副标题或括号注释

## 3. 路径前缀
导航栏案例命名格式：「路径 N · 游戏中文名」
- ✅ 对：路径 1 · 案例游戏 A / 路径 5 · 案例游戏 X
- 让读者一眼能看出归属路径

## 4. 决策选项化提问
追问用户时给 A/B/C 选项，不开放式问。
- ❌ 错：「你想分什么类」
- ✅ 对：「以下三个方向你倾向哪个：A 按平台分 / B 按资源类型 / C 按发行动作类型」

## 5. 反面教材只反衬
反面教材（翻车的案例）不单独立章，只在某条路径里顺带反衬。
- ❌ 错：单独写「四大反面教材」一章
- ✅ 对：在路径 1 的发行启示里加一句"反面参考：超凡大师 PvP 多人付费进 F2P 红海，11 天关服"

## 6. 关键传播素材原话保留
真实在社区流传的素材原话必须保留——这些是案例的关键资产。HTML 用 `<span class="highlight">` 黄底高亮。
- 「Cyber 淞沪保卫战 / 上海不能丢」
- 「打工人 / 牛马 / 996 牛马」
- 「24 小时回本」

## 7. 对发行方的参考意义
每个案例最后必须落到「对发行方的参考意义」，不是只复盘案例本身。
- 必须包含：可借鉴 + 不可复制（拆开两点写）

## 8. 真实性降级
官方未确认的归因 / 动机必须降级处理，不能写死。
- ❌ 错：「Apex 突击上线是因为怕媒体提前曝免费模式」
- ✅ 对：「行业普遍说法是 X，但 EA/Respawn 没有公开材料确认核心动机」

HTML 用 `class="warning-box"` 红色警告框。

## 9. 数据来源链接
每个具体数字、合作时间、销量数据、平台资源声明必须附可点开来源链接。
- 优先级：官方 IR > 平台官方 > 权威数据源 > 主流媒体 > 二级讨论
- 二级讨论只能作为线索，不能作为定论

## 10. 写作风格
- 偏正式汇报风
- 标题避免口语化表达
- 摘要要精炼、自包含、面向非专家读者理解
- 不需要跨游戏对比分析（除非综合洞察阶段）

## 11. 文档保留原文
文档翻译/编辑时尽量保留原文内容不过度精简，兼顾排版美观与内容完整性。

## 12. 提问前先尝试自己解决
研究中遇到不确定的事实，先用 web_search / web_fetch 查权威源，**先尝试自己解决**再问用户。除非是研究方向 / 边界问题——那必须问用户。

## 13. 文件命名规范
- 单案分析：`案例_{编号}_{中文名}.md`
- 内部报告：`shared/internal_reports/Agent1_内部报告_{编号}_{中文名}_{来源}.md`
- 综合洞察：`Agent3_{研究主题}综合洞察.md`
- HTML 集合页：`{研究主题}_集合页.html`
- 包含 agent 编号便于识别（用户偏好）
