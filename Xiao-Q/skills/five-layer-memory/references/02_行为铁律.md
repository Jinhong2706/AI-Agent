## I0. 初始化执行铁律（断点恢复 + 防编造）

> [!danger] 🚨 初始化是最容易出错的环节——AI凭记忆操作、编造题目、跳过步骤是三大高频问题。

### 规则 1：初始化启动前必须先读本文件

```
如果当前上下文中没有本技能的 SKILL.md：
    必须先读取 SKILL.md → 确认当前应执行的步骤 → 再开始操作
    不得凭"记忆"或"印象"继续
    不得跳过读取直接出题或写文件
```

### 规则 2：断点检测（每次进入初始化流程时强制执行）

```
断点检测流程：
  1. IDENTITY.md + USER.md 存在？ → ✅ 步骤2已完成
  2. 公共文件夹（~/[记忆共享中心]/）存在？ → ✅ 步骤3已完成
  3. CORE.md 存在且有内容？ → ✅ 问卷Q1-Q7已完成
  4. 用户基本规则.md 存在且有内容？ → ✅ 问卷Q8-Q23（部分）已完成
  5. 问卷答案汇总.md 存在？ → ✅ 全部问卷已完成
  6. ~/.workbuddy/rules/*.mdc 存在？ → ✅ 步骤6已完成
  7. 路由表.mdc 存在？ → ✅ 步骤7已完成
  8. Skill 文件存在？ → ✅ 步骤8已完成
  9. 洞察报告已生成？（询问用户） → ✅ 步骤9已完成
```

**检测到断点后的行为**：
- 已完成的步骤 → 静默跳过，告诉用户"检测到 XXX 已完成，跳过"
- 未完成的第一步 → 从该步骤继续
- **如果检测到问卷做到一半（如 CORE.md 存在但问卷答案汇总.md 不存在）**：
  - 先读取 `问卷答案汇总.md` 确认哪些题已答过
  - 再读取 `CORE.md` / `用户基本规则.md` 确认哪些答案已写入
  - 从第一道未答的题开始继续，**必须读本文件原样出题**
  - **禁止让用户重做已答过的题**

### 规则 3：严禁初始化过程中"凭记忆操作"

| 禁止行为 | 正确做法 |
|---------|---------|
| 凭记忆出题（题干/选项和原题不一致） | 先读本技能 `references/03_用户问卷Q1-Q33.md`，原样输出 |
| 凭记忆写文件（写入内容和模板不一致） | 先读本技能 `references/04_洞察报告模板.md` 或对应 reference，按模板写入 |
| 跳过步骤（如跳过问卷直接生成CORE.md） | 严格按 SKILL.md 中7步顺序执行 |
| 未检测断点就从头开始（让用户重做） | 先执行断点检测，从断点继续 |
| 对话中断后凭印象继续 | 新对话必须先读 SKILL.md，确认断点再继续 |

### 规则 4：用户质疑时的处理

```
用户说"题目不对"/"这不是原题"/"怎么又让我做这个"时：
  1. 立即停止当前操作
  2. 读取本技能 references/03_用户问卷Q1-Q33.md 核实原题/当前步骤
  3. 如实告知用户检测结果
  4. 从正确的位置继续
```

## I2. 修改清单确认制

> [!warning] 修改文件前必须执行

1. **输出 [修改清单]**：列出「文件名 + 改动位置 + 改动内容摘要」
2. **等用户确认**：收到确认后才写入，不得自行先写入
3. **写完验证**：确认写入无误
4. **报告结果**：给用户一句完成说明，不写长篇汇报

## I3. 先验证再下结论

> [!danger] 核心原则

> **实事求是是底线，不是口号。**

> 判断文件状态/数据是否存在/路径是否正确 → **必须先读/查，不许凭印象判断**。
> 看了才能说"有"或"没有"，读了才能说"完整"或"损坏"。

## I4. 方案提案铁律

> [!warning] 提方案后强制自问

1. **有没有共同盲区？** — 选项里是不是都假设了某个前提？
2. **有没有更简单的方案？** — 有没有因为沉没成本而不敢提出的选项？

> 有 → 补充第三个选项（通常是最简单的那个）。

## I5. 独立观察触发扩展

> [!note] 不只逻辑漏洞才触发

独立观察触发条件：
- 分析有盲区（信息不完整，结论可能有偏）
- 有不同角度（判断跟用户方向不同，值得说出来）
- 有疑虑（对自己的方案不完全确信，标注风险）

跳过条件："闭眼执行"或"忽略原则"

## I9. 自定义层管理规则

> [!success] 自定义记忆层指南

自定义层是五层记忆架构的扩展，用于存储五层无法覆盖的领域专精、项目特定、阶段性目标等内容。

### 何时需要自定义层

- **领域专精**：特定领域的知识、技巧、偏好
- **项目特定**：重要项目的架构、进度、决策
- **阶段性目标**：有明确截止日期的目标
- **持续跟进**：需要长期追踪但不属于五层记忆范畴的内容

### 管理规则

1. **按需创建**：不要预先创建大量自定义层，根据实际需要创建
2. **定期更新**：索引文件每周更新，确保状态准确
3. **TTL设置**：阶段性目标设置明确截止日期，到期自动评估是否归档
4. **联动机制**：项目进展可与周报、月报联动，形成闭环管理
5. **避免重复**：内容已在五层记忆中的（如情境层的目标），不要在自定义层重复存储

## G11. 记忆文件版本管理

> [!success] 版本控制机制

#### 版本管理方法

- **文件命名**：在文件名后添加版本号，如 `CORE.md.v1`
- **变更记录**：在文件末尾添加变更记录，记录每次修改的内容和日期
- **备份机制**：定期备份记忆文件，防止数据丢失
- **回滚机制**：当新版本出现问题时，能够快速回滚到之前的版本

#### 版本控制流程

1. **修改前备份**：在修改记忆文件前，先创建备份
2. **版本号更新**：修改完成后，更新文件版本号
3. **变更记录**：在文件末尾添加变更记录
4. **验证测试**：测试修改后的记忆文件是否正常工作
5. **定期清理**：定期清理旧版本文件，保持目录整洁
