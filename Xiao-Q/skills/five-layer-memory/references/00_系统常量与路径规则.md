## 第一部分：核心设计原则

### 多端协同架构

> [!important] 🔑 **设计前提：多端协同，不是单机专属**

本系统的核心设计目标是：**你可能在多个 AI 平台同时使用（WorkBuddy / Trae / Qclaw 等）**。
记忆数据不能锁死在任何一个平台的原生目录里，否则换平台=丢记忆。

**核心原则：平台无关的数据写公共文件夹，平台特有的机制用原生路径。**

## 第二部分：平台路径映射规则

### 平台检测方式

| 检测方式 | Unix/macOS | Windows |
|---------|-------------|---------|
| 环境变量 `HOME` | 存在，值为 `/Users/用户名` | 存在，值为 `C:\Users\用户名` |
| 环境变量 `USERPROFILE` | 不存在 | 存在，值为 `C:\Users\用户名` |
| 路径 `/Users/` 是否存在 | 存在 | 不存在 |
| 路径 `C:\Users\` 是否存在 | 不存在 | 存在 |

### `~/` 的跨平台映射

| 平台 | `~/` 实际映射路径 | 示例 |
|------|-------------------|------|
| macOS / Linux | `$HOME/` | `~/个人AI档案/` → `/Users/用户名/个人AI档案/` |
| Windows | `%USERPROFILE%\`（即 `C:\Users\用户名\`） | `~/个人AI档案/` → `C:\Users\用户名\个人AI档案\` |

> 💡 **记忆共享中心不在默认位置的情况**：
> 如果用户将记忆共享中心放在非默认位置（如 `D:\AI档案\`），**直接使用用户指定的完整绝对路径**，`~/` 映射规则不适用。

### 路径分隔符规则

| 平台 | 分隔符 | 正确示例 | 错误示例 |
|------|--------|---------|---------|
| macOS / Linux | `/` | `/Users/用户名/个人AI档案/CORE.md` | `C:\Users\...` `\` |
| Windows | `\` | `C:\Users\用户名\个人AI档案\CORE.md` | `/Users/...` `/` |

> ⚠️ **禁止混用分隔符**：Windows 路径中不得出现 `/`（除了盘符后的 `://` 协议头）。

## 第三部分：路径常量定义

### 第一组：公共文件夹路径（★ 多端共享数据）

| 常量名 | 路径 | 说明 |
|--------|------|------|
| `PATH_SHARED_ROOT` | `~/[记忆共享中心]/` | 公共文件夹根目录 |
| `PATH_CORE` | `~/[记忆共享中心]/核心层/CORE.md` | L1核心层（身份+价值观+底线） |
| `PATH_RULES_DIR` | `~/[记忆共享中心]/记忆规则/` | L2/L3 规则文件目录 |
| `PATH_RULES_USER` | `~/[记忆共享中心]/记忆规则/用户基本规则.md` | 用户基本行为规则 |
| `PATH_SNAPSHOT` | `~/[记忆共享中心]/记忆规则/00_动态状态快照.md` | L4 情境层当前状态 |
| `PATH_SHADOW` | `~/[记忆共享中心]/SHADOW.md` | L5 潜意识层（言行偏差） |
| `PATH_SKILLS_DIR` | `~/[记忆共享中心]/技能配置/` | Skill 主版本存放地 |

### 第二组：WorkBuddy 原生路径（仅 WorkBuddy 使用）

| 常量名 | 路径 | 说明 |
|--------|------|------|
| `PATH_IDENTITY` | `~/.workbuddy/IDENTITY.md` | AI 身份档案 |
| `PATH_USER` | `~/.workbuddy/USER.md` | 用户画像 |
| `PATH_RULES_MDC_CORE` | `~/.workbuddy/rules/core-identity.mdc` | L1+L2 规则的 .mdc 副本 |
| `PATH_RULES_MDC_BEHAVIOR` | `~/.workbuddy/rules/behavior-rules.mdc` | L3 行为规则的 .mdc 副本 |
| `PATH_MEMORY_DIR` | `{工作区}/.workbuddy/memory/` | 项目级工作记忆目录 |
| `PATH_MEMORY` | `{工作区}/.workbuddy/memory/MEMORY.md` | L4 情境层载体 |
| `PATH_DIARY` | `{工作区}/.workbuddy/memory/YYYY-MM-DD.md` | 每日日志 |
| `PATH_SKILLS_WB` | `~/.workbuddy/skills/{名称}/SKILL.md` | Skill 软链接 → 指向 PATH_SKILLS_DIR |
| `PATH_ROUTING_TABLE` | `~/[记忆共享中心]/记忆规则/记忆路由表.md` | 信号词路由表 |

## 第五部分：加载机制

### 公共文件夹 vs WorkBuddy 原生：数据流向

```
[记忆共享中心]/CORE.md ──写入──→ ~/.workbuddy/rules/core-identity.mdc ──注入──→ AI 上下文
[记忆共享中心]/用户规则.md ──同步──→ ~/.workbuddy/rules/*.mdc ──注入──→ AI 上下文
[记忆共享中心]/技能配置/SKILL.md ──软链──→ ~/.workbuddy/skills/*/SKILL.md ──按需加载
{工作区}/.workbuddy/memory/MEMORY.md ──────────────────────────────→ 注入项目上下文
```

### 加载机制表

| 文件类型 | 存放位置 | 加载时机 | 加载方式 | 多端可见？ |
|---------|---------|---------|---------|-----------|
| `IDENTITY.md` / `USER.md` | `~/.workbuddy/`（WB原生） | **每次对话自动** | system prompt 内置 | ❌ 仅WB |
| `rules/*.mdc` | `~/.workbuddy/rules/`（从公共同步） | **每次对话自动** | system prompt 内置 | ❌ 仅WB |
| `.workbuddy/memory/MEMORY.md` | 项目级 WB 原生 | **自动注入** | project context | ❌ 仅当前项目 |
| **公共文件夹文件** | `[记忆共享中心]/` | **AI 按需读取** | 工具调用 | ✅ **所有平台** |
| `skills/*/SKILL.md` | 软链接→公共文件夹 | 触发词匹配 | use_skill | ✅ **所有平台** |

## 第六部分：TTL 半衰期规则

| 记忆类型 | 半衰期 | 处理方式 |
|----------|--------|--------|
| 临时状态 | 1-3天 | 过期自动删除，不写入 MEMORY |
| 短期观察 | 7-14天 | 过期前确认"还成立吗？" |
| 中期经验 | 1-3个月 | 定期复盘，更新或归档 |
| 长期认知 | 6-12个月 | 年度大审查，写入 core-identity.mdc |
| 永久原则 | 永不过期 | 极少，仅 core-identity.mdc |

## 第七部分：系统维护总览（G5c）

> [!success] 📅 记忆系统日常维护节奏

| 频率 | 动作 | 触发方式 | 更新目标 |
|------|------|----------|----------|
| **每天** | 写日记 | "写日记"或"今天的日记" | PATH_DIARY → SHADOW.md |

| **每年** | 重做核心层问卷 | "更新我的核心层信息" | Q1-Q7 |
| **随时** | 发现规则需要调整 | 直接告诉 AI | 对应文件 |

## 第八部分：系统验证步骤

> 来源：五层记忆系统v4.1.6-执行文件-WorkBuddy版.md 第7节

### 验证步骤

1. **文件完整性检查**：
   - 检查 `~/.workbuddy/IDENTITY.md` 是否存在并有内容
   - 检查 `~/.workbuddy/USER.md` 是否存在并有内容
   - 检查 `~/.workbuddy/rules/` 下是否有 `.mdc` 文件
   - 检查 `.workbuddy/memory/MEMORY.md` 是否存在

2. **功能测试**：
   - **规则生效测试**：新建一个对话，说"用表格对比一下早睡和晚睡的利弊"，观察是否遵循定义的交互规则
   - **记忆加载测试**：说"我当前的核心目标是什么"，看能否从 MEMORY.md 回答
   - **Skill 测试**：如果有创建 Skill，说对应触发词测试是否正常加载

3. **系统就绪确认**：
   - 以上测试全部通过 → 系统就绪 ✅
   - 有问题 → 根据"先验证再下结论"原则排查
