# Skill Creator 优化工作流

本工作流用于创建、优化、修复和打包 Skill 包。执行时始终围绕北极星目标：产出可触发、可执行、可校验、可恢复、可交付的标准化技能包。

## 0. 启动原则

1. 先识别用户目标：新建技能、优化已有技能、修复结构、补齐元数据、打包交付。
2. 如果用户提供文件，先审计文件结构与关键内容，不直接改写。
3. 默认不覆盖原文件；所有交付物写入用户指定的 output 子目录。
4. 涉及删除、覆盖源文件、外部发布、凭证写入等风险动作时必须确认。

## 1. 输入诊断

按信息完整度分级：

- L0：无有效输入。只给模式选项和最少问题。
- L1：目标模糊。提出不超过 3 个关键问题，同时给出可默认方案。
- L2：目标和主要流程明确。直接生成草案，标注假设。
- L3：材料完整。生成完整技能包并执行 QA。
- L4：已有技能包。先审计，再优化，不盲目重写。

必须确认的信息：

- 技能边界和不应处理的任务；
- 是否允许覆盖原文件；
- 是否涉及真实外部系统写入；
- 是否需要安装到用户技能目录；
- 安全、隐私、权限相关内容。

可自动补全的信息：

- 目录名：英文小写短横线；
- 示例文件名和 references 文件名；
- 基础触发词；
- README 的安装说明；
- 常见 QA 清单。

## 2. 结构审计

优化已有技能包时，检查：

1. 是否包含 `SKILL.md`；
2. `SKILL.md` 是否有 YAML front matter；
3. front matter 是否至少包含 `name` 与 `description`；
4. `name` 是否为小写短横线格式；
5. `description` 是否说明用途、触发场景和边界；
6. 是否存在 `AGENTS.md`、`README.md`、`references/`、`examples/` 等辅助内容；
7. 是否有无效占位符、时间戳命名、绝对私有路径、敏感信息；
8. 是否说明输出目录、打包方式、安装验证方式。

## 3. 生成与优化策略

### 新建技能

推荐结构：

```text
skill-name/
  SKILL.md
  AGENTS.md
  README.md
  references/
    workflow.md
    qa-checklist.md
  examples/
    usage.md
  package.json
```

### 优化技能

优先级从高到低：

1. 修复不可安装或不可触发的问题；
2. 补齐 YAML front matter；
3. 明确触发场景、边界和禁止事项；
4. 增强输入诊断与自动补全；
5. 增强异常处理、重试、断点恢复；
6. 增加 QA 清单和示例；
7. 改善 README 与交付说明。

### 避免过度重构

- 保留原技能目录名，除非存在明显命名错误；
- 保留原文件结构，新增文件优先放入 `references/`；
- 不为了形式统一而删除有效内容；
- 不引入无法执行的复杂流程。

## 4. 异常处理

遇到异常时按级别处理：

- P0 阻塞：缺少核心文件、压缩包损坏、无法读取。停止并说明阻塞。
- P1 高风险：会覆盖原文件、涉及凭证、外部写入。必须确认。
- P2 可自修复：缺少元数据、结构不完整、说明不清。直接修复并记录。
- P3 可提示：命名不优雅、示例不足、描述冗长。优化并说明。

错误汇报格式：

```text
当前卡点：
已检查/尝试：
失败原因：
影响范围：
建议方案：
需要用户决策：
```

## 5. 稳定性与恢复

复杂任务必须维护进度记录，建议写入 `references/progress-template.md` 或交付说明：

- 当前阶段；
- 已完成文件；
- 未完成事项；
- 关键假设；
- 失败重试记录；
- 降级交付方案。

同一问题最多连续自修复 2 次；仍失败时停止硬跑，改为汇报可选路径。

## 6. 输出前 QA

交付前必须检查：

- `SKILL.md` 存在且 front matter 有效；
- `name` 与目录名一致或差异有解释；
- `description` 能让 Agent 判断何时触发；
- 工作流包含输入、处理、异常、输出、校验；
- 示例覆盖新建、优化、异常三类场景；
- 无敏感信息、无私有绝对路径、无无效占位符；
- 输出在指定目录；
- zip 包根目录结构正确，不多包一层无意义目录。

## 7. 交付说明

最终回复包含：

1. 是否完全完成；
2. 输出文件链接；
3. 本次优化点；
4. 已做校验；
5. 未覆盖风险；
6. 后续可选优化建议。

## 工程级安装前检查

在最终打包前，优先运行：

```bash
bash scripts/validate-skill.sh . reports
```

打包后，再运行：

```bash
bash scripts/validate-skill.sh . reports ../skill-creator-optimized.zip
```

如果报告为 `PASS`，可以交付。
如果报告为 `WARN`，需要说明风险和影响范围。
如果报告为 `FAIL`，应先修复失败项；除非用户明确接受降级交付，否则不应交付为完成状态。

### 脚本化 QA 的收益

- 降低 LLM 输出波动造成的结构遗漏。
- 自动发现 front matter、空文件、占位文本等常见问题。
- 为 SkillHub 或团队内部准入评估提供可审计证据。
- 支持复杂任务中断后的快速恢复和复验。
