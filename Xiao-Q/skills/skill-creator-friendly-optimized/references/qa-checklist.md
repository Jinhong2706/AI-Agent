# Skill 包质量门禁清单

用于创建、优化、修复和打包 Skill 前的最终检查。

## 1. 结构检查

- [ ] 技能目录名使用小写英文、数字和连字符。
- [ ] 根目录包含 `SKILL.md`。
- [ ] 推荐包含 `AGENTS.md`、`README.md`、`examples/`、`references/`。
- [ ] 没有把临时文件、系统文件、密钥文件打包。

## 2. 元数据检查

- [ ] `SKILL.md` 顶部存在 YAML front matter。
- [ ] front matter 至少包含 `name` 与 `description`。
- [ ] `name` 与目录名一致或有明确理由。
- [ ] `description` 写明触发场景、能力范围和典型输入。
- [ ] YAML 可解析，没有中文冒号、缩进或引号错误。

## 3. 触发与边界检查

- [ ] 写明“何时使用本技能”。
- [ ] 写明“不应使用本技能”的边界。
- [ ] 对模糊需求有诊断方式，不会过度追问。
- [ ] 对高风险动作要求用户确认。

## 4. 执行质量检查

- [ ] 有清晰工作流。
- [ ] 有输入校验规则。
- [ ] 有自动补全默认值规则。
- [ ] 有异常处理分级。
- [ ] 有自纠错策略。
- [ ] 有断点恢复或进度保存机制。

## 5. 输出检查

- [ ] 输出文件落在用户指定目录。
- [ ] 默认不覆盖原文件。
- [ ] 若多文件交付，打包为 zip。
- [ ] 交付说明包含完成度、改动摘要、剩余风险和后续建议。

## 6. 失败处理检查

- [ ] 失败时说明卡在哪一步。
- [ ] 说明已尝试动作。
- [ ] 说明失败原因。
- [ ] 给出继续、换路径、降级或停止的选项。

## 自动化 QA 检查

交付前应优先执行：

```bash
bash scripts/validate-skill.sh . reports
```

验收标准：

- `reports/qa-report.json` 中 `status` 应为 `PASS`。
- `summary.fail` 应为 `0`。
- 如有 `WARN`，必须在交付说明中解释。
- 若生成 zip，必须再次带 zip 路径运行脚本复验。

推荐将 `reports/qa-report.md` 作为人工复盘附件，将 `reports/qa-report.json` 作为自动化验收附件。
