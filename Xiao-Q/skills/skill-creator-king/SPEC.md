## 核心需求

### REQ-001: 三大入口路由
**描述**：用户从创建/升级/审查三个入口进入，AI 按入口引导对应流程。
**优先级**：P0
**验证标准**：
- "帮我建一个 skill" → 进入五阶段创建流程
- "升级到完整版" → 进入升级流程
- "帮我看看这个 skill" → 进入审查打磨流程
- 消歧规则生效："帮我改改"+"完整版"→升级，"帮我改改"+skill名→审查
- 不明确时反问用户意图

### REQ-002: 五阶段创建流程
**描述**：引导用户从定位到交付，完成一个符合规范的 skill。
**优先级**：P0
**验证标准**：
- Phase 1-5 按顺序引导，不可跳
- 三通道选择（轻量/完整/数据驱动）在 Phase 1 前完成
- 6 个强制阻塞点全部等用户确认
- Phase 5 自动生成 README + CHANGELOG + SESSION-REPORT
- 每个 Phase 结束小结产出，确认后进入下一 Phase

### REQ-003: 审查打磨
**描述**：给已有 skill 做体检+诊断+治疗，输出三位一体报告。
**优先级**：P0
**验证标准**：
- validate.py + estimate-tokens.py 自动扫描
- 抽检 L2 文件按优先级规则（核心工作流 > Token大户 > 随机）
- 三位一体报告含🌟亮点/⚠️需改进/💡学习要点
- 纯诊断出口：只报告不修复
- 修复救援模式：损坏 skill 手写建议
- 进度管理：每修一个展示进度
- 版本 bump：validate.py 扫描列出版本文件清单

### REQ-004: 升级机制
**描述**：轻量 skill 补全为完整版，不重写已有产出。
**优先级**：P1
**验证标准**：
- 对照 full.yaml 识别 gap
- 逐 gap 引导补充
- 不重写用户已有的 SKILL.md/L2 文件
- 升级后过 Phase 5 完整版标准

### REQ-005: 自进化
**描述**：审查其他 skill 时，发现好模式经三道门过滤后纳入自身。
**优先级**：P1
**验证标准**：
- 三道门：适合吗？冲突吗？用户拍板？
- 审查结束时批量呈现，不逐个打断
- 审查 skill-creator-king 自身时跳过（路径比对）
- CHANGELOG 区分 Design Iteration / Learned

### REQ-006: 三个强制交付物
**描述**：每次创建或审查结束自动生成 README.md + CHANGELOG.md + 交付报告。
**优先级**：P0
**验证标准**：
- README.md：名称+用途+触发词+版本
- CHANGELOG.md：标准格式版本条目
- 交付报告：基本信息+完成清单+自检+文件清单+下次建议

### REQ-007: 质量评分（v3.0 缓存）
**描述**：10维度自动化质量审计，内置缓存。3个手动维度提供脚本预检标记辅助AI裁定。，通道自适应评分。评分嵌入 Phase 5 交付流程和审查入口体检步骤，不作为独立文件。
**优先级**：P1
**验证标准**：
- quality-audit.py 运行无错误
- 轻量通道评6维(满分85)，完整通道评10维(满分100)
- 自动维度脚本判，手动维度标记待 AI 裁定
- 评分嵌入 SESSION-REPORT 和 REVIEW-REPORT

### REQ-008: 自反思循环（Ralph Loop，v3.0）
**描述**：Phase 3 产出 SPEC/DESIGN 后和 Phase 5 交付前各执行一次双层自反思（客观+主观）。最多3轮，全过或3轮后 SPEC/DESIGN 冻结。
**优先级**：P1
**验证标准**：
- 客观清单 6 项全部通过
- 主观 4 问全部有回答
- 轻量通道+客观全过 → 可跳过主观
- 3轮后强制冻结（不无限循环）
- 冻结后修改需要新 ADR

### REQ-009: 双层自检
**描述**：SCK 提供 self-audit.py 脚本覆盖 5 项通用检查 + 1 项 SCK 专属检查。
通用层（引用完整性/配置活性/测试覆盖/保鲜度/纪律验证）在新 skill 创建/审查时自动运行，
SCK 专属层（名称一致性）在审查 SCK 自身时使用。嵌入 validate.py --strict。
**优先级**：P1
**验证标准**：
- `validate.py --strict <sck-dir>` 自动嵌入 self-audit 检查
- `self-audit.py list-checks` 返回 6 项检查描述
- self-audit 在 SCK 自身通过 0 issue（含 --sck-mode）
- 三条操作纪律变为可检测的代码检查
