# Phase 1 详细指引：定位

## 入场条件

```bash
python3 scripts/phase-check.py <skill-dir> --phase 1
```
✅ SKILL.md 存在 + frontmatter 模板字段有效 → 进入 Phase 1

## 目标
让用户把"我想建一个什么"说清楚。不急着写代码，先想透。

## 对话步骤

### Step 1：问问题
**Q: 这个 skill 解决什么人的什么问题？**
- 追问："谁用？在什么场景用？解决什么痛点？"
- 如果用户回答太宽（"帮人写东西"），追问："你能想象一个具体的人吗？他在什么情况下会打开这个 skill？"

### Step 2：问信念
**Q: 最核心（不可妥协）的价值是什么？**
- 给例子："little-writer 的核心信念是'真实的比华丽的重要'"
- 如果用户想不出来，反问："如果只能做对一件事，是什么？"

### Step 3：让用户说触发词
**Q: 你希望用户怎么跟这个 skill 说话？给我 1-3 个例子**
- ⚠️ **强制阻塞**：必须等用户自己说出来，AI 不能代填
- 追问："如果用户带着情绪说'烦死了又要写XX'，你觉得该路由到什么？"
- 追问："你的用户会不会用口语化表达？像'搞不定了'这种？"
- 如果用户说的触发词太正式，提醒："真实用户可能不会这么说，更像'烦死了明天要交周报'"

### Step 4：起名
**Q: 给 skill 起个什么名字？**
- AI 从三个角度给候选（不给决定）：
  - 面向功能："周报助手"
  - 面向场景："周五救星"
  - 面向价值："不再凑字数"
- **强制阻塞**：名字必须用户确认

### Step 5：探结构（AI 内部提问指南 — 不向用户暴露"类型"概念）

SCK 在设计阶段需要理解 Skill 的结构特征，以便在 Phase 2 给出匹配的设计检查要点。以下六问题映射六种结构模式，AI 根据用户已说出的信息自然挑选 2-3 个来问。**不需要问完、不需要按顺序、不需要解释为什么问。**

| # | 自然提问 | 在探什么 | 对应关注点 |
|---|---------|---------|-----------|
| 1 | 「这个 skill 的核心是一次把活干完出结果，还是分几个步骤、中间可能需要你确认或调整？」 | 流程节奏 | Pipeline vs 单步 |
| 2 | 「它主要是调外部的什么东西（API/文件/数据库），还是主要靠自己的推理和判断？」 | 执行性质 | Tool vs Advisor |
| 3 | 「它产出的是一个标准化的东西（模板/报告/代码），还是要根据每次的情况灵活应对？」 | 产出模式 | Generator vs Advisor |
| 4 | 「它做之前需要先跟你聊一圈、搞清楚情况再动手，还是你给了指令直接就能跑？」 | 交互模式 | Interviewer vs 直行 |
| 5 | 「它是检查/评审/打分一个已经有的东西，还是从零生成新东西？」 | 输入物 | Reviewer vs Generator |
| 6 | 「它最复杂的那一步，里面会嵌套判断或分支吗？还是每一步都比较直？」 | 内部分支 | Pipeline 内部复杂度 |

**使用原则**：
- 如果用户已说清，跳过对应问题。比如用户说「先抓数据，然后分析，最后出报告」，问题 1 已经有答案了。
- 组合信号比单信号可靠。用户可能同时触发多个模式——Pipeline + Reviewer、Interviewer + Advisor，这很正常。
- 不要试图归约到一个"唯一类型"。记在心里，在 Phase 2 时根据组合信号给出对应的设计检查要点。

**结构信号 → Phase 2 设计关注点速查**（AI 内部用，不做输出）：

| 信号 | Phase 2 必须检查 |
|------|-----------------|
| Pipeline 信号 | 步骤划分是否清晰？每步间 checkpoint 是什么？失败回滚到哪？ |
| Tool 信号 | 外部依赖声明了吗？错误处理路径全覆盖了吗？超时/重试策略？ |
| Generator 信号 | 输出模板在哪？参数校验清单？用户可定制哪些部分？ |
| Reviewer 信号 | 评分 rubric 定义了吗？报告格式固定还是可调？自动打分还是 AI 裁定？ |
| Interviewer 信号 | 问题流画了吗？什么条件下跳分支？默认路径是什么（用户不回答时）？ |
| Advisor 信号 | 参考资料来源声明了吗？什么情况该说「我不知道」？领域边界在哪？ |

## Phase 1 收尾
小结用户的选择，用 3 句话：
1. "你的 skill 叫 {name}，面向 {who}"
2. "核心信念是 {beliefs}"
3. "用户会说 {triggers}"

问："这准确吗？要不要调？" 确认后进入 Phase 2。

## 轻量 vs 完整差异
- 轻量：仅核心文件（SKILL.md + README + CHANGELOG）
- 完整：额外生成 DESIGN.md（需求规格 + 架构决策）
