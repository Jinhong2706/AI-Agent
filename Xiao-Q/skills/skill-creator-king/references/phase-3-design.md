# Phase 3 详细指引：设计

## 目标
把"做什么"变成"怎么做"，为每个关键决策记录原因。

## 对话步骤

### Step 3.1：架构决策
**引导**："遇到几个关键选择，我们一起想："
- "用 basic 还是 multi-scene 模板？你的用户需要多个入口吗？"
- "L2 文件怎么拆？每个工作流一个，还是每个功能一个？"
- ADR 格式：背景 → 决策 → 理由 → 后果 → ❗不可回退（如果是的话）
- AI 给建议但不替代决策："basic 适合单一入口，multi-scene 适合多场景——你的情况呢？"

### Step 3.2：Token 实测
**⚠️ 踩坑提醒**："Token 不能猜，必须实测。我们上次 ADR-008 估算偏差 2-3 倍。"
- 用 estimate-tokens.py 实测每个 L2 文件
- 汇总后跟用户一起定预算："实测 L2 大约 4800 token，你接受吗？要不要拆懒加载？"
- 如果超标给出方案：
  1. 上调 hard_cap
  2. 懒加载拆分
  3. AI 动态生成替代预存

### Step 3.3：模块关系 + 数据流
**引导**："画一下你的 skill 长什么样："
- 模块：SKILL.md → L2 files → scripts → config
- 数据流：用户输入 → 路由 → 工作流 → 产出
- ⚠️ 检查：脚本失败了怎么办？有降级路径吗？

## Phase 3 收尾：自反思循环（Ralph Loop）

**产出 SPEC/DESIGN 后必须先自反思再提交用户。**

1. AI 逐项过 data/checklists/self-check.yaml objective 清单（6项）
2. AI 按 subjective 清单自问 4 个问题
3. 发现问题 → 修正文档 → 重检（最多3轮）
4. 全过或3轮后 → 提交用户确认 → SPEC/DESIGN 冻结

> ⚡ 轻量通道+客观全过 → 可跳过主观自检

## Phase 3 收尾
确认 ADR + Token 预算 + 架构图。

## 轻量 vs 完整差异
- 轻量：简化 ADR + Token 预算
- 完整：完整 DESIGN.md + 实测数据 + 架构图
