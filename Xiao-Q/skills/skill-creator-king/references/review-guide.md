# 审查打磨（Review & Refine）

> 🩺 给已有 skill 做体检 + 治疗 + 教学

## 三步流程

### Step 1：体检

**1.1 自动扫描**：
- `validate.py --strict` → 嵌入 self-audit（引用完整性/配置活性/测试覆盖/保鲜度/纪律验证）。跑在 quality-audit 之前——链接都断了，打分没意义。
- `quality-audit.py` → 10维度评分（内含 validate + estimate-tokens，不需单独跑）。
- 审查入口默认询问是否跑评分。评分缓存同版本秒出，`--no-cache` 重跑。缓存命中+AI已裁定态 → 跳过手动审查，直接展示结果。

**1.2 手动审查清单**：
- 读 SKILL.md → 看 `template` 判断类型（multi-scene：路由+消歧；basic：原则+入口）
- 对照 anti-patterns.yaml 中 `auto_detectable: false` 条目：#9-12
- 抽检 L2 文件（references/ + data/，不含 scripts/）：≤2 全量，>2 抽 2-3，优先级：核心工作流 > Token 最大 > 随机。标注"抽样审查，非全量"
- 检查配置-逻辑分层：每个 workflow 文件有归属注释？
- 检查 README 介绍板块：有没有「这是什么/适合谁/不适合谁」？无 → 从 SKILL.md 自动分析生成草稿，确认后写入
- 检查入口同步清单：炼化/新建/升级类操作有步骤 4.5 式同步清单？
- 检查引用可达性：`references/` 和 `templates/` 下的文件是否在 SKILL.md 工作流中被显式路径引用？仅在 README 文件树中列出不算——LLM 加载 skill 时不一定会去文件树中寻找这些文件
- 六条操作纪律（详见 `rules/`）：不穷举、不双写、不窄检、改必扫、不孤改、表先于盘。审查 SCK 自身时 `--sck-mode` 自动验证

**1.3 0 issue**：基础质量无可挑剔，继续手动审查设计和 L2 内容。

**1.4 修复救援**：validate 返回 success:false → 只看能读的部分，手写修复建议，不跑完整流程。

**1.5 审计策略参考**（AI 自动执行，不逐条确认）：

#### 8 类错误清单（逐一对照）

| ID | 类别 | 检测方式 | self-audit |
|:--:|------|---------|:----------:|
| D1 | 死引用 | 被引用的文件/路径不存在 | check_1 |
| D2 | 内容漂移 | 两处应同义的内容不一致 | check_5 |
| D3 | 窄检 | 验证扫描范围 < 被检查范围 | check_3 |
| D4 | 硬编码腐败 | 数值/字符串过时（Token, 版本号） | check_4 |
| D5 | 义务空缺 | 规则声明了但无验证机制 | check_2+5 + AI审查 |
| D6 | 未测试代码路径 | 特殊模式存在但从未执行 | check_5 |
| D7 | 生成物累积 | 缓存/临时文件无清理 | check_5 |

#### 7 条审计策略（AI 自动执行）

| # | 策略 | 触发时机 | 操作 |
|:--:|------|:-------:|------|
| 1 | 广度→深度 | 第一轮输出后 | 第二轮读代码细节，追踪 ≥3 条执行路径 |
| 2 | 加法→减法 | 方案给出后 | 问：能不能一个机制解决？ |
| 3 | 正向→反证 | 实现完成后 | 构造应该失败的输入，验证正确失败 |
| 4 | 作者→审计者 | 实现完成后 | 强制切换心态，重读所有改动 |
| 5 | 当前→历史 | 审查时 | 读 CHANGELOG，历史错误类型纳入当前检查 |
| 6 | 局部→全局 | 方案确认后 | 当前方案能否让下游产物也受益？ |
| 7 | 单源→双源 | 数据不一致时 | 测试断言 vs 设计文档 vs 实测值，三角验证 |

#### 3 种 AI 认知偏误（审查时自问）

| 偏误 | 自检问题 | 典型案例 |
|------|---------|---------|
| 结构vs执行盲区 | 我在分析结构还是追踪执行？读了代码吗？ | 看到 strict 模式，没读到 all_issues 定义前就被用了 |
| 加法偏好 | 方案是 PATCH（加一行）还是 REFACTOR（删散落→合一）？ | 12项散落方案 vs self-audit.py 一个脚本 |
| 实现惯性 | 刚实现完就接受了？要不要切审计者视角？ | self-audit.py 写完后的两个 bug |

### 1.6 过程级反模式检查清单（PRESCAN/VERIFY 阶段对照扫描）

| # | 反模式 | 表现 | 检测方式 | SCK 对应 |
|:--:|--------|------|---------|:--------:|
| 1 | 散文式纪律 | 操作规则以散文写在 SKILL.md 中 | rules/ 空但有「必须/禁止」描述 | 纪律 Rule 化 |
| 2 | 硬编码阈值 | 脚本含魔法数字 | 静态扫描裸数字（排除 0/1/-1） | D4 |
| 3 | 版本号不同步 | 核心文件版本不一致 | self-audit/validate 交叉比对 | D5 |
| 4 | 占位符残留 | TODO/FIXME/待补充 | grep 扫描 | validate --strict |
| 5 | named-but-missing | SPEC/DESIGN 声明了但文件不存在 | 声明-实际对比 | D1 |
| 6 | 单向依赖 | A 引用 B 的资产，B 不知道 | 人工审查 | 暂无 |
| 7 | 测试用硬编码 | 手写预期值而非实测值 | 审查 test assert 值 | D4 |
| 8 | README 滞后 | README 信息落后于实际文件 | PRESCAN 文件清单对比 | validate |
| 9 | 过期引用 | references 引用不存在的文件 | grep + 存在性检查 | D1 |
| 10 | 文档-代码分叉 | SKILL.md 流程 ≠ scripts/ 行为 | 人工审查+端到端测试 | D2 |
| 11 | 无收敛循环 | 循环无最大迭代/超时 | 静态扫描 scripts/ | self-audit |
| 12 | 隐式契约 | SKILL.md 和 DESIGN.md（或 SPEC.md）边界不一致 | 契约-实现交叉比对 | D2 |
| 13 | 级联断裂 | 新增资产类型未联动更新 | 触发 rules/operational-rules.md §5 检查 | 新增 Rule |
| 14 | Phase-0 跳过 | 未确认平台/入口/通道直接进入 | phase-check.py 入场条件 | ADR-013 |

使用方式：手动审查逐一过表，命中 → 引用反模式编号（如"反模式 #13：级联断裂"）。

### Step 2：诊断

**2.0 好模式识别**（每次审查必跑）：反模式检测找"什么错了"——好模式识别找"什么做得好"。不依赖程序化检测，由 LLM 在审查语境下判断。

| 类别 | 检测标准 | 示例 |
|------|---------|------|
| **架构优雅** | 结构简单但覆盖全场景，没有过度设计 | 只用 3 个 reference 文件完成了 5 种场景，每个文件职责清晰 |
| **预算精准** | L0/L1/L2 比例合理，hard_cap 不过大 | L0:200 / L1:3000 / L2:8000 — 轻量快启，深度够用 |
| **触发设计** | trigger 覆盖主要入口但不过泛，有明确边界 | 7 个 trigger 精确映射 3 个场景，没有模糊匹配 |
| **反模式零命中** | 14 维评分中"反模式扫描"得满分，13 个反模式一个没踩 | 无 AP 命中 |
| **创新模式** | 做了 SCK 当前框架未预见到的新设计，且逻辑自洽 | 用 YAML 画像库替代传统 reference 文件，结构化程度更高 |
| **信息密度** | Token 预算里塞了超出预期的可用内容 | L1_core 4000t 覆盖 6 阶段完整工作流 + 决策框架 |

识别结果写入审查报告的「🌟 好模式」章节，同时追加到 `review-history.jsonl` 的 `patterns` 字段。格式：`{"category": "架构优雅", "detail": "具体描述", "evidence": "佐证（文件/行号/评分维度）"}`。

**2.1 呈现**：按严重度分组（Critical/High/Medium），询问用户优先顺序。

**2.1.2 扣分分析**（分数 < 满分时必跑）：逐维度输出——少了什么导致扣分、建议修复吗、修复成本（低/中/高）。用户决策是否修复。

**2.1.5 功能膨胀检查**（每次诊断必跑）：追问「哪些不修也能完成 80% 核心任务？」→ 标记低优先级，放入未处理建议。

**2.2 深挖**（每个选定问题）：
- 引用 anti-patterns.yaml 的反模式
- 踩坑故事解释根因
- 影响分析：改动涉及 frontmatter/路由/L2 路径时检查所有引用处

**2.3 纯诊断出口**：用户只看不改 → 输出体检报告+改进建议 → REVIEW-REPORT.md → 仍进入 Step 4 → 结束。

### Step 3：治疗 + 复查

**3.1 逐项改**：改一个 → re-validate → 绿灯 → 下一个。改动涉及 references/data → 重跑 estimate-tokens.py。

**3.2 P0 门槛**：P0 未通过 = 不能收工。P1/Medium 可放入未处理建议。

**3.2.5 Drift Guard**：有 tests/ → 跑一遍确认无回归。无则跳过。

**3.3 版本 bump**：
- 更新 SKILL.md frontmatter version（唯一权威来源）
- 同步 CHANGELOG/SPEC/DESIGN 版本历史表
- 运行 validate.py 确认无 version_scattered
- README/SKILL.md body 显示引用顺手更新

**3.4 报告存储**：默认 `{skill}/REVIEW-REPORT-{DATE}.md`。

### Step 4（可选）：自进化

发现 scK 能学的好模式 → 批量呈现。审查 SCK 自身时跳过。采纳后 CHANGELOG 标注来源，重跑 quality-audit 确认不降分。

### Step 5：记录审查历史

追加 `data/review-history.jsonl`：
```json
{"skill": "human-stars", "version": "2.1.0", "score": 108, "date": "2026-05-18", "reviewer": "SCK", "key_findings": ["README缺介绍板块"], "patterns": [{"category": "架构优雅", "detail": "只用3个reference文件完成5种场景"}]}
```
字段：skill/version/score/date/reviewer/key_findings（≤5项）/patterns（好模式数组，可为空）

## 三位一体审查报告

**🌟 亮点**：是什么 + 为什么好 + 能学到什么（三要素）
**⚠️ 需改进**：反模式引用 + 根因 + 修复建议 + 踩坑故事
**💡 学习要点**：本次审查可带走的 2-3 条设计教训

语气：用"一起踩过的坑"口吻，不说教。

---

## 话术规则

- 有亮点有不足 → 先说亮点再提问题
- 只有问题无亮点 → 先肯定某处做得干净，再谈问题
- 全绿无问题 → "基础质量无可挑剔，标注几个值得继续坚持的模式"
