# Phase 0 详细指引：出发前确认

## 入场条件

Phase 0 无前置条件——它是所有流程的起点。

```bash
python3 scripts/phase-check.py <skill-dir> --phase 0
# → ✅ Phase 0 无前置条件（出发前确认）
```

## 目标

在跳进具体工作前，对齐三件事：在什么平台上、做什么事、走什么通道。这一步跳过直接开工，后面返工代价巨大。

## 对话步骤

### Step 0.1：确认目标平台

**Q: 这个 skill 在哪个平台上跑？**

| 平台 | 路径 | 特征 |
|------|------|------|
| WorkBuddy | `~/.workbuddy/skills/` | 支持 token_budget、semantic_consistency、README同步、CHANGELOG |
| OpenClaw | `~/.openclaw/skills/` | 无 token_budget（满分 81） |
| Hermes | `~/.hermes/skills/` | 无 token_budget（满分 81） |
| Universal | `~/.agents/skills/` | 无 token_budget，通用可移植 |

- 用户可能不知道平台名称。问："你的 skill 放哪个目录？" 用路径反推平台。
- `scripts/init_skill.py` 和 `scripts/platform.py` 会自动检测，但先确认避免路径猜错。

### Step 0.2：确认入口

**Q: 你想做什么？**

| 入口 | 你可能会说 |
|------|-----------|
| 🆕 创建 | "帮我建一个 skill"、"新建"、"做一个工具" |
| 🔄 升级 | "升级到完整版"、"帮我补全" |
| 🩺 审查 | "帮我看看这个 skill"、"体检"、"打分" |
| 📊 评分 | "打一下分" |

- 如果用户说的话模糊（"我想搞个东西"），追问："是从零建一个，还是改现有的？"
- 四个入口共享五阶段流程，但起点不同：
  - 创建：Phase 0 → 1 → 2 → 3 → 4 → 5
  - 升级：Phase 0 → 直接进 Phase 2/3（补缺）
  - 审查：Phase 0 → 直接进审查三步流程（体检→诊断→治疗）
  - 评分：Phase 0 → 直接跑 quality-audit.py

### Step 0.3：确认通道

**Q: 这次走轻量还是完整？**

| 通道 | 时间 | 产出 | 适合 |
|------|:---:|------|------|
| 🚀 轻量 | ~30min | SKILL.md + README.md + CHANGELOG.md | 快速验证想法、简单工具 |
| 🏗️ 完整 | ~2h | 以上 + DESIGN.md（合并 WHAT+HOW）+ scripts/ + tests/ | 复杂 skill、长期维护 |

- 不替用户选通道，但给建议：
  - "不确定的话先轻量——轻量随时能升级到完整"
  - "如果这个 skill 只有你自己用、逻辑简单，轻量够了"
  - "如果要给别人用、或者逻辑会变复杂，完整更保险"
- 升级入口下通道自动设为完整（轻量升级本身就是补全）。

## Phase 0 收尾

三句话小结：
1. "在 {平台} 上，{创建/升级/审查/评分}"
2. "走 {轻量/完整} 通道"
3. "确认后进入 Phase 1。"

问："这三项对吗？" 确认后进入下一步。

## 必须用户决策清单

| # | 决策点 | Phase |
|---|--------|:---:|
| 1 | 目标平台 | Phase 0 |
| 2 | 入口（创建/升级/审查/评分） | Phase 0 |
| 3 | 通道（轻量/完整） | Phase 0 |

---

> Phase 0 跳过是最大的时间陷阱。三问花 2 分钟，省后面 2 小时返工。
