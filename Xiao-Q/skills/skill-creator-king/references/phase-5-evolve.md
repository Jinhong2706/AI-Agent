# Phase 5 · 持续迭代

## 入场条件

```bash
python3 scripts/phase-check.py <skill-dir> --phase 5
```
✅ Phase 4 VERIFY 通过 → 进入 Phase 5

## 目标

不要让 skill 变成僵尸文件。主动反思、持续改进、适时退役。

## 对话步骤

### Step 1：自我反思

引导 AI 回顾本次构建过程：

- 「哪些决策是在信息不充分时做的？现在回头看，还成立吗？」
- 「构建过程中有没有出现超过 3 轮反复的地方？那是设计缺陷还是沟通不清？」
- 「如果有机会从头做，哪个架构决策会改？」

### Step 2：改进建议

基于反思，生成按优先级的改进清单：

- **P0（必须修）**：验证阶段残留的未通过项
- **P1（应该修）**：anti-pattern 报告中的建议项
- **P2（可以修）**：优化项——更好的 L2 拆分、Token 压缩、懒加载

### Step 3：活性审计（v3.0+）

当满足触发条件时（3+ 版本 / SKILL.md > 200 行 / 脚本 > 5 个），执行四问法：

| # | 问题 | 判断方式 |
|---|------|---------|
| 1 | 最近一次实际工作是什么时候？ | 运行日志、触发记录。从未触发 → 考虑退役 |
| 2 | 触发条件现在还成立吗？ | 依赖的文件/状态/条件是否仍存在 |
| 3 | 有别处也在做同样的事吗？ | 搜索重复逻辑。重叠 → 合并 |
| 4 | 是因为需要加的，还是因为加了所以留着？ | 回溯添加原因。原因已消失 → 移除 |

详见 `vitality-audit.md`。

### Step 4：版本记录

更新 CHANGELOG.md，记录本次迭代的：
- 新增功能
- 修改项
- 废弃项
- 已知问题

## Phase 5 收尾

Skill 进入「维护模式」——不再主动修改，等待下一次触发迭代。

> 一个好的 skill 是活着的——它在使用中生长，在反思中进化，在不再被需要时体面退役。
