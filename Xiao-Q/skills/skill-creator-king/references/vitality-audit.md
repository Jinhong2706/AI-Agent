# 活性审计（Vitality Audit）

> 迭代 3+ 次的 skill 建议执行。帮你识别：**哪些构件已经死了但还留在技能里。**

触发条件（满足任一）：
- 3+ 个版本（从 CHANGELOG 检测）
- SKILL.md > 200 行
- 脚本 > 5 个

## 四问法

对每个构件（启动步骤、后任务、工作流分支、文档节、脚本、模板）：

| # | 问题 | 判断方式 |
|---|------|---------|
| 1 | 最近一次实际工作是什么时候？ | 运行日志、task history、触发记录。从未触发 → 🗑️ |
| 2 | 触发条件现在还成立吗？ | 依赖的文件/状态/条件是否仍存在 |
| 3 | 有别处也在做同样的事吗？ | 搜索重复逻辑。另一构件覆盖同一场景 → 🔀 合并 |
| 4 | 是因为需要加的还是因为加了所以留着？ | 当初为什么加？原因还成立吗？答不上来 → 🗑️ |

## 输出格式

```
| 构件 | 功能 | Q1 | Q2 | Q3 | Q4 | 判决 |
|------|------|:--:|:--:|:--:|:--:|:----:|
| step 8 | bootstrap 检查 | 永假 | BOOTSTRAP 已删 | - | - | 🗑️ DELETE |
| step 1+2 | 配置读取 | ✅ | ✅ | 重复读同一文件 | ✅ | 🔀 MERGE |
```

## 与评分的关系

活性审计不是评分维度——不给你打分，帮你找可以删的东西。在 10 维度评分完成后执行。
