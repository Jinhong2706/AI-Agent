# DESIGN 阶段 · 子步骤 2/2：架构设计

> 所属阶段：Phase 2 DESIGN（合并 WHAT+HOW，v3.14+）
> 前置：子步骤 1/2 需求定义 已完成（DESIGN.md REQ 冻结）

## 入场条件

```bash
python3 scripts/phase-check.py <skill-dir> --phase 2
```
✅ DESIGN.md REQ 冻结（非占位）→ 进入架构设计

## ⚠️ 双文件兼容（旧架构）

如果目标 skill 同时存在 `SPEC.md` 和 `DESIGN.md`（v3.14 之前的旧架构），子步骤 1/2 应已建议合并。若用户选择保留双文件，更新 DESIGN.md 时同步更新 SPEC.md 的 NFR/版本号部分，保持两者一致。

## 目标
把"做什么"变成"怎么做"，为每个关键决策记录原因。

**DESIGN.md 分层原则（v3.17.2+）**：skill 目录下的 DESIGN.md 保持轻量——架构总览 + 一条代表性决策 + 候选方案对比。逐条设计决策按日期归档到项目的 `2-design/` 目录（project-harness 管理），不在 skill DESIGN.md 里堆。这保证 SKILL 部署包小巧（audit tool 扫得到关键结构），同时设计历史可追溯（项目目录里每条决策都有背景/决定/理由/影响）。

## 对话步骤

### Step 1：架构决策
**引导**："遇到几个关键选择，我们一起想："
- "用 basic 还是 multi-scene 模板？你的用户需要多个入口吗？"
- "L2 文件怎么拆？每个工作流一个，还是每个功能一个？"
- ADR 格式：背景 → 决策 → 理由 → 后果 → ❗不可回退（如果是的话）
- AI 给建议但不替代决策："basic 适合单一入口，multi-scene 适合多场景——你的情况呢？"

### Step 2：Token 实测
**⚠️ 踩坑提醒**："Token 不能猜，必须实测。我们上次 ADR-008 估算偏差 2-3 倍。"
- 用 estimate-tokens.py 实测每个 L2 文件
- 汇总后跟用户一起定预算："实测 L2 大约 4800 token，你接受吗？要不要拆懒加载？"
- 如果超标给出方案：
  1. 上调 hard_cap
  2. 懒加载拆分
  3. AI 动态生成替代预存

### Step 3：模块关系 + 数据流
**引导**："画一下你的 skill 长什么样："
- 模块：SKILL.md → L2 files → scripts → config
- 数据流：用户输入 → 路由 → 工作流 → 产出
- ⚠️ 检查：脚本失败了怎么办？有降级路径吗？

## 子步骤收尾：自反思循环（Ralph Loop）

**产出 DESIGN.md 后必须先自反思再提交用户。**

1. AI 逐项过 scripts/validate.py 结构检查，确认无 CRITICAL/HIGH 问题
2. AI 按 subjective 清单自问 4 个问题
3. 发现问题 → 修正文档 → 重检（最多3轮）
4. 全过或3轮后 → 提交用户确认 → DESIGN 冻结

> ⚡ 轻量通道+客观全过 → 可跳过主观自检

## DESIGN 阶段收尾

### Step 4：确认输出文件约定（条件触发）

**触发判断**：检查 skill 是否涉及以下文件产出动作：
- 关键词检测：`生成` `导出` `保存` `输出` `写入` `下载` `create` `export` `save` `output` `write` `generate` `download` `produce`
- 示例信号：工作报告、导出 CSV、生成代码、保存配置、下载图片

> 若不涉及上述动作 → 跳过本步骤，直接进入 Phase 3 BUILD。

**若触发，进入强制确认**：

> ⚠️ **强制阻塞**：以下 4 项全部确认前，不能进入 Phase 3 BUILD。参考 Phase 1 的强制阻塞模式。

**确认交互模板**：

```
你的 skill 会产出文件。我需要确认一下文件输出的约定，避免后续返工：

1. 【输出目录】文件输出到哪里？
   a) 当前工作目录（执行命令时所在的目录）
   b) skill 固定子目录（如 outputs/、data/）
   c) 用户每次指定目录
   d) 系统临时目录 /tmp/
   （请选择或说明你的方案）

2. 【文件命名】文件名叫什么？多次运行会不会覆盖？
   a) 固定名称（如 report.md，每次覆盖）
   b) 带时间戳（如 report-20260620-1430.md）
   c) 用户指定名称（每次交互时询问）
   d) UUID 随机名（如 a1b2c3d4.json）
   （请选择或说明你的方案）

3. 【文件格式】文件是什么格式？
   a) Markdown (.md)
   b) JSON (.json)
   c) CSV (.csv)
   d) 纯文本 (.txt)
   e) 其他（请说明）
   （请选择或说明你的方案）

4. 【错误输出】如果写入失败或磁盘满了怎么办？
   a) 向用户报错（通过对话输出错误信息）
   b) 写入错误日志文件
   c) 降级到默认路径重试
   d) 返回结构化错误让调用者处理
   （请选择或说明你的方案）
```

**确认后记录格式**（写入 SKILL.md 的 `## 输出文件约定` 章节）：

```markdown
## 输出文件约定
- **输出目录**：{用户选择，如 outputs/}
- **文件命名**：{用户选择，如 report-YYYYMMDD-HHMM.md}
- **文件格式**：Markdown (.md)
- **错误输出**：对话中报错 + 返回结构化错误
- **路径示例**：outputs/report-20260620-1430.md
```
