# README 最低内容标准

> SCK 创建和审查 skill 时，README.md 必须覆盖以下 5 个段落。

## 必须包含

| # | 段落 | 核心问题 | 最小要求 |
|:-:|------|------|------|
| 1 | **是什么** | 一句话说清这个 skill 做什么 | 不超过 50 字，不含"功能强大""持续更新"等空话 |
| 2 | **怎么用** | 用户打开后怎么上手、怎么触发 | 需独立「**快速开始**」（上手方式）和「**触发方式**」（具体触发词）两个子段落；触发方式至少 2 个具体示例（不是泛泛的"说XX即可"） |
| 3 | **核心能力** | 能做什么、和什么联动 | 列表或表格，每条一个动词开头 |
| 4 | **资源引用** | 更多信息去哪看 | 指向 SKILL.md / CHANGELOG.md / DESIGN.md 的链接 |
| 5 | **文件结构** | 文件在哪、各是什么 | 树形结构，每个文件一行说明 |

## 推荐包含

| # | 段落 | 场景 |
|:-:|------|------|
| 6 | **边界与注意事项** | 有已知局限、不适用场景、依赖前提时务必写清 |
| 7 | **使用场景** | 有多个使用入口或多个用户画像时 |
| 8 | **进阶玩法** | 有隐性能力（如跨会话记忆、自炼新星）时 |
| 9 | **常见问题** | 已知用户会踩的坑 |

## 评分标准

quality-audit.py 的 `readme_quality` 维度按此标准评估（8 项自动检查：概述/标题 + 快速开始 + 触发方式 + 核心能力 + 适合谁 + 边界与注意事项 + 资源引用 + 文件结构）：
- **6/6**：5 项必须全部覆盖 + 至少 1 项推荐
- **4-5/6**：5 项必须全部覆盖
- **2-3/6**：覆盖了 3-4 项必须
- **1/6**：仅覆盖了 1-2 项
- **0/6**：README 不存在或仅占位

## 反模式

- ❌ 空泛描述："强大的XX工具""持续更新中"
- ❌ 只说"加载skill后使用"，不给具体触发词
- ❌ 能力列表只写名词（"对话""辩论"），不写动词
- ❌ 文件树没有注释
- ❌ 版本号硬编码（应用委托模式："当前版本见 SKILL.md"）
