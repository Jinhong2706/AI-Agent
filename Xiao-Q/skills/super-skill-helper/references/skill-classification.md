# 技能分类指南

本指南帮助你将技能正确分类，以便采用最佳的设计模式和组织结构。

## 4 类型技能分类

### 1. Tool（工具型）

**定义**：执行特定技术操作或处理特定文件格式的技能。

#### 特征

- ✅ 明确的输入/输出格式
- ✅ 可客观验证的结果
- ✅ 通常需要脚本支持
- ✅ 低自由度（需要精确执行）
- ✅ 错误处理至关重要

#### 适用场景

- 文件格式转换（PDF ↔ Word, Markdown ↔ HTML）
- 数据处理（CSV 清洗、JSON 解析）
- 图像处理（缩放、裁剪、格式转换）
- 代码生成（模板填充、样板代码）
- 自动化操作（文件重命名、批量处理）

#### 设计要点

1. **包含测试用例**：验证功能的正确性
2. **优先使用脚本**：确保确定性执行
3. **明确错误处理**：处理各种异常情况
4. **提供使用示例**：展示典型用法

#### 示例结构

```
pdf-processor/
├── SKILL.md
├── scripts/
│   ├── convert_pdf.py
│   ├── extract_text.py
│   └── merge_pdfs.py
├── evals/
│   └── evals.json
└── assets/
    └── sample.pdf
```

#### 触发条件设计

```yaml
description: 处理 PDF 文件的技能。当需要将 PDF 转换为其他格式、提取 PDF 文本、合并多个 PDF、
分割 PDF、旋转 PDF 页面或压缩 PDF 时使用。包括批量处理场景。
```

---

### 2. Workflow（工作流型）

**定义**：指导完成多步骤流程或复杂任务的技能。

#### 特征

- ✅ 多步骤顺序执行
- ✅ 可能需要条件判断
- ✅ 中到高自由度
- ✅ 需要清晰的导航指引
- ✅ 决策点明确

#### 适用场景

- 应用部署流程（开发 → 测试 → 生产）
- 数据分析流水线（收集 → 清洗 → 分析 → 可视化）
- 内容发布流程（起草 → 审查 → 发布 → 推广）
- 项目管理流程（规划 → 执行 → 监控 → 收尾）
- 客户 onboarding 流程

#### 设计要点

1. **使用渐进式披露**：按步骤组织内容
2. **提供决策树**：清晰的条件分支
3. **按领域组织参考资料**：避免加载无关内容
4. **包含检查清单**：确保不遗漏步骤

#### 示例结构

```
app-deployment/
├── SKILL.md
├── references/
│   ├── aws-deployment.md
│   ├── gcp-deployment.md
│   ├── azure-deployment.md
│   └── checklist.md
└── scripts/
    └── validate_config.py
```

#### 触发条件设计

```yaml
description: 应用部署工作流。当需要将应用部署到云平台时使用，支持 AWS、GCP、Azure。
包括部署前检查、配置验证、部署执行、部署后验证。适用于 Web 应用、API 服务、微服务部署。
```

---

### 3. Capability（能力型）

**定义**：增强 AI 在特定领域专业能力的技能。

#### 特征

- ✅ 领域专业知识库
- ✅ 判断和决策支持
- ✅ 高自由度（需要上下文理解）
- ✅ 大量参考资料
- ✅ 推理框架

#### 适用场景

- 财务分析（报表分析、风险评估）
- 法律咨询（合同审查、合规检查）
- 医学研究（文献综述、数据分析）
- 技术咨询（架构设计、技术选型）
- 市场分析（竞争分析、趋势预测）

#### 设计要点

1. **组织清晰的领域知识架构**：便于导航
2. **使用参考资料管理大量内容**：保持 SKILL.md 简洁
3. **提供推理框架**：而非硬性规则
4. **包含案例研究**：展示实际应用

#### 示例结构

```
finance-advisor/
├── SKILL.md
├── references/
│   ├── financial-statements.md
│   ├── ratio-analysis.md
│   ├── valuation-methods.md
│   ├── industry-benchmarks.md
│   └── case-studies/
│       ├── tech-company.md
│       └── retail-company.md
└── assets/
    └── templates/
        └── analysis-template.xlsx
```

#### 触发条件设计

```yaml
description: 财务分析专家。当需要分析公司财务状况、评估投资价值、进行财务预测或解释财务指标时使用。
包括利润表分析、资产负债表分析、现金流量表分析、财务比率计算、行业对标、估值建模。
```

### 4. Scenario（场景型）

**定义**：针对特定使用场景或用户角色定制的技能。

#### 特征

- ✅ 角色扮演的 AI 行为
- ✅ 特定场景的响应模式
- ✅ 高自由度
- ✅ 强调沟通风格
- ✅ 情感智能

#### 适用场景

- 客服助手（处理投诉、回答咨询）
- 面试教练（模拟面试、提供反馈）
- 创意伙伴（头脑风暴、创意生成）
- 学习导师（解释概念、出题测试）
- 健康顾问（提供建议、激励支持）

#### 设计要点

1. **明确定义人设和语气**：保持一致性
2. **提供场景化的示例对话**：展示期望行为
3. **平衡灵活性和一致性**：允许变化但有边界
4. **包含边界情况处理**：什么不应该做

#### 示例结构

```
customer-service/
├── SKILL.md
├── references/
│   ├── tone-guide.md
│   ├── common-scenarios.md
│   ├── escalation-policy.md
│   └── product-faq.md
└── assets/
    └── response-templates/
        ├── apology.md
        └── solution.md
```

#### 触发条件设计

```yaml
description: 客服助手。当需要处理客户咨询、投诉、退换货请求或产品问题时使用。
语气友好专业，善于倾听和共情。包括安抚不满客户、提供解决方案、升级复杂问题。
```

## 分类决策树

使用以下决策树帮助确定技能类型：

```
开始
│
├─ 是否执行特定的技术操作？
│  ├─ 是 → Tool（工具型）
│  └─ 否 → 继续
│
├─ 是否涉及多步骤流程？
│  ├─ 是 → Workflow（工作流型）
│  └─ 否 → 继续
│
├─ 是否需要深厚的领域专业知识？
│  ├─ 是 → Capability（能力型）
│  └─ 否 → 继续
│
└─ 是否需要特定的沟通风格或角色扮演？
   ├─ 是 → Scenario（场景型）
   └─ 否 → 重新评估需求
```

## 混合类型处理

有些技能可能跨越多个类型。处理策略：

### 策略 1：主类型 + 子类型

确定主要类型，将其他作为子功能。

```
主要类型：Tool
子功能：包含 Workflow（多步骤处理）
```

### 策略 2：模块化组织

将不同类型的功能模块化到不同的参考文件中。

```
main-skill/
├── SKILL.md（核心工作流）
├── references/
│   ├── tool-functions.md
│   ├── workflow-guide.md
│   └── domain-knowledge.md
```

### 策略 3：分层设计

顶层是 Workflow，底层是 Tool。

```
顶层：指导整体流程
中层：调用专业能力
底层：执行具体操作
```

## 类型转换

随着技能演进，类型可能转换：

### Tool → Workflow

当工具功能增多，需要编排多个操作时。

**转换信号**：
- 用户经常组合多个操作
- 需要条件判断
- 操作步骤超过 3 个

### Workflow → Capability

当工作流需要专业判断时。

**转换信号**：
- 需要领域知识做决策
- 用户询问"为什么"而不仅是"怎么做"
- 需要处理模糊情况

## 最佳实践

### 1. 早期确定类型

在技能设计初期就确定类型，影响整体架构。

### 2. 类型驱动设计

根据类型选择：
- 目录结构
- 文档组织
- 测试策略
- 触发条件设计

### 3. 定期重新评估

随着技能演进，定期重新评估类型是否仍然合适。

### 4. 用户反馈验证

通过用户使用情况验证类型选择是否正确：
- 用户是否找到需要的内容？
- 技能是否在正确场景触发？
- 输出是否符合期望？

## 常见错误

### ❌ 错误 1：所有技能都做成 Tool

**问题**：忽视需要专业判断或工作流的场景。

**解决**：使用决策树正确分类。

### ❌ 错误 2：类型混杂无组织

**问题**：SKILL.md 中包含不相关的多种类型内容。

**解决**：模块化组织，清晰导航。

### ❌ 错误 3：忽视类型特定的测试策略

**问题**：用相同方式测试所有类型。

**解决**：
- Tool：客观断言
- Workflow：流程完整性
- Capability：专业准确性
- Scenario：沟通质量
