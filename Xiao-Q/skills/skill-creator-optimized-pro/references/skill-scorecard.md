# Skill 评分表草案（100 分制）

本文用于对 Skill 包进行结构化评审、批量审核与交付前质检。评分以“是否能被稳定触发、渐进加载、可执行、可审计、可交付”为核心目标，满分 100 分。

## 一、评分总览

| 维度 | 分值 | 评审重点 |
|---|---:|---|
| 1. 触发与适用边界 | 15 | Skill 何时使用、何时不用、触发条件是否清晰 |
| 2. 结构完整性 | 15 | YAML front matter、目录、入口文档、引用材料是否完整 |
| 3. 渐进式加载 | 15 | 是否避免一次性加载过多内容，是否有分层阅读路径 |
| 4. 可执行性 | 20 | 步骤是否能落地执行，工具/输入/输出是否明确 |
| 5. 可审计性 | 10 | 过程、引用、决策、检查点是否可追溯 |
| 6. 异常处理 | 10 | 缺输入、权限、工具失败、质量风险等场景是否有处理策略 |
| 7. 交付契约 | 15 | 输出路径、命名、格式、验收标准、禁止事项是否明确 |
| **合计** | **100** |  |

## 二、详细评分标准与扣分规则

### 1. 触发与适用边界（15 分）

**满分标准：**
- 明确说明 Skill 的使用场景、典型任务、输入信号和触发关键词。
- 明确说明不适用场景，避免误触发。
- 触发描述具体、可判断，不依赖模糊表达。
- 能与相近 Skill 区分，避免职责重叠。

**扣分规则：**
- 缺少触发条件：扣 6-10 分。
- 触发条件过泛，例如“处理文档时使用”：扣 3-6 分。
- 未说明不适用场景：扣 2-4 分。
- 与其他 Skill 边界明显重叠且无区分说明：扣 3-5 分。
- 描述存在歧义，审核者无法判断是否应调用：扣 2-5 分。

### 2. 结构完整性（15 分）

**满分标准：**
- 包含可解析的 YAML front matter，字段齐全且格式正确。
- 主入口文档清楚说明目标、工作流、输入、输出、限制。
- references、examples、assets 等辅助材料按需组织，命名清晰。
- 文件层级合理，不把所有内容堆在入口文档中。
- 无缺失引用、死链、空文件、重复文件或明显无关文件。

**扣分规则：**
- 缺少 YAML front matter：扣 8-12 分。
- YAML 存在解析错误：扣 6-10 分。
- 关键字段缺失或描述无效：扣 3-6 分。
- 目录结构混乱，入口与引用材料关系不清：扣 3-6 分。
- 引用文件不存在、路径错误或文件名不一致：每处扣 1-3 分，累计不超过 6 分。
- 包含明显运行态文件、临时日志或无关产物：扣 2-5 分。

### 3. 渐进式加载（15 分）

**满分标准：**
- 入口文档保持精简，优先提供决策树、执行摘要和加载顺序。
- 大体量说明、模板、示例、检查清单放入 references 或 examples。
- 明确“先读什么、按需读什么、什么情况下不用读”。
- 能降低上下文占用，避免无关信息进入任务执行。
- 对复杂任务支持分阶段加载与阶段性校验。

**扣分规则：**
- 所有内容集中在单一长文档中，无分层机制：扣 6-10 分。
- 缺少加载顺序或按需引用说明：扣 3-6 分。
- 示例、模板、背景资料混杂在执行步骤中：扣 2-5 分。
- 入口文档冗长且影响快速判断：扣 2-5 分。
- 存在重复内容，导致加载成本明显增加：扣 2-4 分。

### 4. 可执行性（20 分）

**满分标准：**
- 工作流按步骤组织，动作、判断、输入、输出明确。
- 每一步都能被执行者直接操作，而不是停留在原则描述。
- 明确使用哪些工具、何时使用、使用前后需要检查什么。
- 对常见任务给出模板、示例或检查清单。
- 能指导执行者完成从输入解析到最终交付的闭环。

**扣分规则：**
- 只有理念或原则，缺少可操作步骤：扣 8-14 分。
- 步骤顺序不清，执行者无法稳定复现：扣 4-8 分。
- 输入、输出、依赖条件未定义：扣 3-7 分。
- 工具使用说明缺失或与任务不匹配：扣 2-6 分。
- 示例不可用、模板不完整或与正文矛盾：扣 2-5 分。
- 存在明显无法执行、互相冲突的指令：扣 5-10 分。

### 5. 可审计性（10 分）

**满分标准：**
- 明确要求记录关键判断、引用来源、文件路径和版本差异。
- 有评审检查点，能判断每阶段是否达标。
- 对外部资料、用户文件、生成内容的来源有追溯要求。
- 交付前包含自检或复核清单。

**扣分规则：**
- 缺少任何审计或复核要求：扣 5-8 分。
- 只要求“检查”，但没有检查项：扣 2-4 分。
- 引用来源、文件路径或版本信息不可追溯：扣 2-5 分。
- 未区分事实、推断与生成内容：扣 2-4 分。

### 6. 异常处理（10 分）

**满分标准：**
- 覆盖输入缺失、格式异常、权限不足、工具失败、结果不确定等场景。
- 明确哪些情况应停止、降级、请求确认或更换路径。
- 有质量风险提示机制，避免带病交付。
- 对危险操作、覆盖文件、不可逆修改有额外保护。

**扣分规则：**
- 缺少异常处理章节：扣 5-8 分。
- 只笼统写“遇到错误请处理”，无具体策略：扣 3-5 分。
- 未处理缺输入或用户文件不可访问：扣 2-4 分。
- 未说明工具失败后的替代路径或停止条件：扣 2-4 分。
- 对覆盖、删除、权限、隐私等风险无保护：扣 2-5 分。

### 7. 交付契约（15 分）

**满分标准：**
- 明确交付物目录、命名规则、格式要求、引用方式和验收标准。
- 明确禁止写入的位置、禁止修改的文件、禁止输出的内容。
- 对多文件交付、打包、清理临时文件有规则。
- 最终回复需包含交付路径、完成状态、核心结果和风险提示。
- 输出格式能直接被用户使用或复核。

**扣分规则：**
- 缺少交付路径或输出格式要求：扣 5-8 分。
- 命名、覆盖、打包规则不清：扣 2-5 分。
- 未说明验收标准：扣 3-6 分。
- 未约束禁止事项，可能污染工作区或修改原文件：扣 3-6 分。
- 最终汇报要求不完整：扣 2-4 分。

## 三、通用扣分项

以下问题可在对应维度扣分，也可作为通用扣分项追加扣分。总分最低为 0 分。

| 问题 | 建议扣分 |
|---|---:|
| 指令之间存在明显冲突，且无优先级说明 | 5-15 分 |
| 内容大量重复、空泛或不可验证 | 3-10 分 |
| 与安全、隐私、合规要求冲突 | 10-30 分，严重时直接判为不合格 |
| 需要外部依赖但未说明安装、权限或替代方案 | 3-8 分 |
| 示例与实际工作流不一致 | 2-6 分 |
| 语言混乱，影响执行者理解 | 2-6 分 |
| 文件引用、路径或命名错误 | 每处 1-3 分，累计不超过 10 分 |
| 未保留用户原始目标，过度发散 | 3-8 分 |

## 四、评级区间

| 分数 | 等级 | 结论 | 建议 |
|---:|---|---|---|
| 90-100 | A | 可直接发布 | 仅做轻量校对和回归验证 |
| 80-89 | B | 基本可用 | 修复扣分项后发布，重点检查边界和交付契约 |
| 70-79 | C | 有条件可用 | 需要补齐关键结构、执行步骤或异常处理后再发布 |
| 60-69 | D | 风险较高 | 不建议直接发布，应进行系统性重写或重构 |
| 0-59 | F | 不合格 | 暂停发布，重新定义目标、结构与执行流程 |

## 五、批量审核建议

### 1. 批量审核流程

1. **元数据预检**：先批量检查 YAML front matter 是否存在、可解析、字段完整。
2. **目录结构扫描**：检查入口文档、references、examples、assets 等目录是否符合约定。
3. **触发边界审核**：按 Skill 名称、description、适用场景判断是否存在职责重叠或触发过泛。
4. **渐进加载审核**：统计入口文档长度、引用文件数量、加载顺序说明，识别“单文件堆料”。
5. **可执行性抽检**：随机抽取任务场景，按文档步骤模拟执行，检查是否能闭环交付。
6. **异常处理审核**：统一检查是否覆盖缺输入、权限失败、工具失败、不可逆操作、质量不确定等场景。
7. **交付契约审核**：检查输出路径、命名、格式、验收、最终汇报要求是否明确。
8. **评分归档**：为每个 Skill 保留评分表、主要扣分原因、修改建议和复审状态。

### 2. 批量审核抽样策略

- **新建 Skill**：100% 全量审核。
- **高频使用 Skill**：每次变更后全量审核，至少覆盖触发、可执行性、交付契约三项。
- **低风险小改动**：可只审核变更文件及其直接引用文件。
- **大版本重构**：按新建 Skill 标准重新评分。
- **同类 Skill 批量创建**：先抽检 20%-30%，若发现结构性问题，则扩大到 100% 全量审核。

### 3. 批量审核记录字段

建议为每个 Skill 记录以下字段：

| 字段 | 说明 |
|---|---|
| Skill 名称 | 被审核 Skill 的目录名或注册名 |
| 审核日期 | 完成评分的日期 |
| 审核人/审核方式 | 人工、自动化、人工加自动化 |
| 总分 | 0-100 分 |
| 等级 | A/B/C/D/F |
| 主要扣分项 | 列出前三个最关键问题 |
| 阻断项 | 是否存在禁止发布的问题 |
| 修改建议 | 需要补充、删除、重构或验证的内容 |
| 复审结论 | 通过、待修复、驳回 |

## 六、阻断发布条件

出现以下任一情况，即使总分达到 70 分，也建议判定为“暂缓发布”：

- YAML front matter 缺失或不可解析。
- Skill 触发条件严重不清，可能频繁误触发。
- 工作流无法实际执行，不能形成稳定交付闭环。
- 交付契约缺失，可能写错目录、覆盖用户文件或污染工作区。
- 存在安全、隐私、合规风险。
- 存在与上级系统规则或用户明确要求冲突的指令。

## 七、建议评分口径

- 优先评价“能否稳定完成任务”，而不是文档写得是否漂亮。
- 同一问题不要在多个维度重复重扣；若确实影响多个维度，应说明原因。
- 对关键阻断问题要明确标注，不要被总分掩盖。
- 修改建议应具体到文件、章节或字段，避免只写“需要优化”。
- 评分后应给出是否可发布的明确结论。
