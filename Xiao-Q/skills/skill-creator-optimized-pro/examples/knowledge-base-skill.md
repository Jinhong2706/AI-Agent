# 知识库类 Skill 示例

> 本示例用于展示如何编写“知识库类 Skill”。知识库类 Skill 的核心价值，是把可复用的业务知识、流程规范、术语口径、引用规则和交付标准沉淀为稳定指令，帮助助手在需要查阅、引用或补充知识时保持一致、可靠、可追溯。

## 1. YAML front matter 示例

以下为一个知识库类 Skill 的注册元数据示例。实际使用时，请根据技能名称、业务范围和触发条件调整字段内容。

```yaml
---
name: hotel-operations-knowledge-base
description: "酒店运营知识库 Skill：用于检索、引用和维护酒店运营 SOP、服务标准、常见客诉处理口径及培训材料。"
category: knowledge-base
tags:
  - knowledge-base
  - sop
  - hotel-operations
  - customer-service
version: "1.0.0"
author: "office-raccoon"
---
```

### 字段说明

- `name`：技能唯一标识，建议使用小写英文与短横线，避免空格和中文。
- `description`：清晰描述技能用途、业务边界和主要能力；如包含英文冒号加空格，建议整体加引号，避免 YAML 解析异常。
- `category`：知识库类技能建议标记为 `knowledge-base`。
- `tags`：用于检索、归类和触发匹配，建议包含业务域、资料类型和高频任务关键词。
- `version`：用于区分知识库规则版本。
- `author`：维护者或团队标识。

## 2. 适用场景

本类 Skill 适合以下任务：

1. **知识检索**：从企业制度、SOP、FAQ、培训材料、项目文档中查找答案。
2. **口径统一**：确保对外回复、内部汇报、客服话术、培训说明使用统一表述。
3. **材料引用**：在报告、纪要、方案中引用知识库内容，并标明来源。
4. **知识沉淀**：将用户确认过的业务规则、复盘结论、最佳实践补充到知识库草稿。
5. **流程辅助**：根据知识库规则生成检查清单、处理流程、风险提示和交付说明。

不适合以下任务：

- 需要实时外部事实核验但知识库未覆盖的任务。
- 涉及法律、医疗、金融等高风险专业判断且缺少权威来源的任务。
- 用户要求绕过权限、读取私密信息或泄露内部资料的任务。

## 3. 触发描述

当用户请求符合以下任一条件时，可触发知识库类 Skill：

- “查一下知识库里关于……的规定”
- “按我们之前的 SOP 写一版……”
- “用公司标准口径回复客户……”
- “把这次确认的新规则沉淀下来……”
- “引用知识库依据，整理成报告/纪要/FAQ/培训材料”
- “检查这份内容是否符合内部规范”

### 示例触发语

```text
请根据酒店前台接待 SOP，整理一份新人培训清单，并注明引用依据。
```

```text
把刚才确认的客户投诉升级规则，整理成知识库更新草稿，不要直接写入正式库。
```

## 4. 搜索规范

知识库检索应遵循“先定位、再判断、后引用”的原则。

### 4.1 查询策略

- 优先使用用户原始关键词检索。
- 同时扩展同义词、业务简称、历史叫法和相关流程名称。
- 对时间敏感内容，优先查找最新版本、最新生效日期或最近确认记录。
- 对制度类内容，优先引用正式制度、SOP、公告、管理层确认记录；其次引用会议纪要、历史问答或草稿。

### 4.2 结果判断

检索到多条结果时，应按以下优先级判断：

1. 明确标注为正式发布、已生效、当前版本的资料。
2. 与用户问题业务场景最匹配的资料。
3. 有明确责任部门、发布日期、版本号或审批记录的资料。
4. 用户在当前会话中补充或确认的信息。

### 4.3 无结果处理

如未检索到可靠结果：

- 不编造知识库内容。
- 明确说明“当前知识库未找到可引用依据”。
- 可基于通用经验给出“待确认建议”，但必须与知识库结论区分。
- 提醒用户补充资料、指定文件或确认是否需要联网/人工核验。

## 5. 写入规范

知识库写入应遵循“用户确认、边界清晰、可追溯”的原则。

### 5.1 允许写入的内容

- 用户明确确认的新规则、新流程、新口径。
- 已从正式文件中抽取并结构化的制度、SOP、FAQ。
- 项目复盘中明确沉淀的经验教训和最佳实践。
- 经用户授权整理的模板、清单、话术和交付标准。

### 5.2 禁止直接写入的内容

- 未经用户确认的推测、个人判断或临时建议。
- 涉及个人隐私、商业秘密、敏感身份信息的原文内容。
- 与现有正式制度冲突但未标注状态的内容。
- 来源不明、无法追溯、可能过期的信息。

### 5.3 写入格式建议

每条知识建议包含：

- 标题
- 适用范围
- 生效状态：草稿 / 待确认 / 已确认 / 已废止
- 来源依据
- 更新时间
- 责任人或确认人
- 正文内容
- 相关标签
- 冲突或例外说明

### 5.4 写入前确认

除非用户明确要求自动沉淀到指定知识库，否则应先输出“知识库更新草稿”，由用户确认后再写入正式库。

## 6. 引用规范

知识库类 Skill 的输出应尽量做到“结论有依据，引用可复核”。

### 6.1 引用内容

引用时应包含：

- 来源标题或文档名
- 章节、条款、页码或段落位置
- 版本号或发布日期（如有）
- 引用结论与原文之间的对应关系

### 6.2 引用表达

推荐格式：

```text
依据《前台接待 SOP》第 3.2 节“入住身份核验”要求，客人办理入住时需完成证件核验、订单匹配与押金确认。
```

如来源不完整：

```text
当前知识库仅检索到历史培训材料中的相关说明，未找到正式制度文件。以下内容建议作为待确认口径使用。
```

### 6.3 禁止事项

- 禁止伪造文档名、章节号、审批记录或来源链接。
- 禁止把模型推断包装成知识库引用。
- 禁止引用与问题无关或明显过期的材料作为依据。

## 7. 隐私边界

处理知识库内容时必须遵守以下边界：

1. 不输出无关个人信息，如身份证号、手机号、住址、账号、合同敏感字段等。
2. 不在无授权场景下披露客户名单、员工资料、交易明细、供应商报价等敏感信息。
3. 对必须使用的敏感字段进行脱敏，例如：`张三，手机号 138****5678`。
4. 用户请求导出或复述内部敏感资料时，应确认授权和必要性。
5. 若知识库中存在权限分级，必须遵守“最小必要披露”原则。

## 8. 异常处理

### 8.1 检索结果冲突

处理方式：

- 列出冲突来源。
- 比较版本、发布日期、适用范围和权威等级。
- 优先采用最新且正式生效的来源。
- 无法判断时，标注“存在冲突，需用户或责任部门确认”。

### 8.2 内容缺失

处理方式：

- 明确说明缺失项。
- 给出需要补充的资料清单。
- 可提供临时处理建议，但标注为“非知识库依据”。

### 8.3 权限不足

处理方式：

- 不尝试绕过权限。
- 告知用户当前无法访问相关资料。
- 建议用户上传授权文件、提供可访问版本或指定替代资料。

### 8.4 来源过期

处理方式：

- 标注资料可能过期。
- 尝试检索更新版本。
- 输出时提示“需以最新正式制度为准”。

## 9. 交付契约

知识库类 Skill 的交付结果应满足以下要求：

- **清晰结论**：先回答用户问题，再展开依据和说明。
- **来源可查**：凡引用知识库内容，应标明来源。
- **不确定性透明**：对缺失、冲突、过期、未确认内容明确标注。
- **隐私合规**：只披露完成任务所必需的信息，并对敏感信息脱敏。
- **可执行**：如输出流程、清单、话术或模板，应能直接被用户使用。
- **不越权写入**：正式写入知识库前必须满足授权和确认要求。

### 推荐输出结构

```markdown
## 结论

## 引用依据

## 适用范围与限制

## 可执行建议 / 流程 / 模板

## 待确认事项
```

## 10. 质量检查清单

交付前请逐项检查：

- [ ] 是否使用了有效 YAML front matter，且可被 YAML 解析器解析？
- [ ] `description` 中是否存在可能破坏 YAML 的冒号、引号或特殊字符，并已正确处理？
- [ ] 是否明确说明该 Skill 的适用场景和不适用场景？
- [ ] 是否提供了清晰的触发描述和示例触发语？
- [ ] 是否说明搜索策略、结果优先级和无结果处理方式？
- [ ] 是否区分“知识库正式内容”“待确认草稿”“模型建议”？
- [ ] 是否规定写入前必须获得用户授权或确认？
- [ ] 是否提供引用格式，并禁止伪造来源？
- [ ] 是否包含隐私保护和最小必要披露要求？
- [ ] 是否覆盖冲突、缺失、权限不足、来源过期等异常场景？
- [ ] 是否定义交付契约，保证输出可执行、可复核、边界清晰？
- [ ] 是否未修改知识库类 Skill 范围以外的文件？

## 11. 简短模板

可将以下模板复制到新知识库 Skill 中作为起点：

```markdown
---
name: example-knowledge-base
description: "示例知识库 Skill：用于检索、引用和维护指定业务域的知识资料。"
category: knowledge-base
tags:
  - knowledge-base
  - example
version: "1.0.0"
author: "office-raccoon"
---

# 示例知识库 Skill

## 适用场景

用于回答、引用和维护指定业务域的知识资料。

## 触发描述

当用户要求查询知识库、引用内部规则、沉淀新口径或检查内容是否符合规范时触发。

## 搜索规范

优先检索正式、最新、与业务场景最匹配的资料；无可靠结果时不得编造。

## 写入规范

仅在用户明确确认或授权后写入；未确认内容先形成更新草稿。

## 引用规范

引用需标明来源标题、章节、版本或日期；不得伪造来源。

## 隐私边界

遵守最小必要披露原则，对敏感信息脱敏。

## 异常处理

对冲突、缺失、权限不足、来源过期分别标注并请求确认。

## 交付契约

输出结论、依据、限制、可执行建议和待确认事项。

## 质量检查清单

- [ ] YAML front matter 可解析
- [ ] 搜索、写入、引用规则完整
- [ ] 隐私和异常边界清晰
- [ ] 交付结果可复核、可执行
```
