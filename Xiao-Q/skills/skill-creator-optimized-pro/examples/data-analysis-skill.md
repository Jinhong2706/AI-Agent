# 数据分析类 Skill 示例

以下示例用于说明如何编写一个面向数据分析任务的 Skill。内容可作为技能包中的示例文档，帮助创作者理解：如何定义触发条件、如何读取和清洗数据、如何开展分析与可视化，以及如何约束最终交付物质量。

## 1. YAML front matter 示例

```yaml
---
name: data-analysis-assistant
description: "用于处理 Excel、CSV、JSON 等结构化数据的数据读取、清洗、统计分析、可视化和结果报告生成任务。"
version: "1.0.0"
author: "skill-creator"
tags:
  - 数据分析
  - 数据清洗
  - 可视化
  - 报告生成
triggers:
  - "分析这份数据"
  - "帮我清洗表格"
  - "生成数据洞察报告"
  - "做一份销售/运营/财务数据分析"
inputs:
  - type: file
    formats:
      - .xlsx
      - .xls
      - .csv
      - .json
  - type: text
    description: "用户补充的分析目标、口径说明、筛选条件或输出格式要求。"
outputs:
  - type: markdown
    description: "数据分析结论与可执行建议。"
  - type: file
    formats:
      - .xlsx
      - .png
      - .pdf
      - .zip
permissions:
  file_read: true
  file_write: true
  network: false
---
```

> 注意：YAML front matter 中的 `description` 建议使用引号包裹，避免冒号、中文标点或特殊字符导致解析失败。

## 2. 适用场景

本 Skill 适用于以下结构化数据分析任务：

- 销售数据分析：销售额、订单量、客单价、渠道贡献、区域表现、同比/环比变化。
- 运营数据分析：用户增长、留存、活跃、转化漏斗、活动效果、异常波动识别。
- 财务数据分析：收入、成本、毛利、费用结构、预算执行、利润变化归因。
- 客户数据分析：客户分层、复购、流失预警、RFM 分析、客户价值排序。
- 表格清洗整理：字段标准化、缺失值处理、重复数据去除、格式统一、异常值标记。
- 管理汇报支持：从原始数据中提炼核心指标、发现问题、输出结论和建议。

不适用于以下任务：

- 无数据文件、无字段说明且无法确认分析目标的任务。
- 需要实时数据库权限、内部系统权限或外部付费接口才能完成的任务。
- 需要凭空编造数据、篡改真实结果或隐藏异常的数据处理任务。

## 3. 触发描述

当用户提出以下意图时，可以触发该 Skill：

- “帮我分析这份 Excel。”
- “把这份 CSV 清洗一下，并输出关键发现。”
- “根据这份订单数据生成经营分析报告。”
- “看看哪些指标异常，给我做图。”
- “帮我做同比、环比、TOP 排名和趋势分析。”
- “把分析结果整理成一份管理层能看的结论。”

触发后应优先确认三类信息：

1. **分析目标**：用户想解决什么业务问题，例如增长、利润、效率、异常、客户结构等。
2. **数据范围**：文件名称、表名、时间范围、字段含义、是否存在多个数据源。
3. **交付形式**：只要文字结论，还是需要 Excel、图表、PDF、PPT 或 ZIP 打包。

若用户没有明确说明分析目标，应先基于字段结构进行探索性分析，并在结论中说明默认分析口径。

## 4. 数据读取与清洗

### 4.1 数据读取

推荐流程：

1. 检查文件格式：`.xlsx`、`.xls`、`.csv`、`.json`。
2. 对 Excel 文件，先识别工作表名称、行列规模、表头位置。
3. 对 CSV 文件，检查编码、分隔符、是否存在乱码。
4. 对 JSON 文件，判断是列表型、嵌套型还是键值型结构。
5. 读取后记录基础信息：行数、列数、字段名、字段类型、缺失率、重复行数量。

示例伪代码：

```python
import pandas as pd

file_path = "../input.xlsx"
df = pd.read_excel(file_path, sheet_name=0)

summary = {
    "rows": len(df),
    "columns": len(df.columns),
    "fields": list(df.columns),
    "missing_rate": df.isna().mean().sort_values(ascending=False),
    "duplicate_rows": df.duplicated().sum(),
}
```

### 4.2 数据清洗

清洗动作应可追溯，不应静默改变关键业务数据。

常见清洗步骤：

- 字段名清洗：去除前后空格、统一大小写、统一中英文命名。
- 类型转换：日期字段转为日期类型，金额/数量字段转为数值类型。
- 缺失值处理：区分“真实缺失”“不适用”“未填写”“系统默认空值”。
- 重复值处理：识别完全重复、主键重复、业务重复，必要时保留明细并标记。
- 异常值处理：对负数金额、极端订单额、未来日期、无效分类值进行标注。
- 口径统一：统一币种、单位、时间粒度、渠道名称、地区名称、产品分类。

清洗结果建议输出一份“数据质量说明”，至少包括：

- 原始数据规模。
- 清洗后数据规模。
- 删除或排除的记录数量。
- 被填补或转换的字段。
- 保留但标记为异常的数据。
- 对分析结论可能产生影响的数据质量风险。

## 5. 分析/可视化

### 5.1 分析方法

根据业务目标选择合适的分析方法：

- 描述统计：总量、均值、中位数、最大值、最小值、分位数。
- 趋势分析：按日、周、月、季度统计变化趋势。
- 结构分析：渠道、区域、产品、客户类型等维度占比。
- 排名分析：TOP N、长尾分布、贡献集中度。
- 对比分析：同比、环比、目标达成率、预算偏差。
- 漏斗分析：曝光、访问、注册、下单、支付等转化步骤。
- 分层分析：客户分群、价格带、订单金额区间、产品层级。
- 异常检测：突增突降、极端值、缺失集中、重复集中。

### 5.2 可视化建议

图表应服务结论，不为画图而画图。

推荐图表类型：

- 趋势变化：折线图、面积图。
- 结构占比：条形图、堆叠条形图；谨慎使用饼图。
- 排名比较：横向条形图。
- 分布情况：直方图、箱线图。
- 相关关系：散点图、热力图。
- 目标达成：柱状图加目标线。

可视化规范：

- 标题直接表达图表主题，必要时包含时间范围。
- 坐标轴标注单位，例如“万元”“订单数”“%”。
- 数值格式统一，例如千分位、百分比、小数位。
- 图例清晰，颜色数量克制。
- 单图不要承载过多维度，避免管理层阅读负担。
- 若图表基于清洗后数据，应在备注中说明口径。

## 6. 异常处理

执行过程中如遇异常，应先定位原因，再决定降级或中止。

常见异常与处理方式：

| 异常类型 | 典型表现 | 处理方式 |
| --- | --- | --- |
| 文件无法读取 | 文件损坏、格式不匹配、密码保护 | 汇报无法读取原因，请用户提供可打开版本或密码 |
| 编码错误 | CSV 乱码、字段名异常 | 尝试常见编码；仍失败则保留样例并说明 |
| 表头不规范 | 多行标题、合并单元格、空列 | 识别真实表头；必要时输出清洗假设 |
| 字段缺失 | 缺少日期、金额、主键等关键字段 | 说明无法完成的分析项，改做可完成部分 |
| 数据量过小 | 样本不足、时间范围太短 | 降级为描述性分析，不输出过度推断 |
| 异常值过多 | 极端值影响均值或趋势 | 同时给出含异常/剔除异常两种口径 |
| 口径不明 | 用户未说明业务定义 | 使用默认口径并明确写入报告 |
| 输出失败 | 图表或文件写入异常 | 改用可生成格式，保留核心结论 |

异常处理原则：

- 不隐瞒数据问题。
- 不擅自删除关键记录。
- 不把相关性写成因果关系。
- 不在样本不足时给出确定性判断。
- 不能完成的分析项要明确说明原因和替代方案。

## 7. 交付契约

最终交付应根据用户要求选择一种或多种形式。

### 7.1 文本结论

建议结构：

1. **一句话结论**：直接回答用户最关心的问题。
2. **关键发现**：列出 3-5 条最重要的数据洞察。
3. **原因分析**：解释变化、差异或异常的可能原因。
4. **业务建议**：给出可执行动作，而不是泛泛而谈。
5. **数据口径**：说明时间范围、筛选条件、清洗规则。
6. **风险提示**：指出数据缺陷或分析限制。

### 7.2 文件交付

若生成文件，应遵循以下规则：

- 所有交付物写入 `output/` 目录。
- 文件名使用小写英文、短横线分隔，不使用时间戳、空格或中文。
- 单文件直接交付；多文件应打包为 ZIP。
- 图表文件应具备可读标题和清晰单位。
- Excel 文件可包含：原始数据摘要、清洗后数据、透视结果、指标明细、异常记录。
- 报告文件应包含结论、图表、口径、风险提示。

### 7.3 交付引用

对外回复中引用文件时，应使用 `local-file://` 加绝对路径的格式，确保用户可以点击打开。

示例：

```markdown
[数据分析报告](local-file:///absolute/path/to/output/data-analysis-report.pdf)
```

实际使用时应按运行环境要求生成正确路径，避免使用相对路径、占位符路径或不可渲染链接。

## 8. 质量检查清单

交付前应完成以下检查：

### 8.1 输入检查

- [ ] 是否确认了用户的分析目标？
- [ ] 是否识别了文件格式、表名、字段名和数据规模？
- [ ] 是否发现并说明关键字段缺失问题？
- [ ] 是否确认时间范围、筛选条件和统计口径？

### 8.2 清洗检查

- [ ] 是否记录原始数据行数和清洗后行数？
- [ ] 是否处理或标记缺失值、重复值、异常值？
- [ ] 是否说明删除、填补、转换或排除数据的规则？
- [ ] 是否避免静默修改关键业务字段？

### 8.3 分析检查

- [ ] 是否围绕用户问题输出结论，而不是只罗列指标？
- [ ] 是否区分事实、推断和建议？
- [ ] 是否避免把相关性误写成因果关系？
- [ ] 是否对同比、环比、占比等指标使用正确公式？
- [ ] 是否对样本量不足或口径不明的结论加风险提示？

### 8.4 可视化检查

- [ ] 图表是否有标题、坐标轴、单位和必要图例？
- [ ] 图表是否能支撑关键结论？
- [ ] 是否避免过多颜色、过多维度和不可读标签？
- [ ] 是否确保图表中的数值与分析表一致？

### 8.5 交付检查

- [ ] 交付物是否位于 `output/` 目录？
- [ ] 文件命名是否符合小写英文、短横线分隔规则？
- [ ] 多文件是否已打包为 ZIP？
- [ ] 回复中是否清楚说明完成内容、核心结论和风险？
- [ ] 文件链接是否使用正确的 `local-file://` 格式？

## 9. 示例输出摘要

当用户要求“分析这份销售数据”时，理想回复应类似：

```markdown
已完成销售数据分析，核心发现如下：

1. 本期销售额为 1,280 万元，环比增长 12.4%，增长主要来自华东区域和企业客户渠道。
2. TOP 10 产品贡献了 63.2% 的销售额，收入集中度较高，需要关注单品波动风险。
3. 客单价提升 8.1%，但订单量仅增长 3.5%，说明增长更多来自高价值订单，而非客户规模扩张。
4. 数据中存在 26 条金额异常记录，已单独标记，未直接删除；管理口径建议复核后再纳入正式汇报。

已输出：清洗后数据、指标明细、异常记录和趋势图。分析口径与风险说明已写入报告。
```

该示例强调：先回答业务问题，再给证据和交付物；先讲结论，再讲过程；先说明可用信息，再提示风险。
