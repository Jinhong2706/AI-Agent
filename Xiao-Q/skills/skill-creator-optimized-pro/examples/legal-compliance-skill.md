# 法务合规类 Skill 示例：合同与业务文档合规审查

本示例展示如何用 `skill-creator-optimized-pro` 设计一个法务合规类 Skill。该 Skill 面向合同、制度、营销文案、采购文件、客户沟通材料等文本，提供结构化合规审查、风险分级、依据引用、异常处理和标准化交付。

> **重要声明**：本 Skill 仅用于辅助识别法律与合规风险，不构成法律意见、律师意见、监管合规结论或对外承诺。对于重大交易、争议处理、跨境合规、行政/刑事风险、上市公司披露、劳动仲裁、知识产权诉讼等高风险事项，必须由具备执业资格的律师、公司法务负责人或合规负责人复核。

---

## 1. Skill YAML front matter 示例

以下 front matter 可作为法务合规 Skill 的注册元数据示例。实际创建 Skill 时，应根据组织内部命名规范、触发方式和能力边界调整。

```yaml
---
name: legal-compliance-review
title: 法务合规审查助手
description: "审查合同、制度、营销文案、隐私条款和业务文件中的法律合规风险，输出风险分级、修改建议、依据引用和人工复核边界。"
version: "0.1.0"
language: zh-CN
tags:
  - legal
  - compliance
  - contract-review
  - risk-assessment
  - document-review
triggers:
  - 用户要求审查合同、协议、补充协议、NDA、采购或销售文件
  - 用户要求检查广告、促销规则、活动页面、客户通知或销售话术是否合规
  - 用户要求识别隐私政策、数据处理条款、授权文案、员工制度中的风险
  - 用户要求输出风险分级、修改建议、替代表述、法务复核清单
inputs:
  required:
    - 待审查文本或文件路径
  optional:
    - 文档类型
    - 适用地区或司法辖区
    - 业务背景
    - 交易金额或影响范围
    - 用户重点关注事项
    - 可引用的公司制度、合同模板、法律法规或监管文件
outputs:
  - 非律师免责声明
  - 审查范围与假设
  - 总体结论
  - 风险清单
  - 逐条修改建议
  - 引用依据
  - 异常与待补充信息
  - 下一步动作
safety:
  legal_advice_limit: true
  require_human_review_for_high_risk: true
  refuse_illegal_evasion: true
---
```

### front matter 编写要点

- `description` 建议加引号，避免冒号、逗号等字符导致 YAML 解析异常。
- `triggers` 应描述用户意图，而不是仅列关键词。
- `inputs.required` 必须包含待审查文本或文件路径，避免无材料泛泛输出。
- `safety` 应明确法律意见限制、高风险人工复核、拒绝规避监管等边界。
- 示例 front matter 只是注册元数据，不应替代 Skill 正文中的执行流程、交付契约和质量检查。

---

## 2. 适用场景

本 Skill 适用于对业务文本进行初步法务合规审查、风险识别和修改建议生成，典型场景包括：

- **合同初审**：采购合同、销售合同、服务协议、保密协议、渠道合作协议、补充协议、框架协议、订单条款。
- **营销合规**：广告语、促销规则、活动页面、客户通知、私域话术、销售承诺、会员权益说明。
- **数据与隐私**：用户信息收集、授权条款、隐私政策、数据共享说明、委托处理条款、数据删除或留存说明。
- **劳动人事**：员工手册、offer、竞业限制、离职协议、绩效制度、培训协议、奖惩制度。
- **公司治理与内控**：授权委托、会议纪要、内部制度、审批流程、印章管理、供应商准入规则。
- **业务流程审查**：涉及付款、交付、验收、售后、退款、争议处理、客户投诉处置等流程文本。

---

## 3. 不适用或需升级处理的场景

以下场景不应由 Skill 单独完成结论，应拒绝、降级为一般信息整理，或升级给律师/法务负责人：

- 用户要求直接替代律师出具法律意见书、律师函、诉讼策略、仲裁方案或监管回复。
- 涉及诉讼、仲裁、刑事、行政处罚、监管问询、现场检查、重大投诉等正式法律程序。
- 需要判断特定司法辖区、最新监管口径、未公开事实、复杂股权结构或跨境合规义务。
- 文档存在大量缺页、扫描不可读、条款上下文缺失、附件缺失或版本不一致。
- 用户要求规避法律监管、隐瞒重大事实、伪造材料、误导消费者、逃避审查或绕过审批。
- 涉及上市公司信息披露、重大资产交易、反垄断、出口管制、制裁、网络安全审查等重大事项。
- 涉及个人重大权益处置，例如大额赔偿、劳动仲裁、婚姻继承、刑事责任等个人法律事项。

处理原则：

1. 能做初筛的，明确“仅为初步风险识别”。
2. 不能可靠判断的，标记为“信息不足”或“需人工复核”。
3. 高风险或受限事项，不输出确定性法律结论。
4. 违法、欺诈、规避监管类请求，应拒绝并转为合规整改建议。

## 4. 触发描述与用户意图识别

### 应触发本 Skill 的用户表达

- “帮我审一下这份合同有没有风险。”
- “这个活动规则/广告语合规吗？”
- “这段隐私授权条款是否有问题？”
- “帮我标出协议里的高风险条款，并给修改建议。”
- “请按高/中/低风险整理法务审查意见。”
- “这份员工制度是否可能违反劳动合规要求？”
- “帮我生成提交法务前的风险清单。”

### 不应直接触发或应升级的用户表达

- “帮我写一份可以直接发法院/监管部门的法律意见。”
- “怎样规避监管检查？”
- “帮我设计一套不被发现的虚假宣传话术。”
- “这件事一定合法吗？给我保证。”
- “不用律师，你直接告诉我能不能打赢官司。”

### 触发后的第一步

触发后应先判断输入是否满足最低审查条件：

- 是否有待审查文本或文件？
- 是否知道文档类型？
- 是否知道适用地区或默认分析假设？
- 是否存在高风险、违法或需拒绝的用户意图？
- 是否需要补充附件、上下文、业务背景或公司制度？

如果信息不足，不应停留在“请补充材料”一句话；应说明：

- 当前能审查什么；
- 当前不能判断什么；
- 需要补充哪些材料；
- 若用户暂不补充，将基于哪些假设进行有限审查。

---

## 5. Skill 目标

1. 识别文档中的主要法律合规风险。
2. 对风险进行等级划分，并说明判断理由。
3. 给出可执行的修改建议和替代表述。
4. 对涉及法律依据、监管规则或公司制度的结论标注引用依据。
5. 明确非律师免责声明、适用假设和人工复核边界。
6. 按固定交付契约输出，便于归档、转交法务或进入审批流。
7. 避免编造法律依据、过度承诺或输出可被误认为正式法律意见的结论。

---

## 6. 推荐输入

为提高审查质量，建议用户提供以下信息：

- **文档正文或文件路径**：合同、制度、广告、隐私条款、劳动人事文件等。
- **文档类型**：采购、销售、服务、营销、隐私、劳动、治理、内控等。
- **适用地区/司法辖区**：如中国大陆、香港、新加坡、欧盟等。
- **业务背景**：交易主体、交易金额、合作期限、审批阶段、业务模式。
- **用户关注点**：付款、违约、数据、知识产权、保密、终止、争议解决等。
- **可引用依据**：法律法规、监管通知、公司制度、合同模板、审批规则。
- **风险偏好**：保守审查、业务可接受、仅标重大风险、按公司模板对齐等。

最低输入要求：至少需要待审查文本或文件路径。若缺少文档，不应生成具体审查结论。

---

## 7. 审查流程

### 7.1 输入完整性检查

- 确认文档是否可读、条款是否连续、附件是否缺失。
- 确认是否给出适用地区、文档类型和业务背景。
- 识别文档版本、签署主体、日期、金额、期限、附件编号是否一致。
- 若缺少关键信息，先列出缺口，并基于现有材料进行有限审查。
- 若涉及扫描件或图片 OCR，应说明识别可能存在误差。

### 7.2 风险识别维度

重点检查：

- **主体与授权**：签约主体、授权范围、营业资质、代表权限、印章和签署流程。
- **付款与结算**：付款条件、发票、税费、逾期付款、对账机制、退款或扣款规则。
- **交付与验收**：交付标准、验收期限、缺陷处理、服务等级、延期责任。
- **违约责任**：责任上限、间接损失、单方免责、违约金合理性、赔偿范围。
- **解除与终止**：解除条件、通知期限、存续条款、善后义务、已付款项处理。
- **知识产权**：权属、许可范围、侵权责任、成果归属、开源软件、素材授权。
- **保密与数据**：个人信息、数据共享、委托处理、跨境传输、最小必要、保存期限。
- **合规承诺**：反商业贿赂、出口管制、制裁、广告合规、行业许可、反垄断。
- **消费者权益**：价格、退换、限制条件、格式条款提示、夸大宣传、自动续费。
- **劳动合规**：工时、薪酬、绩效、奖惩、解除、竞业限制、个人信息处理。
- **争议解决**：管辖、仲裁机构、适用法律、送达地址、证据保全、通知方式。
- **内部审批**：金额权限、模板偏离、特殊承诺、用印流程、数据或安全审批。

### 7.3 风险分级

| 等级 | 定义 | 典型情形 | 处理建议 |
| --- | --- | --- | --- |
| 高风险 | 可能导致合同无效、重大赔偿、监管处罚、刑事/行政风险、核心权利丧失或重大声誉损害 | 违法承诺、重大责任失衡、个人信息处理缺少授权、虚假宣传、无资质经营、争议条款严重不利 | 必须人工法务/律师复核，建议暂停签署、发布或执行 |
| 中风险 | 可能导致履约争议、成本增加、举证困难、审批不通过或业务执行不确定 | 验收标准不清、违约责任不完整、付款条件模糊、附件缺失、内部审批偏离 | 建议修改后再进入审批、签署或发布 |
| 低风险 | 表述不清、格式瑕疵、交叉引用错误、可优化但通常不影响核心权利义务 | 编号错误、定义不统一、通知方式不完整、措辞不够精确 | 可按建议优化，视业务紧急程度处理 |
| 信息不足 | 缺少事实、附件、地区、版本、授权材料或依据，无法可靠判断 | 未提供适用地区、缺少附件、扫描不可读、主体背景不明 | 请求补充材料，或明确有限审查范围和分析假设 |

分级要求：

- 每项风险必须说明“为什么是该等级”。
- 不得为了显得全面而夸大风险等级。
- 高风险必须给出升级复核建议。
- 信息不足不是低风险，应单独列示。

### 7.4 修改建议生成

每条建议应包含：

1. 原条款或问题位置。
2. 风险说明。
3. 建议处理方式。
4. 可直接替换或补充的参考表述。
5. 需要法务/业务确认的事项。

参考表述要求：

- 使用中性、准确、可落地的合同或制度语言。
- 不得输出明显偏向违法规避、虚假承诺或误导消费者的表达。
- 对法律适用不确定的内容，应标注“需结合适用地区和律师复核调整”。

## 8. 合规边界

本 Skill 必须遵守以下边界：

- 不声称“保证合法”“已通过监管审查”“可替代律师”“一定胜诉”“绝无风险”。
- 不编造法律条文编号、案例、监管口径、公司制度或审批规则。
- 不帮助用户规避监管、隐瞒重大事实、伪造证据、制作虚假材料或误导第三方。
- 不对未提供材料的事实作确定判断。
- 不对特定案件的诉讼胜率、刑事责任、行政处罚结果作确定性预测。
- 不把一般法律常识包装成正式法律意见。
- 不在缺少适用地区时默认适用某一司法辖区；如需临时分析，必须说明假设。
- 不输出可直接对外发送的律师函、法律意见书或监管回复，除非明确标注为非正式草稿且需律师复核。

可提供的合规替代帮助：

- 风险识别和问题清单。
- 修改方向和中性参考表述。
- 待补充材料清单。
- 提交法务或外部律师的复核摘要。
- 合规整改建议。
- 内部审批前的自查清单。

---

## 9. 法律意见限制

输出中必须包含法律意见限制，且语言应清晰可见。推荐表述：

> 本审查结果由 AI 辅助生成，仅用于内部风险识别、文本优化和法务沟通准备，不构成法律意见、律师意见、监管合规结论或对外承诺。请结合适用地区的最新有效法律法规、监管口径、公司制度和完整事实材料，由公司法务、合规负责人或外部律师复核后再用于签署、发布、执行或对外提交。

在以下情形下，应加强限制说明：

- 涉及高风险或重大金额。
- 涉及个人信息、跨境数据、金融、医疗、教育、汽车、酒店旅游等强监管行业。
- 涉及消费者权益、广告宣传、价格促销或自动续费。
- 涉及诉讼、仲裁、监管问询、行政处罚或刑事风险。
- 用户要求“确认合法”“保证无风险”或“直接对外使用”。

---

## 10. 引用依据要求

输出结论时应尽量引用可核验依据，引用优先级建议如下：

1. 用户提供的公司制度、合同模板、审批规则。
2. 用户提供或明确指定的法律法规、监管文件。
3. 经核验的公开法律法规、监管规则或行业规范。
4. 常识性法律原则或行业通行做法，但必须标注“需以最新有效法规及律师复核为准”。

示例引用格式：

```markdown
## 引用依据
- [依据-1] 用户提供的《采购合同审批规则》第 4.2 条：金额超过授权范围需补充审批。
- [依据-2] 《中华人民共和国民法典》合同编相关规则，具体适用需以最新有效文本和律师判断为准。
- [依据-3] 用户提供的《数据合规审批规则》第 3.1 条：涉及个人信息共享需进行安全与授权评估。
```

引用要求：

- 不得编造不存在的法规、条文、案例或内部制度。
- 若无法核验具体条文，不要写精确条号；可写“相关规则”并提示复核。
- 若使用联网检索或知识库资料，必须保留来源编号、标题和访问/检索日期。
- 对过期、地区不明或适用性不明的依据，应注明限制。

---

## 11. 异常处理

| 异常情况 | 处理方式 | 输出要求 |
| --- | --- | --- |
| 文档不可读 | 停止对不可读部分作结论 | 说明不可读页码/范围和影响 |
| 事实缺失 | 标记为“信息不足” | 列出待补充事实和可能影响 |
| 法律辖区不明 | 不默认适用某地区法律 | 请求补充；如继续分析，声明假设 |
| 附件缺失 | 不推定附件内容 | 标出涉及附件的条款和待补附件 |
| 版本冲突 | 不直接合并结论 | 要求确认审查版本或列出冲突点 |
| 用户要求规避监管 | 拒绝违法规避方案 | 转为合规整改建议 |
| 结论不确定 | 不强行下定论 | 说明不确定原因和复核路径 |
| 高风险事项 | 升级处理 | 要求律师或法务负责人复核 |
| 缺少法律依据 | 不编造依据 | 标注“依据不足，需复核” |

异常处理示例：

```markdown
### 异常与待补充信息
- 附件 2《服务范围说明》未提供，因此无法判断第 3 条交付义务是否完整；该事项标记为“信息不足”。
- 文档未说明适用地区。以下分析暂按中国大陆一般商业合同场景作有限审查，不构成特定司法辖区法律意见。
```

---

## 12. 交付契约

Skill 的标准输出必须包含以下部分，建议保持固定顺序：

1. **非律师免责声明**：明确本结果仅供辅助参考，不构成法律意见。
2. **审查范围与假设**：说明审查对象、版本、地区、用户提供的信息、未提供的信息和关键假设。
3. **总体结论**：一句话说明建议继续推进、修改后推进、暂停或升级法务。
4. **风险概览**：列出高/中/低/信息不足的数量。

## 14. 质量检查清单

生成或审查法务合规 Skill 时，应逐项检查：

### 14.1 元数据与触发

- [ ] 是否包含可解析的 YAML front matter 示例。
- [ ] `description` 是否准确说明能力和边界，且不会被误解为正式法律意见。
- [ ] `triggers` 是否覆盖合同、营销、隐私、劳动、制度等典型意图。
- [ ] 输入字段是否区分 required 与 optional。
- [ ] 是否声明高风险需人工复核、违法规避需拒绝。

### 14.2 适用范围与边界

- [ ] 是否清楚列出适用场景。
- [ ] 是否清楚列出不适用或需升级场景。
- [ ] 是否明确非律师免责声明。
- [ ] 是否禁止保证合法、替代律师、编造依据、规避监管。
- [ ] 是否在司法辖区不明时要求说明假设。

### 14.3 风险识别与分级

- [ ] 是否覆盖主体、付款、交付、违约、终止、知识产权、保密、数据、合规、争议解决等维度。
- [ ] 是否提供高/中/低/信息不足的分级定义。
- [ ] 是否要求每项风险说明判断理由。
- [ ] 是否将“信息不足”单独列示，而不是降为低风险。
- [ ] 是否要求高风险暂停或升级复核。

### 14.4 引用与法律依据

- [ ] 是否规定引用依据优先级。
- [ ] 是否禁止编造法律条文、案例或监管口径。
- [ ] 是否要求无法核验时标注限制。
- [ ] 是否要求联网或知识库资料保留来源信息。
- [ ] 是否提示法律法规需以最新有效文本和律师复核为准。

### 14.5 异常处理

- [ ] 是否覆盖文档不可读、事实缺失、辖区不明、附件缺失、版本冲突等异常。
- [ ] 是否对用户违法规避请求给出拒绝和合规替代路径。
- [ ] 是否要求不确定事项说明原因和复核路径。
- [ ] 是否避免对不可读或未提供内容作确定结论。

### 14.6 交付契约

- [ ] 是否包含非律师免责声明、审查范围、总体结论、风险概览、风险清单、逐条建议、引用依据、异常信息、下一步动作。
- [ ] 是否提供可复用输出模板。
- [ ] 是否要求逐条建议包含原条款、风险说明、修改方向、参考表述和待确认事项。
- [ ] 是否能直接转交法务、合规负责人或业务负责人。

### 14.7 输出风格

- [ ] 是否使用清晰、克制、可执行的语言。
- [ ] 是否避免恐吓式、绝对化、过度承诺表达。
- [ ] 是否区分事实、假设、判断和建议。
- [ ] 是否避免把一般信息包装成正式法律意见。

## 15. Skill 设计注意事项

- 本示例是 Skill 设计模板，不是法律服务产品说明，也不是正式法律意见模板。
- 对高风险事项要主动升级，而不是仅给文字润色。
- 对用户明显违法或规避监管的需求，应拒绝并给出合规替代路径。
- 对引用依据和事实假设保持透明，不要用“根据相关法律规定”掩盖依据不足。
- 对缺失信息保持谨慎，优先标注“信息不足”，不要为了完整性编造事实。
- 对组织内部制度、审批权限、合同模板偏离等事项，应优先使用用户提供的内部依据。
- 若 Skill 被用于多地区、多行业场景，应要求用户明确适用地区和行业监管要求。
