# PPT 生成类 Skill 示例

> 本示例用于指导 Skill 作者创建「PPT 生成类」技能包。目标是让技能在接到用户的演示文稿任务后，能够稳定完成需求澄清、内容规划、版式设计、素材处理、视觉 QA 与最终交付。

## 1. YAML front matter 示例

以下示例展示一个 PPT 生成类 Skill 的注册元数据。实际使用时，应根据技能名称、适用范围和入口文件进行调整。

```yaml
---
name: ppt-generator-pro
description: "根据用户主题、资料和目标受众，生成结构完整、视觉统一、可交付的 PowerPoint 演示文稿"
version: "1.0.0"
category: presentation
tags:
  - ppt
  - presentation
  - slide-generation
  - visual-qa
  - business-report
entry: skill.md
author: office-raccoon
---
```

编写注意事项：

- `description` 建议加引号，避免冒号、破折号等字符导致 YAML 解析失败。
- `name` 使用小写英文与短横线，避免空格、中文和特殊符号。
- `tags` 应覆盖主要触发词，例如 `ppt`、`presentation`、`slide-generation`。
- 每次发布前必须验证 YAML front matter 可解析，而不是只检查是否存在。

## 2. 适用场景

该类 Skill 适用于以下任务：

1. 根据用户提供的主题、文档、表格、会议纪要或调研资料，生成完整 PPT。
2. 将长文档、报告、方案、商业计划书转化为演示文稿。
3. 为路演、汇报、培训、复盘、产品介绍、销售提案生成结构化幻灯片。
4. 在已有 PPT 基础上进行内容重构、视觉统一、版式优化和补页。
5. 根据品牌规范或用户指定风格，输出可直接演示或继续编辑的 `.pptx` 文件。

不适合直接使用的场景：

- 用户只要求提取文字、翻译、摘要，且不需要演示文稿交付。
- 用户需要复杂三维动画、交互式演示或视频级动效。
- 用户没有提供任何主题、受众、目标或基础信息，且无法合理默认。

## 3. 触发描述

当用户出现以下意图时，可以触发 PPT 生成类 Skill：

- “帮我做一份 PPT”
- “根据这份文档生成演示稿”
- “把这份方案整理成路演 PPT”
- “做一份汇报材料 / 培训课件 / 产品介绍 / 商业计划书”
- “优化这个 PPT 的结构和版式”
- “把 Excel 数据做成管理层汇报 PPT”

触发后应优先识别以下关键信息：

| 信息项 | 说明 |
| --- | --- |
| 主题 | PPT 要讲什么 |
| 目标 | 说服、汇报、培训、复盘、销售、立项等 |
| 受众 | 管理层、客户、员工、投资人、合作伙伴等 |
| 页数 | 用户指定页数或根据内容合理规划 |
| 风格 | 商务、科技、简洁、高端、活泼、品牌化等 |
| 输入资料 | 文档、表格、图片、旧 PPT、网页资料等 |
| 输出格式 | 通常为 `.pptx`，必要时可附 PDF、图片预览或素材包 |

## 4. 内容规划

生成 PPT 前，应先建立清晰的信息架构，避免直接堆素材。

### 4.1 推荐流程

1. 明确演示目标：本 PPT 最终要让观众理解、相信或采取什么行动。
2. 提炼核心观点：用 1 句话定义整份 PPT 的主结论。
3. 设计故事线：按照问题、洞察、方案、价值、行动的逻辑组织内容。
4. 拆分页结构：将内容拆成封面、目录、章节页、正文页、总结页、附录页。
5. 分配页面任务：每页只承担一个主要观点，避免一页放多个互相竞争的信息。
6. 生成页面脚本：为每页写清标题、核心结论、正文要点、图表或视觉元素。

### 4.2 常见结构模板

#### 管理层汇报

1. 封面
2. 核心结论摘要
3. 关键指标概览
4. 主要进展
5. 问题与风险
6. 原因分析
7. 后续行动计划
8. 资源需求 / 决策事项
9. 总结页

#### 商业方案 / 销售提案

1. 封面
2. 客户背景与痛点
3. 市场趋势或机会
4. 解决方案
5. 产品 / 服务能力
6. 实施路径
7. 预期价值
8. 案例或证明
9. 合作模式与下一步

#### 培训课件

1. 封面
2. 学习目标
3. 课程地图
4. 背景知识
5. 核心概念
6. 方法步骤
7. 案例演练
8. 测验 / 讨论
9. 总结与行动建议

## 5. 版式 / 素材

PPT 生成类 Skill 应同时关注内容可读性和视觉一致性。

### 5.1 版式原则

- 页面尺寸优先使用 16:9，除非用户明确要求 4:3 或其他比例。
- 每页必须有清晰的信息层级：标题、结论、正文、图表、注释。
- 同一级标题、正文、页码、图例在全稿中保持一致。
- 避免大段文字堆积，正文尽量转化为短句、表格、流程图、矩阵图或数据图。
- 保持足够留白，避免内容贴边或遮挡。
- 重要页面应突出一个视觉焦点，不要平均用力。

### 5.2 素材策略

- 用户提供的品牌 Logo、产品图、截图、照片应优先使用。
- 外部图片、图标、插画需要注意版权与来源；无法确认授权时，应使用可替代的抽象图形或自行生成的非侵权素材。
- 图表数据必须来自用户资料或明确标注的来源，不能编造关键业务数据。
- 截图类素材需要保证清晰度，避免拉伸变形。
- 生成型图片不应包含复杂文字，文字信息应在 PPT 中用可编辑文本呈现。

### 5.3 视觉风格示例

| 风格 | 适用场景 | 视觉特征 |
| --- | --- | --- |
| 商务简洁 | 管理层汇报、经营复盘 | 深色标题、低饱和配色、表格与图表清晰 |
| 科技蓝 | 产品发布、技术方案 | 蓝紫渐变、线框、光效、数据可视化 |
| 高端黑金 | 路演、品牌发布 | 深色背景、金色点缀、大图留白 |
| 轻量活泼 | 培训、内部宣导 | 明亮色块、插画、卡片式布局 |

## 6. 视觉 QA

PPT 交付前必须进行视觉 QA，不能只检查文本。

### 6.1 必查项目

- 是否存在文字溢出、遮挡、重叠、被裁切。
- 是否存在图片变形、低清晰度、透明背景异常。
- 页眉、页脚、页码、章节标识是否一致。
- 标题层级是否统一，字号是否可读。
- 图表坐标、图例、数据标签是否完整。
- 配色是否统一，重点色是否滥用。
- 页面是否有明显空洞或过度拥挤。
- 中英文、数字、单位、标点是否风格一致。

### 6.2 QA 方法

1. 导出每页截图或预览图。
2. 对封面、目录、章节页、正文页、图表页、总结页分别抽查。
3. 对所有页面进行缩略图级别检查，确认整体节奏和风格统一。
4. 对关键页面进行放大检查，确认文字、图表和素材无错误。
5. 如发现问题，应回到源 PPT 修正后重新导出检查。

## 7. 异常处理

### 7.1 输入不足

当用户只说“做个 PPT”，但缺少主题、目标或受众时：

- 如果上下文已有足够信息，可基于上下文合理默认，并在交付说明中标明假设。
- 如果完全无法判断主题，应先要求用户补充，而不是编造内容。

### 7.2 文件无法读取

当用户提供的资料无法打开、格式损坏或内容为空时：

- 明确说明无法读取的文件名和原因。
- 尝试可行的替代路径，例如读取文本层、截图预览、转换格式。
- 如果仍失败，应请求用户重新上传或提供可读版本。

### 7.3 数据冲突

当多个资料中的口径不一致时：

- 优先使用用户最新提供或明确指定的资料。
- 在备注中列出冲突点，不擅自合并矛盾数据。
- 对关键数字保持原始来源和单位一致。

### 7.4 版权与敏感内容

- 不使用来源不明、可能侵权的商用图片作为核心素材。
- 不生成政治、色情、暴力、歧视、隐私泄露等不合规内容。
- 涉及客户、员工、财务、合同等敏感信息时，避免在不必要页面中扩散。

### 7.5 工具或渲染失败

- 如 PPT 生成、导出或截图失败，应定位失败环节并重试一次。
- 连续失败时应降级交付，例如先提供大纲、逐页文案、HTML 预览或可复制的页面脚本。
- 不应伪造已经生成的 PPT 文件。

## 8. 交付契约

PPT 生成类 Skill 的最终交付应满足以下契约：

1. 所有交付物写入工作区 `output/` 目录下的任务专属子目录。
2. 主交付物通常为 `.pptx` 文件，命名使用小写英文、短横线，不含空格、中文、时间戳或 UUID。
3. 如同时交付多个文件，应打包为 `.zip`，并剔除日志、缓存、临时截图、运行态文件等无关内容。
4. 对外引用文件时，使用 `local-file://` 加绝对路径的格式。
5. 汇报时说明：文件路径、页数、核心结构、已完成 QA 项、已知风险或限制。
6. 如果未能生成最终 PPT，必须明确说明未完成原因、已完成部分和可选下一步。

示例交付说明：

```markdown
已完成 PPT 生成，共 12 页，包含封面、目录、3 个章节、总结页和附录页。

文件：
[PPT 文件](local-file:///absolute/path/to/output/business-report.pptx)

已完成检查：
- 页面结构完整
- 标题与正文层级统一
- 图表无明显遮挡
- 图片未发现变形

已知风险：
- 第 7 页市场规模数据来自用户原始资料，未额外联网核验
```

## 9. 质量检查清单

发布或使用该 Skill 前，应逐项确认：

### 9.1 Skill 元数据

- [ ] 存在 YAML front matter。
- [ ] YAML front matter 可被解析。
- [ ] `name`、`description`、`version`、`tags`、`entry` 等字段完整。
- [ ] `description` 没有未转义的冒号等 YAML 风险字符。

### 9.2 内容能力

- [ ] 能识别用户的主题、目标、受众和页数要求。
- [ ] 能根据不同场景选择合适的 PPT 结构。
- [ ] 每页能形成明确标题、核心观点、正文要点和视觉建议。
- [ ] 不编造关键事实、数据、案例和来源。

### 9.3 版式能力

- [ ] 页面比例、字体、字号、颜色、页码规则一致。
- [ ] 图表、表格、流程图、矩阵图等元素布局清晰。
- [ ] 图片不变形、不遮挡、不低清。
- [ ] 重点信息有明确视觉强调。

### 9.4 QA 能力

- [ ] 能导出或查看页面预览。
- [ ] 能检查文字溢出、重叠、裁切和遮挡。
- [ ] 能检查全稿风格一致性。
- [ ] 能发现并修复典型视觉问题。

### 9.5 交付能力

- [ ] 交付文件位于 `output/` 目录。
- [ ] 文件命名符合小写英文和短横线规则。
- [ ] 多文件交付已打包且包内无临时文件。
- [ ] 汇报中明确说明文件路径、核心内容、QA 结果和风险。

## 10. 最小可用 Skill 指令骨架

```markdown
# PPT 生成类 Skill

当用户要求生成或优化 PPT 时，按以下步骤执行：

1. 识别主题、目标、受众、页数、风格、输入资料和输出格式。
2. 如果信息不足，基于上下文合理默认；无法默认时再请求补充。
3. 先规划整份 PPT 的故事线和页面结构。
4. 为每页生成标题、核心结论、正文要点、图表或素材建议。
5. 根据统一视觉规范生成 PPT 文件。
6. 导出预览并执行视觉 QA。
7. 修复发现的问题。
8. 将最终文件写入 `output/`，并按交付契约汇报。
```

该骨架可以作为 Skill 的起点，但正式技能应补充具体的生成流程、工具调用约束、模板规则、视觉 QA 标准和异常处理策略。
