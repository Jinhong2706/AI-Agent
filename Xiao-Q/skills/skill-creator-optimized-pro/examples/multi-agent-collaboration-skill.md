# 多 Agent 协作类 Skill 示例

本示例用于指导 Skill 作者设计“主控 Agent + 多个子 Agent”的协作型 Skill。它不是鼓励无差别并行，而是帮助主控 Agent 在复杂任务中，把工作拆成可隔离、可复核、可合并、可追责的单元，最终形成稳定交付。

适用任务包括：多文件审阅、批量内容生成、并行数据核查、复杂文档局部起草、跨维度质量检查、分片 QA、分区域/分业务线分析等。

> 使用本示例时，建议将其改写为具体 Skill 的 `SKILL.md` 内容，并保留“主控负责最终合并与验收、子 Agent 不覆盖最终交付物”的核心原则。

---

## 0. YAML front matter 示例

协作型 Skill 的 `SKILL.md` 必须包含可解析的 YAML front matter。下面是一个最小但完整的示例：

```yaml
---
name: multi-agent-collaboration-skill
description: "当任务包含多个可独立处理的文件、页面、章节、数据切片或检查维度时，使用多 Agent 协作完成拆分、并行处理、合并复核与最终交付。"
version: "1.0.0"
category: workflow
tags:
  - multi-agent
  - collaboration
  - parallel-review
  - batch-processing
triggers:
  - "需要审阅 5 个以上结构相似文件"
  - "需要按章节、页面、区域、业务线或数据切片并行处理"
  - "需要多个独立维度同时检查，例如事实、格式、引用、合规、口径"
  - "需要先生成多个局部草稿，再由主控统一合并"
inputs:
  - name: user_goal
    required: true
    description: "用户的最终目标、交付物类型、输出路径和验收标准。"
  - name: source_materials
    required: true
    description: "原始文件、目录、页面范围、数据字段或其他可分片输入。"
outputs:
  - name: final_deliverable
    description: "由主控 Agent 合并、复核后生成的最终交付物。"
  - name: drafts
    description: "各子 Agent 独立生成的中间产物、检查报告或局部草稿。"
safety:
  no_shared_write: true
  child_agents_must_not_overwrite_final: true
  require_master_validation: true
---
```

YAML 编写注意事项：

- `description` 中如果出现英文冒号加空格、列表符号、引号等复杂字符，建议整体加双引号。
- `triggers` 应写成用户真实会说的话，而不是抽象内部术语。
- `safety` 字段可根据运行框架裁剪，但原则必须保留：不共享写入、不让子 Agent 覆盖最终稿、主控必须验收。
- front matter 必须能被 YAML 解析器正常解析，不要只看 Markdown 渲染效果。

## 1. 北极星目标

多 Agent 协作的核心不是“多派几个 Agent”，而是让复杂任务在可隔离、可验收、可合并的边界内更稳、更快地完成。

一个合格的协作型 Skill 应确保：

1. 主控 Agent 始终掌握用户目标、交付契约、风险边界与最终验收标准。
2. 子 Agent 只处理可隔离的局部任务，不直接覆盖最终交付物。
3. 并行任务之间没有共享写入冲突。
4. 所有子任务产物都能被主控 Agent 复核、合并、追责与回滚。
5. 异常、缺口和风险能够被明确上报，而不是被静默吞掉。
6. 最终交付不是子任务草稿的简单拼接，而是经过统一口径、去重、冲突处理和质量检查后的成品。

---

## 2. 适用与不适用场景

### 2.1 适用场景

优先在以下任务中启用多 Agent 协作：

| 场景 | 拆分方式 | 示例 |
| --- | --- | --- |
| 多文件结构相似 | 按文件拆分 | 10 份合同逐份提取风险点 |
| 多章节独立起草 | 按章节拆分 | 白皮书每章先独立草拟 |
| 多页面检查 | 按页码或页组拆分 | PPT 每 5 页为一组做视觉和文案 QA |
| 多维度检查 | 按检查维度拆分 | 同时检查事实、格式、引用、合规、口径 |
| 多数据切片分析 | 按地区、业务线、时间段拆分 | 分区域分析销售波动原因 |
| 多候选方案生成 | 按方案方向拆分 | 分别产出保守版、增长版、品牌版文案 |

### 2.2 不适用场景

以下任务不建议直接拆成多个子 Agent；如有必要，只拆出辅助检查任务：

- 用户目标、交付路径或验收标准尚不清楚，需要先澄清。
- 强依赖全局上下文，局部处理容易造成口径漂移。
- 多个子任务必须频繁修改同一个文件、同一段落、同一张表或同一对象。
- 存在高风险操作，例如删除、覆盖、发布、外发、批量改权限。
- 子任务之间存在严格先后依赖，无法独立完成。
- 原始材料很少、任务很短，拆分成本高于收益。
- 子任务结果难以复核来源，无法建立责任边界。

## 3. 触发描述设计

Skill 的触发描述应让主控 Agent 能判断“什么时候必须拆、什么时候可以拆、什么时候不拆”。

### 3.1 强触发

满足以下任一条件时，应优先考虑多 Agent 协作：

- 存在 5 个以上结构相似且可独立处理的单元，例如文件、页面、章节、表格、区域、字段组。
- 用户明确要求“分别检查”“逐份分析”“批量生成”“并行处理”“多维度审核”。
- 任务需要在多个独立维度上同时保证质量，例如事实准确、格式统一、引用可追溯、合规风险、表达风格。
- 单个最终交付物由多个局部模块组成，且每个模块可独立起草。

### 3.2 弱触发

满足以下条件时，可以只派发少量子 Agent 做辅助：

- 任务主要由主控完成，但需要独立 QA 或事实核查。
- 只有 2-4 个可拆单元，但每个单元内容较长或风险较高。
- 需要多个候选方案，最后由主控选择合并。

### 3.3 反触发

出现以下情况时，不应启用多 Agent 并行处理：

- 需要先向用户确认关键信息。
- 输出必须实时连续编辑同一个文件。
- 子任务会互相依赖对方未完成的结果。
- 任何子 Agent 的误写都可能破坏用户原文件或最终交付物。

---

## 4. 主控与子 Agent 职责边界

### 4.1 主控 Agent 职责

主控 Agent 必须负责：

1. 理解用户目标、交付物、路径、格式、限制条件和验收标准。
2. 判断是否适合多 Agent 协作。
3. 制定拆分方案、编号规则、输入范围、输出路径和禁止事项。
4. 派发子 Agent，并确保每个子任务边界清楚、写入位置独立。
5. 收集子任务产物，检查遗漏、重复、越界、冲突和证据质量。
6. 统一术语、口径、格式、语气、引用和交付结构。
7. 生成最终交付物并执行最终验收。
8. 向用户汇报完成情况、产物路径、关键结论和残余风险。

### 4.2 子 Agent 职责

子 Agent 只负责：

- 处理被明确分配的文件、页面、章节、数据字段或检查维度。
- 按指定格式输出局部结果。
- 明确说明证据来源、未处理范围、异常和不确定性。
- 将产物写入指定的独立草稿文件，或只返回结构化结果。

子 Agent 不得：

- 修改用户原文件，除非任务明确授权且该子 Agent 是唯一写入者。
- 覆盖最终交付物。
- 修改其他子 Agent 的草稿。
- 自行扩大范围处理未分配内容。
- 为了补齐结果而编造数据、引用、事实或用户意图。

## 5. 拆分策略

### 5.1 拆分优先级

建议按以下优先级拆分：

1. **按天然边界拆分**：文件、页码、章节、表格、sheet、业务线、地区、时间段。
2. **按质量维度拆分**：事实核查、格式检查、引用检查、合规检查、风格检查。
3. **按交付模块拆分**：摘要、正文、附录、图表说明、风险清单。
4. **按方案方向拆分**：候选标题、候选文案、候选结构、候选策略。

避免按模糊边界拆分，例如“你负责前半部分”“你负责优化一下”“你看哪里有问题”。

### 5.2 拆分粒度

拆分粒度应满足：

- 子任务足够小，可以独立完成和验收。
- 子任务足够完整，不需要频繁询问其他子 Agent。
- 子任务之间没有共享写入。
- 主控合并成本可控，不会产生过多碎片。

常见粒度建议：

| 任务类型 | 推荐粒度 |
| --- | --- |
| 合同/报告审阅 | 每个文件 1 个子任务，或每 3-5 个短文件 1 个子任务 |
| PPT 检查 | 每 3-5 页 1 个子任务；另设格式/事实总检子任务 |
| 长文起草 | 每章或每个一级标题 1 个子任务 |
| 数据分析 | 每个业务切片、地区、时间段或主题 1 个子任务 |
| 质量审查 | 每个维度 1 个子任务 |

### 5.3 派发前模板

主控 Agent 在派发子任务前，应写清楚：

```text
总目标：生成一份可交付的客户调研报告。
当前子任务编号：unit-03。
子任务范围：只分析问卷数据中的“价格敏感度”部分。
输入材料：survey.xlsx 的 price_* 字段，以及 README.md 中的字段说明。
输出位置：drafts/unit-03-price-analysis.md。
禁止事项：不得修改最终报告；不得推断未出现的数据；不得处理其他字段。
输出格式：包含范围、结论、证据、异常、风险、建议合并方式。
验收标准：每条结论可追溯到字段或统计结果；异常值单独说明。
```

## 6. 并行安全

### 6.1 可并行条件

满足以下条件时，子 Agent 可以并行：

- 输入可分片：每个子 Agent 有明确文件、页码、章节、字段或维度范围。
- 输出独立：每个子 Agent 写入独立草稿，或只返回结构化结果。
- 规则一致：主控 Agent 已提供统一口径、格式和验收标准。
- 失败可隔离：某个子任务失败不会破坏其他子任务产物。
- 合并可追溯：主控可以判断每个结论来自哪个子任务或原始材料。

### 6.2 禁止并行条件

出现以下情况时，不得并行写入：

- 多个子 Agent 修改同一个源文件。
- 多个子 Agent 同时维护同一份索引、目录、摘要、最终稿或状态文件。
- 子 Agent 需要相互读取对方尚未完成的结果。
- 子任务产物无法区分责任来源。
- 写入顺序会影响结果正确性。

### 6.3 并行安全检查

派发前，主控 Agent 应确认：

```text
[ ] 每个子任务有唯一编号。
[ ] 每个子任务有明确输入范围。
[ ] 每个子任务有唯一输出路径。
[ ] 没有两个子任务写同一个文件。
[ ] 子 Agent 不会覆盖用户原文件或最终交付物。
[ ] 子任务不依赖其他未完成子任务。
[ ] 主控保留最终合并和验收责任。
```

## 7. 文件写入隔离

### 7.1 推荐目录结构

```text
output/project-name/
├── drafts/
│   ├── unit-01-contract-a.md
│   ├── unit-02-contract-b.md
│   ├── check-facts.md
│   └── check-format.md
├── final-report.md
└── validation-report.md
```

### 7.2 命名规则

使用稳定、可追踪、不会冲突的命名：

```text
drafts/unit-01-contract-a.md
drafts/unit-02-contract-b.md
drafts/unit-03-chapter-risk.md
drafts/check-format.md
drafts/check-facts.md
```

避免使用：

```text
draft.md
final.md
result-new.md
temp.md
版本2.md
```

### 7.3 写入原则

- 子 Agent 只能写入自己的草稿文件或指定唯一目录。
- 主控 Agent 才能写最终交付物。
- 如需多个文件交付，由主控 Agent 统一打包或统一引用。
- 中间产物、检查报告、最终产物必须区分清楚。
- 不要让子 Agent 修改共享索引、总目录、最终摘要或最终报告。

## 9. 汇总、合并与验证

### 9.1 合并责任

最终合并只能由主控 Agent 完成。子 Agent 不应直接生成、覆盖或发布最终交付物，除非用户任务明确要求该子 Agent 只负责一个独立最终文件，且不会与其他子任务冲突。

### 9.2 汇总步骤

主控 Agent 应按以下顺序汇总：

1. 收集所有子任务产物，核对编号、范围和输出路径。
2. 检查是否存在遗漏、重复、越界或空结果。
3. 检查关键结论是否有证据来源。
4. 对同一主题的结论进行去重、排序和合并。
5. 识别事实冲突、口径冲突、格式冲突和建议冲突。
6. 统一术语、时间范围、统计口径、标题层级、引用格式和语气。
7. 生成最终交付物。
8. 对最终交付物执行质量检查，而不是只拼接草稿。

### 9.3 常见冲突类型

- 事实冲突：两个子 Agent 对同一数据给出不同结论。
- 口径冲突：同一术语、时间范围、样本范围或统计口径不一致。
- 格式冲突：标题层级、表格字段、引用格式不一致。
- 优先级冲突：一个建议保留，另一个建议删除。
- 责任边界冲突：子 Agent 越界处理了非分配范围。
- 证据强度冲突：一个结论有来源，另一个只是推断。

### 9.4 冲突处理策略

主控 Agent 应按以下顺序处理：

1. 回到用户目标和交付契约，判断哪个结果更贴近目标。
2. 优先采用证据更完整、来源更可靠、可复核性更强的结果。
3. 对无法判断的冲突做显式标注，而不是强行合并。
4. 必要时追加一个“冲突复核”子任务，但不得让原冲突双方互相覆盖。
5. 在最终交付中说明关键取舍和残余风险。

### 9.5 最终验证

最终交付前，主控 Agent 应验证：

- 是否覆盖用户要求的全部范围。
- 是否遗漏任何子任务产物。
- 是否把中间草稿误当成最终交付物。
- 是否存在未处理的冲突或互相矛盾的结论。
- 是否符合用户指定格式、路径、命名和引用方式。
- 是否能追溯到原始材料或子任务输出。

## 10. 异常处理

### 10.1 子任务失败

当子 Agent 失败时，主控 Agent 应记录：

- 失败子任务编号。
- 失败发生阶段。
- 已完成内容。
- 失败原因。
- 对最终交付的影响。
- 可选补救方案。

处理建议：

1. 如果失败不影响关键目标，可降级交付，并在最终说明中列出限制。
2. 如果失败影响关键目标，应停止合并或请求用户补充输入。
3. 如果是工具或权限问题，不要无限重试同一路径，应切换可行方案或说明阻塞。
4. 如果是子任务边界不清，主控应重新拆分并只补派必要子任务。

### 10.2 输入缺失

如果某个子任务缺少必要输入：

1. 不得编造内容。
2. 标注缺失字段、文件、页码或上下文。
3. 判断是否影响总目标。
4. 若影响关键交付，应停止合并并向用户说明。
5. 若不影响关键交付，可降级交付并写明限制。

### 10.3 工具或权限异常

如果工具不可用、文件无法访问或权限不足：

- 不要反复重试同一路径。
- 记录已尝试的方法。
- 切换可行替代路径。
- 若替代路径仍不可行，给出降级交付方案。

### 10.4 子 Agent 越界

如果子 Agent 处理了未分配范围：

- 不直接采用越界内容。
- 标记越界部分并由主控判断是否有参考价值。
- 若越界内容影响其他子任务，应优先采用原责任子任务结果。
- 必要时补派复核子任务。

### 10.5 结果质量不足

如果子 Agent 输出缺少证据、结论空泛或格式不符：

- 要求补充证据或重新按契约输出。
- 若无法补充，主控只能将其作为低置信参考。
- 不得把无证据结论写入最终交付的关键结论部分。

## 11. 交付契约

协作型 Skill 必须明确最终交付契约：

- **输出位置**：所有交付物必须写入约定目录。
- **命名规则**：文件名稳定、描述性、可覆盖，避免时间戳、随机数和模糊后缀。
- **引用方式**：如果需要在回复中引用文件，应使用当前系统约定的本地文件链接格式。
- **中间产物**：草稿、检查报告、合并依据应与最终交付物区分。
- **不覆盖原则**：子 Agent 不覆盖用户原文件和最终交付物。
- **可复核性**：最终结果应能追溯到子任务输出或原始材料。
- **多文件交付**：如果最终包含多个文件，由主控统一打包或统一列明，不让子 Agent 分散交付。
- **风险披露**：关键缺失、降级处理、未解决冲突和低置信结论必须在最终汇报中说明。

---

## 12. 最小实现流程

```text
1. 主控 Agent 解析用户目标、输入材料、输出路径、格式要求和验收标准。
2. 判断是否满足多 Agent 拆分条件；若不满足，单线完成或只做辅助 QA。
3. 设计拆分方案：编号、范围、输入、输出、禁止事项、验收标准。
4. 检查并行安全：确认无共享写入、无未完成依赖、无最终稿覆盖风险。
5. 并行派发子 Agent，并为每个子 Agent 指定独立输出。
6. 收集子任务产物，检查缺失、重复、越界、低质和异常。
7. 主控 Agent 进行冲突检测、事实复核、口径统一和格式统一。
8. 主控 Agent 生成最终交付物。
9. 主控 Agent 完成质量检查和交付契约检查。
10. 主控 Agent 向用户汇报路径、结论、完成度和风险。
```

## 15. 质量检查清单

### 15.1 Skill 设计检查

- [ ] YAML front matter 存在且可被 YAML 解析器解析。
- [ ] `description` 清楚说明何时使用多 Agent 协作。
- [ ] `triggers` 覆盖批量、多文件、多章节、多维度、分片检查等真实触发表达。
- [ ] 明确写出适用场景和不适用场景。
- [ ] 明确主控 Agent 与子 Agent 的职责边界。
- [ ] 明确子 Agent 不得覆盖最终交付物。

### 15.2 拆分与并行检查

- [ ] 子任务按文件、页面、章节、字段、维度或模块等明确边界拆分。
- [ ] 每个子任务都有唯一编号。
- [ ] 每个子任务都有明确输入范围。
- [ ] 每个子任务都有唯一输出路径。
- [ ] 不存在两个子 Agent 写入同一文件。
- [ ] 不存在子 Agent 依赖其他未完成子 Agent 结果的情况。
- [ ] 失败子任务可以隔离处理，不会破坏其他结果。

### 15.3 文件与交付检查

- [ ] 草稿、中间检查报告和最终交付物目录清晰区分。
- [ ] 子 Agent 只写自己的草稿文件或指定唯一目录。
- [ ] 主控 Agent 负责最终合并、打包、引用和交付。
- [ ] 文件命名稳定、描述性、可覆盖，无随机后缀。
- [ ] 最终交付物路径符合用户或系统约定。

### 15.4 内容质量检查

- [ ] 所有关键结论有证据来源。
- [ ] 子任务遗漏、重复、越界已检查。
- [ ] 事实冲突、口径冲突、格式冲突已处理或显式说明。
- [ ] 术语、时间范围、统计口径、标题层级和引用格式已统一。
- [ ] 未把低质量、无证据或越界结果直接写入最终结论。
- [ ] 异常、缺失、降级交付和残余风险已披露。

### 15.5 最终汇报检查

- [ ] 汇报说明任务是否完成。
- [ ] 汇报最终交付物路径。
- [ ] 汇报关键处理结果和质量检查结果。
- [ ] 汇报未完成项、降级项或残余风险。
- [ ] 如有进一步优化建议，明确列出但不偏离用户原目标。
