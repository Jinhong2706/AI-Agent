# 多 Agent 协作 Skill

## 适用场景

当用户任务包含多个结构相似、可独立处理、可独立验收的单元时，使用本 Skill 设计主控 Agent 与子 Agent 的协作流程。典型场景包括：多文件审阅、批量章节起草、多维度 QA、分区域数据分析、并行资料核验。

## 核心原则

1. 主控 Agent 负责目标理解、任务拆分、派发、合并、验收和最终交付。
2. 子 Agent 只负责明确边界内的局部任务。
3. 子 Agent 不得覆盖最终交付物，不得修改其他子 Agent 的产物。
4. 每个子任务必须有独立输入范围、独立输出路径和明确验收标准。
5. 所有冲突、异常、缺失和风险必须回到主控 Agent 统一处理。

## 工作流程

### 1. 确认总目标与交付契约

在拆分前先确认：

- 用户最终要什么。
- 输出文件应放在哪里。
- 文件命名和格式要求是什么。
- 是否允许覆盖既有文件。
- 哪些操作需要用户确认。

### 2. 任务拆分

按文件、章节、页码、数据切片或检查维度拆分任务。每个子任务说明：

- 子任务编号。
- 处理范围。
- 输入材料。
- 输出路径。
- 禁止事项。
- 验收标准。

### 3. 并行边界控制

只有在以下条件同时成立时才并行：

- 子任务之间没有写入同一文件的需求。
- 子任务之间没有未完成结果依赖。
- 每个子 Agent 的责任边界清楚。
- 主控 Agent 能独立验证每个子任务结果。

禁止多个子 Agent 同时写入最终文件、索引文件、目录文件或同一草稿文件。

### 4. 子 Agent 输出格式

每个子 Agent 应返回或写入以下结构：

```markdown
# 子任务结果

## 范围
- 已处理：
- 未处理：

## 主要结论
- 

## 证据来源
- 

## 风险与不确定性
- 

## 建议合并方式
- 
```

### 5. 冲突合并

主控 Agent 合并时应检查：

- 事实是否冲突。
- 口径是否一致。
- 格式是否统一。
- 是否存在越界处理。
- 是否有缺失输入被误写成结论。

合并优先级：用户目标 > 明确证据 > 权威来源 > 一致口径 > 文风统一。无法判断时保留风险说明，不强行合并。

### 6. 主控验收

最终交付前，主控 Agent 必须完成验收：

- 覆盖范围完整。
- 产物路径正确。
- 没有非授权文件被修改。
- 关键结论可追溯。
- 异常和风险已说明。
- 最终文件可直接使用。

### 7. 异常处理

遇到失败、权限不足、输入缺失或工具不可用时：

1. 停止在同一路径上反复尝试。
2. 记录已尝试动作和失败原因。
3. 判断是否影响最终目标。
4. 给出继续、降级、换路径或停止的建议。
5. 需要用户决策时，明确说明决策点。

## 派发模板

```text
你是子 Agent，只负责以下局部任务，不要修改最终交付物。
任务编号：unit-01
处理范围：
输入材料：
输出路径：drafts/unit-01.md
禁止事项：不得处理范围外内容；不得覆盖其他文件。
验收标准：结论有证据；风险单独列出；输出格式符合约定。
```

## 汇报模板

```text
已完成多 Agent 协作：
- 已完成子任务：
- 未完成或降级子任务：
- 已生成最终交付物：
- 已处理的冲突：
- 残余风险：
```
