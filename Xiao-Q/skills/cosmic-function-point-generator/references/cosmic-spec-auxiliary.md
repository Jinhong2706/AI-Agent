## 1. 设计文档关键信息字段提取规则

### 1.1 章节定位锚点

智能体必须**严格按以下锚点**在设计文档中定位关键信息。多个锚点指向同一字段时取**首个非空匹配**。

| # | 待提取字段 | 定位锚点（按优先级） | 备注 | 优先级 |
|:-:|:-----------|:---------------------|:-----|:-------|
| 1 | **核心需求全称** | ① 文档标题<br>② 1.2 节「需求矩阵」表头为「一级需求」列的所有非空值<br>③ 2.3 节「功能实现」一级模块名 | 取与原文完全一致的全称，禁止简写 | 🔴 |
| 2 | **二级需求列表** | ① 1.2 节「需求矩阵」中「二级需求」列的所有非空去重值<br>② 2.3 节「功能实现」的二级标题 | 用于填写 C 列「功能用户需求」 | 🔴 |
| 3 | **三级需求（功能过程）列表** | ① 1.2 节「需求矩阵」中「三级需求」列的所有非空去重值<br>② 2.3 节下三级功能点标题 | 用于填写 E 列「功能过程」 | 🔴 |
| 4 | **业务动作原文** | 2.3 节「功能实现 → 功能描述」下的**所有正文段落** | 用于 F 列子过程拆分的唯一原文依据 | 🔴 |
| 5 | **交互系统/接口对端** | ① 2.3 节原文中出现的「XX 系统/服务/平台」类实体<br>② 2.5 节「外部接口设计 → 接口列表」中的「接口对端系统」列 | 用于 B 列「功能用户」 | 🔴 |
| 6 | **触发动作** | 2.3 节功能描述中描述「触发方式」的语句（如"点击…""调用…""定时…"等） | 用于 D 列「触发事件」 | 🔴 |
| 7 | **数据字段/入参/属性** | 2.3 节功能描述中**顿号、分号、逗号分隔的字段集合**或"包括…等字段" | 用于 I 列「数据属性」 | 🔴 |
| 8 | **核心数据对象** | 2.3 节功能描述中**"XX 表/XX 信息/XX 记录"** 类实体 | 用于 H 列「数据组」 | 🔴 |
| 9 | **开发类型** | 2.3 节中是否出现「新增/复用/利旧」字样（如未显式标注，则按 J 列配比规则智能分配） | 用于 J 列「复用度」 | 🟡 |

### 1.2 提取操作流程

```
1) 解析 .docx 段落与表格，构建 DOM 树
2) 抽取所有 H1~H6 标题，建立大纲索引
3) 定位 1.2 节 → 抽取需求矩阵三列（A=一级、B=二级、C=三级），去重
4) 定位 2.3 节 → 对每个二级需求，按三级功能点分组，收集其下所有正文段落
5) 对每个三级功能点正文段落，执行"分点扫描"（见 1.3）形成"动作集合"
6) 以"动作集合"为输入进入数据移动类型映射流程（见 cosmic-rules.md 3.1）
```

### 1.3 前置分点扫描规则

🔴 强约束。**目的**：确保所有数字分点、并列条件、独立动作都不被遗漏。

| 扫描对象 | 触发拆分的判定特征 | 处理方式 | 优先级 |
|:---------|:-------------------|:---------|:-------|
| 段落内的步骤描述 | 段首出现 `1.` `2.` `3.` `① ② ③` 等数字分点 | 每个分点视为**独立子过程候选** | 🔴 |
| 段落内的并列动作 | `顿号` `、` `；` `分号` `"且/或"` `"以及"` 连接 | 拆分为多个独立子过程候选 | 🔴 |
| 段落内的"包括…等字段" | 字段集合罗列 | 字段集合作为单条子过程的「数据属性」入参 | 🔴 |
| 段落内的"通过 X 查询 Y" | "通过…查询/获取…"句式 | 视为「R 读取」动作，独立成行 | 🔴 |
| 段落内的"保存…到…表" | "保存/写入/更新…到…表/库"句式 | 视为「W 写入」动作，独立成行 | 🔴 |
| 段落内的"反馈/返回/推送" | "反馈/返回/推送/调用…接口"句式 | 视为「X 输出」动作，独立成行 | 🔴 |
| 段落内的"解析/获取…入参" | "解析/获取…入参/接口"句式 | 视为「E 输入」动作，**强制作为首行** | 🔴 |
| 多步连续写操作 | "生成 + 保存"、"更新 + 记录日志" | **必须**拆分为多行，禁止合并 | 🔴 |

**扫描产物**：每个三级功能点产出 `N` 个子过程候选（N ≥ 原文分点总数），作为后续映射的输入。

## 2. 12 列字段详解

### 2.1 A 列：客户需求

🔴 强约束。
- 取值：整表统一填写核心需求全称
- 来源：概设标题、1.2 节需求矩阵一级需求
- 约束：不得简写/修改

### 2.2 B 列：功能用户

🔴 强约束。固定格式：`发起者：XX系统/服务；接收者：XX系统/服务`

| 业务场景 | 发起者 | 接收者 |
|:---------|:-------|:-------|
| 页面可视化类（Web/APP 端发起） | 业务服务接口 | 业务 WEB 服务 |
| 流程自动执行类 | 流程引擎服务 | 业务服务接口 |
| 跨系统接口类 | 调用方系统服务 | 被调用方系统服务 |
| 定时任务类 | 流程引擎服务 | 业务服务接口 |
| 外部系统反馈类 | 外部系统 | 业务服务接口 |

> 无明确交互系统时按上表场景匹配。

### 2.3 C 列：功能用户需求

🔴 强约束。
- 取值：同一功能模块下统一填写二级需求名称
- 来源：1.2 节需求矩阵二级需求、2.3 节功能实现一级模块
- 约束：与原文完全一致

### 2.4 D 列：触发事件

🔴 强约束。**优先从原文提取**触发动作（Web/APP 端点击类保留原文表述如"点击查询"）

**兜底 4 类**（按顺序匹配）：
1. 流程自动流转
2. 接口调用
3. 人工操作
4. 定时任务

> 仅允许原文提取值 + 4 类标准化值。

### 2.5 E 列：功能过程

🔴 强约束。
- 取值：同一三级功能点下统一填写，取三级功能点名称
- 来源：2.3 节功能实现三级功能点
- 约束：不得合并/拆分

### 2.6 F 列：子过程描述

🔴 强约束。详见 `cosmic-rules.md` 第 2.5 节。

### 2.7 G 列：数据移动类型

🔴 强约束。详见 `cosmic-rules.md` 第 3.1 节。

### 2.8 H 列：数据组

🔴 强约束。
- 取值：核心业务数据对象名称
- 命名模式：`XX信息` / `XX数据` / `XX表` / `XX结果` / `XX输出`
- 长度 ≤ 30
- 全表不重复，同功能过程不重复
- 命名源自原文

### 2.9 I 列：数据属性

🔴 强约束。
- 取值：100% 来自原文的字段集合
- 多字段用顿号分隔
- 加专属修饰语（格式：`修饰语+字段` 或 `字段（修饰语）`）
- 长度 ≤ 30
- 全表不重复，同功能过程不重复
- 优先穷尽原文字段，超限时按核心字段优先删减

### 2.10 J 列：复用度

🔴 强约束。固定取值：新增 / 复用 / 利旧。

### 2.11 K 列：CFP

🔴 强约束。固定映射：
- 新增 → 1.00
- 复用 → 0.33
- 利旧 → 0.00

### 2.12 L 列：ΣCFP

🔴 强约束。
- 全表 CFP 累加总和
- 仅首行业务数据行（第 4 行）填写
- 其余业务数据行 L 列留空

## 3. 数据移动类型映射规则详解

### 3.1 判定算法（伪代码）

```python
def classify_data_movement(sub_process_desc: str) -> str:
    """
    判定子过程描述对应的数据移动类型 G
    """
    keywords_E = ['解析入参', '获取入参', '接收推送', '调用接口入参']
    keywords_R = ['通过.*查询', '读取', '解析.*结果', '判断']  # 单独"判断"需改写
    keywords_W = ['保存', '写入', '更新', '记录日志', '生成']
    keywords_X = ['反馈', '返回', '推送', '发送', '调用.*接口']

    # 按优先级判定
    if any(kw in sub_process_desc for kw in keywords_E):
        return 'E'
    elif any(kw in sub_process_desc for kw in keywords_R):
        return 'R'
    elif any(kw in sub_process_desc for kw in keywords_W):
        return 'W'
    elif any(kw in sub_process_desc for kw in keywords_X):
        return 'X'
    else:
        raise ValueError(f"无法判定数据移动类型: {sub_process_desc}")
```

### 3.2 数据移动序列约束验证

```python
def validate_data_movement_sequence(process_id, g_column_sequence):
    """
    验证同一三级功能点下的 G 列序列是否合规
    """
    assert g_column_sequence[0] == 'E', f"功能过程 {process_id} 首行必须为 E"
    assert g_column_sequence[-1] in ('W', 'X'), f"功能过程 {process_id} 末行必须为 W 或 X"
    assert g_column_sequence.count('E') == 1, f"功能过程 {process_id} 仅允许 1 个 E"
    assert len(g_column_sequence) <= 8, f"功能过程 {process_id} 行数 ≤ 8"
    assert all(g in ('E', 'R', 'W', 'X') for g in g_column_sequence)
    assert all(g in ('R', 'W') for g in g_column_sequence[1:-1]), "中间行仅允许 R/W"
```

## 4. 功能过程识别与边界划分详解

### 4.1 三级层次模型

```
一级需求（核心需求全称）   →  A 列
  └── 二级需求（功能模块）   →  C 列
        └── 三级功能点        →  E 列（功能过程）
              └── 子过程      →  F 列
                    └── 数据移动 →  G 列
```

### 4.2 功能过程识别算法

```python
def identify_functional_processes(doc):
    """
    从设计文档识别功能过程
    """
    # Step 1: 提取 1.2 节需求矩阵
    req_matrix = extract_requirements_matrix(doc)

    # Step 2: 提取 2.3 节功能描述
    func_desc = extract_function_descriptions(doc, section="2.3")

    # Step 3: 一一对应校验
    matrix_3level = set(req_matrix['三级需求'])
    doc_3level = set(func_desc.keys())
    assert matrix_3level == doc_3level, f"需求矩阵与功能实现不匹配: 差集={matrix_3level ^ doc_3level}"

    # Step 4: 构建功能过程列表
    processes = []
    for level3_name in matrix_3level:
        proc = {
            'level1': req_matrix['一级需求'],
            'level2': req_matrix['二级需求'][level3_name],
            'level3': level3_name,
            'actions': func_desc[level3_name],
        }
        processes.append(proc)

    return processes
```

### 4.3 功能过程边界划分规则

🔴 强约束。

| 划分依据 | 判定 | 示例 |
|:---------|:-----|:-----|
| 不同的二级需求 | 必须分为不同功能过程 | 「新增扩容工单装维退单流程」与「优化装机工单装维退前台流程」不可合并 |
| 不同的三级功能点 | 必须分为不同功能过程 | 「查询扩容工单装维退单原因」与「装维发起扩容工单装维退单流程」不可合并 |
| 不同的触发事件 | 可作为子功能过程拆分的参考 | 同一三级功能点内"接口调用"与"定时任务"可拆分为子功能过程（一）/（二） |
| 不同的功能用户 | 触发"功能用户"列变更，必须列为新 B 列值 | 发起者从"装维APP"切换为"流程引擎服务" |
| 不同的业务阶段 | 可作为子功能过程拆分的参考 | 同一三级功能点内"参数获取"与"业务执行"可拆分 |

def self_check(xlsx_path, source_docx_path):

        ('check_step1_yuanwen', check_step1_yuanwen),         # 原文一致性
        ('check_step2_jiegou', check_step2_jiegou),           # 结构合规性
