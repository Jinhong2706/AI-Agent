# 第一部分 通用规范

## 1.1 核心总则

> ⚠️ 违反任意 1 条，直接判定 COSMIC 表格不合格，必须重拆。

- 🔴 **规则 1.1.1 源数据唯一原则**：COSMIC 表格所有内容必须完全来自设计文档原文，禁止新增设计文档未提及的需求、功能、字段、系统；禁止修改原文表述、禁止过度解读。
- 🔴 **规则 1.1.2 一一对应原则**：表格中每一行业务数据，必须在设计文档的「功能实现/详细设计」章节有唯一对应的原文描述，无原文支撑的行禁止新增。
- 🔴 **规则 1.1.3 固定结构优先原则**：表格前 3 行固定格式不可修改、不可删减、不可调整列顺序，严格遵循本规则定义的标准结构（见 2.1）。
- 🔴 **规则 1.1.4 度量统一原则**：严格遵循文档内固化的 COSMIC 度量规则，数据移动类型、复用度、CFP 值仅可使用规则内给定的固定取值，禁止自定义。
- 🔴 **规则 1.1.5 约束刚性执行原则**：严格遵守字段长度、风险词、禁用过程、数据移动类型顺序、复用度占比 5 项强制规则。
- 🔴 **规则 1.1.6 合并统一原则**：A~E 列内容完全相同的相邻行垂直合并，仅首行保留内容。
- 🔴 **规则 1.1.7 字段唯一性强制规则**：H 数据组、I 数据属性**全文档内绝对唯一**，同名字段/对象必须添加**标准化场景定语**；全表不重复、同功能过程不重复，命名完全源自子过程描述。
- 🔴 **规则 1.1.8 子过程描述完整原则**：子过程描述**禁止过度精简、禁止概括省略、必须逐行还原所有独立业务动作**，必须完整还原原文步骤的**动作 + 操作对象 + 业务目的**，确保语义通顺、流程可追溯。

## 1.2 分点强制拆分原则

### 1.2.1 🔴 必须逐拆的场景（满足任意 1 条即拆为独立行）

- 原文用数字序号（1/2/3…）分点的操作/判定/场景
- 原文用顿号、分号、"且/或"连接的平行条件
- 原文多业务场景分支（如场景 1/2/3）
- 原文多维度查询（按地址/按订单号/按账号）
- 原文多步骤写操作（生成+保存+更新）

### 1.2.2 🔴 绝对禁止的合并

- 禁止把"场景 1 查 A、场景 2 查 B"合并为"按场景查询"
- 禁止把"判断 A、判断 B"合并为"判断条件"
- 禁止把"读取 X、读取 Y"合并为"读取信息"
- 漏拆任意 1 个分点/分支，直接判定不合格

### 1.2.3 🔴 行数溢出处理

若按原文拆分超过 8 行，需将原三级功能点拆为两个子功能过程（名称后追加序号区分），各自满足 E→…→W/X 完整流程且各自 ≤ 8 行。

## 1.3 适用范围与不适用场景

- 🟢 **适用**：需求设计说明书/概要设计说明书/详细设计说明书已明确三级功能点的中小型软件项目
- 🟢 **不适用**：
  - 设计文档未列出功能点/业务流程，仅有架构描述
  - 设计文档为非结构化纯文本/PPT/图片
  - 仅需快速估算 CFP 总量而不需要表格

# 第三部分 计算逻辑

## 3.1 数据移动类型 G 列判定算法

### 3.1.1 数据移动类型定义

| 类型 | 简称 | 判定公式 | 典型动作关键词 | 出现位置约束 |
|:----:|:-----|:---------|:---------------|:-------------|
| **E** | Entry（输入） | 数据从系统边界外 → 系统边界内 | 解析、获取、接收、推送入参、调用接口、读取入参 | **必须是每个三级功能点的首行**，且每个三级功能点下**仅允许出现 1 次** |
| **R** | Read（读） | 从系统内部持久化存储读取数据 | 通过…查询、读取、判断（需改写为"读取并判断"）、校验、解析已落库数据 | 中间行，可多次 |
| **W** | Write（写） | 向系统内部持久化存储写入数据 | 保存、生成、更新、记录日志、写入 | 末行或中间行 |
| **X** | eXit（输出） | 从系统边界内 → 系统边界外 | 反馈、返回、推送、调用外部接口、页面展示、发送 | 末行（**必须 W 或 X 结尾**） |

### 3.1.2 判定算法

```
对每个候选子过程描述（含动作关键词）：
  1) 若包含「解析/获取/接收/调用入参/外部推送入参」 → G=E
  2) 否则若包含「通过 XX 查询/读取/校验」 → G=R
  3) 否则若包含「保存/写入/更新/记录/生成 + 保存」 → G=W
  4) 否则若包含「反馈/返回/推送/发送/调用 XX 系统接口」 → G=X
  5) 多关键词冲突时按"动作目的"判定
     （如"解析 IVR 接口出参"虽含"解析"，但目的是读取落库结果，判 R）
  6) 同一三级功能点下，G 列序列必须满足：
        ① 首行 = E
        ② 末行 ∈ {W, X}
        ③ 中间行 ∈ {R, W}
        ④ 整个序列行数 ≤ 8
```

### 3.1.3 🔴 强制约束清单

| 约束 | 规则 | 违规处理 |
|:-----|:-----|:---------|
| **E 开头** | 每个三级功能点的第 1 行 G 列必须 = E | 行数超标时按 3.2 拆分为子功能过程 |
| **E 唯一** | 同一三级功能点下，E 动作仅允许首行出现 1 次 | 后台自动流程的"入参获取/数据拉取"必须为首行 |
| **W/X 结尾** | 每个三级功能点的最后 1 行 G 列必须 = W 或 X | 不允许以 R 结尾 |
| **行数 ≤ 8** | 同一三级功能点下的子过程行数 ≤ 8 | 超限时拆分为两个子功能过程（名称后追加序号 1/2） |
| **不允许类型** | F 列子过程描述中**禁止单独出现**：封装、加密、redis 缓存、持久化数据 | 必须改写为等价业务动作 |

## 3.2 行数超限拆分规则

🟡 建议约束。当某三级功能点的子过程候选 > 8 行时，按业务阶段拆分为两个子功能过程。

**拆分原则**：
- 拆分位置：选在"业务阶段自然分界点"（如接口调用前后、定时任务执行前后）
- 命名规则：原三级功能点名称后追加 `（一）` `（二）` 或 `1` `2`
- 数据移动序列约束：拆分后每个子功能过程各自满足 E→…→W/X，**首行仍为 E**
- 业务来源：仍对应原三级功能点的需求范围，**不得新增需求**

## 3.3 复用度配比规划算法

🔴 强约束。算法如下：

```
输入：业务数据总行数 N
计算：
  新增行数 = round(N * 0.2)        # 20% 占比，下限 ceil(N×0.2)，上限 floor(N×0.25)
  利旧行数 = round(N * 0.1)        # 10% 占比
  复用行数 = N - 新增 - 利旧        # 70% 占比
校验：
  assert 0.20 <= 新增/N <= 0.25
  assert 新增:复用:利旧 ≈ 2:7:1
分配原则（按业务重要性）：
  - 「保存/写入/生成」类 W 动作优先分配「新增」
  - 「调用既有接口/获取入参」类 E / X 动作优先分配「利旧」
  - 其余 R / 中间 W 动作分配「复用」
```

## 3.4 ΣCFP 计算规则

🔴 强约束。

- L 列 = SUM(K 列) = ΣCFP
- 仅第 1 条业务数据行（第 4 行）填写 L 列
- 其余业务数据行 L 列留空
- 计算结果保留 2 位小数

## 3.5 字段唯一性校验算法

🔴 强约束。

```
H 列全表去重 → 任意 2 行 H 值相同 → 违规
I 列全表去重 → 任意 2 行 I 值相同 → 违规
H 列同功能过程（E 列相同）内去重 → 重复 → 违规
I 列同功能过程（E 列相同）内去重 → 重复 → 违规
```

修复方法：按 2.7.3 优先级添加专属定语。

## 3.6 字符长度校验算法

🔴 强约束。

```
对 F / H / I 列每个单元格：
  长度 = LEN(去除所有空格后的内容)  # 含中文/全角空格
  若 长度 > 30 → 该行不合格
```

# 第四部分 特殊场景处理

## 4.1 批量操作

🔴 强约束。

| 场景 | 处理方式 | F 列描述模板 |
|:-----|:---------|:-------------|
| 单次接口推送 N 条数据 | 视为**单次 X 输出**，不重复计数 | `批量反馈【对象】至【对端】` |
| 批量查询后逐条处理 | 拆分为：`批量查询（R）` + `逐条处理（W）` | `批量查询【对象】获取【N】条记录` |
| 批量保存多条记录 | 视为**单次 W 写入**，不重复计数 | `批量保存【对象】至【表】` |
| 批量读取后写入 | `批量读取（R）` + `批量写入（W）` | 按动作拆分 |

## 4.2 复用组件 / 公共逻辑

🔴 强约束。

| 复用程度 | 复用度取值 | CFP | F 列子过程描述模板 |
|:---------|:------------|:----|:-------------------|
| **新增**（全新开发） | `新增` | 1.00 | 完整描述实现逻辑（`根据【依据】生成【对象】并保存至【表】`） |
| **复用**（调用基础公共逻辑 + 少量定制） | `复用` | 0.33 | `调用【公共组件】处理【业务对象】` |
| **利旧**（完全使用既有逻辑，无开发量） | `利旧` | 0.00 | `调用【既有接口】获取【对象】` |

## 4.3 接口调用类

🔴 强约束。

| 接口方向 | G 列 | F 列模板 |
|:---------|:-----|:---------|
| 接收外部入参 | E | `解析【接口名】入参` |
| 推送至外部 | X | `调用【对端系统】【接口名】推送【对象】` |
| 解析外部出参 | R | `解析【接口名】出参获取【对象】` |

## 4.4 定时任务类

🔴 强约束。

- D 列 = `定时任务`
- B 列 = `发起者：流程引擎服务；接收者：业务服务接口`
- **首行强制 E**：`获取定时任务配置参数`
- 末行强制 W：`记录定时任务执行日志` 或 `更新待执行状态`

## 4.5 流程自动触发类

🔴 强约束。

- D 列 = `流程自动流转`
- B 列 = `发起者：流程引擎服务；接收者：业务服务接口`
- **首行强制 E**：`获取流程触发的当前【对象】id`
- 末行强制 W：`更新当前【对象】流程实例状态`

## 4.6 日志记录类

🔴 强约束。

- 视为**单次 W 写入**（不重复计数）
- F 列模板：`保存【对象】操作轨迹日志` / `记录【对象】执行日志`
- H 列：必须加专属定语，如 `退单发起操作轨迹日志数据`
- 同一三级功能点下，**最多 1 条**日志类 W 动作

## 4.7 解析入参类

🔴 强约束。

- 视为**单次 E 输入**（不重复计数）
- F 列模板：`解析【接口名】入参`
- **禁止**将"解析入参"与"通过入参查询"合并为一行，必须拆分为 2 行

## 4.8 字段唯一性冲突场景

🔴 强约束。按 2.7.3 优先级添加专属定语。

## 4.9 空字段处理

🟢 参考建议。

- 设计文档中无内容的功能模块 → **不生成任何业务数据行**
- 单个三级功能点下若功能描述为空 → 整行功能过程**不下发**，跳过处理

## 4.10 行数超限的拆分处理

🟡 建议约束。详见 3.2。

## 4.11 禁用过程与改写规则

🔴 强约束。

| 禁用原词 | 改写为 | 改写理由 |
|:---------|:-------|:---------|
| 单独「判断」 | `读取【对象】并判断【条件】是否满足` | 「判断」必须有 R 动作伴随 |
| 「封装」 | `生成【对象】并保存至【表】` / `反馈【对象】至【对端】` | 「封装」非业务动作 |
| 「加密」 | 通常需要改写为业务等效动作 | 加密属技术实现，不计 CFP |
| 「redis 缓存」 | 改写为对核心业务表的 R / W | 缓存层不计入功能点 |
| 「持久化数据」 | `保存【对象】至【表】` | 同「封装」处理 |

# 附录 B：常见错误反例库

| # | 错误类型 | 错误示例 | 正确示例 |
|:-:|:---------|:---------|:---------|
| 1 | I 列无定语导致重复 | `装机工单号`、`工程师工号` | `装机预约解析_装机工单号`、`装机预约查询_工程师工号` |
| 2 | 子过程合并分点 | `解析工单号并查询工程师信息` | 拆 2 行：`解析工单号` + `查询工程师信息` |
| 3 | 单独出现「判断」 | `判断审批条件是否满足` | `读取审批记录并判断审批条件是否满足` |
| 4 | 数据组命名无定语 | `工单信息`、`短信信息` | `改约工单查询信息`、`预约提醒短信生成信息` |
| 5 | 首行非 E | 首行直接 R/W | 强制首行 E（入参获取） |
| 6 | 末行非 W/X | 末行为 R | 强制末行 W 或 X |
| 7 | 行数 > 8 | 某三级功能点 9 行 | 拆分为 2 个子功能过程（一）（二） |
| 8 | 多 E | 同一三级功能点 2 个 E | 仅首行 E，后续用 R 替代"二次入参" |
| 9 | 复用度超占比 | 新增占比 35% | 控制在 20%~25% |
| 10 | 长度超限 | F 列 35 字符 | 简化到 30 字符内 |
| 11 | 字段过度精简 | 把 3 个动作合并为 1 行 | 逐行拆分每个独立动作 |
| 12 | 无依据新增 | 概设未提及"日志归档"却加上 | 严格禁止无原文依据的新增 |
