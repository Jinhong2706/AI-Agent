# generate_image 图片生成&编辑技能

## Purpose
为 Agent 提供标准化、可约束的图像生成与编辑能力，严格区分文生图 / 图生图场景，**在多轮对话中精准定位用户要修改的目标图片**，保证生图效果与用户意图完全一致。

## Core Rules & Constraints

### 场景分类与行为要求
| 场景 | 定义 | Agent 必须遵守 |
|------|------|----------------|
| **文生图** | 无参考图，纯文本生成 | 1. reference_images = []<br>2. enable_revise = true |
| **图生图** | 基于参考图修改/风格化 | 1. **禁止解析图片内容**<br>2. **禁止扩写/改写用户指令**<br>3. prompt 必须原样透传用户原话<br>4. enable_revise = false<br>5. **必须携带正确的目标图片URL**<br>6. **style 参数：仅在用户明确要求风格转换时使用** |

### 图生图要求
- 不准自己看图描述内容
- 不准把"图片里有什么"加到 prompt
- 不准优化、润色、扩展用户指令
- 必须直接透传：**正确的目标图片 URL** + 用户原始指令

## style 参数使用规则

### 何时使用 style 参数
- **必须使用**：当用户明确要求特定风格时（如"换成吉卜力风格"、"用美式卡通风格"）
- **禁止使用**：当用户没有明确要求风格时，或者只是描述图片内容（如"画一只猫"、"把这张图改成蓝色调"）
- **特别注意**：不要因为用户使用了类似"卡通"、"动漫"等词汇就自动添加 style 参数，只有当用户明确要求**列表中的特定风格关键词**时才使用

### 风格判断标准
```
✅ 正确使用：用户说"换成吉卜力风格" → style = "吉卜力"
✅ 正确使用：用户说"用美式卡通风格画" → style = "美式卡通"
❌ 错误使用：用户说"画一只卡通猫" → 不使用 style 参数
❌ 错误使用：用户说"把这张图改成卡通风格" → 不使用 style 参数（除非用户明确要求列表中的具体风格）
```

### 常见错误场景与纠正

#### style 参数错误使用示例
```
❌ 错误场景1：用户说"画一只卡通猫"
错误做法：style = "卡通"
正确做法：不使用 style 参数，prompt = "一只卡通猫"

❌ 错误场景2：用户说"把这张图改成动漫风格"
错误做法：style = "动漫"
正确做法：不使用 style 参数，prompt = "改成动漫风格"（除非用户明确要求"清透动漫"等具体风格）

✅ 正确场景：用户说"换成吉卜力风格"
正确做法：style = "吉卜力"，prompt = "换成吉卜力风格"
```

## image_size 参数使用规则

### 何时使用 image_size
- **必须使用**：当用户明确指定**画面比例**或**横/竖屏方向**时，例如：
  - "比例 16:9 / 16比9 / 9:16 / 4:3 / 3:4 / 1:1"
  - "横版 / 横屏 / 横图 / 宽屏"
  - "竖版 / 竖屏 / 竖图 / 手机壁纸"
  - "正方形 / 方图 / 头像比例"
- **不要使用**：用户没有提到比例 / 方向时，不要自作主张填这个参数（让后端走默认）。
- **严禁**：仅在 `prompt` 里写"比例为 16:9"等文字而不填 `image_size` —— 模型不会根据文本调整画布。

### 仅支持的尺寸白名单（必须严格匹配，不允许自由组合）

| 比例语义                | width | height | 适用场景                    |
|-------------------------|-------|--------|-----------------------------|
| 1:1（正方形 / 头像）    | 1024  | 1024   | "方图""正方形""头像比例"    |
| 4:3（横版常规）         | 1024  | 768    | "4:3""横版"                 |
| 4:3（横版稍大）         | 1152  | 864    | "4:3"且需要更大画幅         |
| 3:4（竖版常规）         | 768   | 1024   | "3:4""竖版"                 |
| 3:4（竖版稍大）         | 864   | 1152   | "3:4"且需要更大画幅         |
| **9:16（竖屏 / 手机）** | 768   | 1280   | **"9:16""竖屏""手机壁纸"**  |
| **16:9（横屏 / 宽屏）** | 1280  | 768    | **"16:9""横屏""宽屏"**      |

> 上述尺寸**唯一合法**。若用户提了不在表内的比例（如 21:9、2:3），就近映射到表中**最接近的同方向**比例；无法判定时退回 1024×1024。

### 用户表述 → 参数映射

| 用户原话                            | image_size                          |
|-------------------------------------|-------------------------------------|
| "比例 9:16" / "竖屏" / "手机壁纸"   | `{ "width": 768,  "height": 1280 }` |
| "比例 16:9" / "横屏" / "宽屏"       | `{ "width": 1280, "height": 768  }` |
| "4:3" / "横版"                      | `{ "width": 1024, "height": 768  }` |
| "3:4" / "竖版"                      | `{ "width": 768,  "height": 1024 }` |
| "正方形" / "方图" / "1:1"           | `{ "width": 1024, "height": 1024 }` |
| 用户未提比例                        | **不传 image_size**                 |

### image_size 常见错误
```
❌ 错误1：用户说"画一只小企鹅，比例为9:16"
错误做法：仅把 "比例为9:16" 拼进 prompt，不传 image_size
正确做法：prompt = "一只小企鹅"，image_size = { "width": 768, "height": 1280 }

❌ 错误2：方向搞反
错误做法：用户说 "16:9 横屏" → image_size = { "width": 768, "height": 1280 }
正确做法：横屏是宽>高 → { "width": 1280, "height": 768 }

❌ 错误3：自由组合非白名单尺寸
错误做法：用户说"16:9" → { "width": 1920, "height": 1080 }
正确做法：必须用白名单里的 { "width": 1280, "height": 768 }

❌ 错误4：用户没说比例硬塞参数
错误做法：用户说"画一只猫" → image_size = { "width": 1024, "height": 1024 }
正确做法：不传 image_size
```

#### 图片 URL 推理错误示例
```
✅ 正确场景：只要最后一轮用户有上传图片，那么一定使用这张图片的URL
总是使用当前对话轮次中最新上传或生成的图片

❌ 错误场景：用户在第1轮上传图片A，第2轮上传图片B，第3轮说"修改这张图"
错误做法：使用图片A的URL
正确做法：使用图片B的URL（最新上传的图片）

❌ 错误场景：多轮对话中图片URL混淆
错误做法：将不同对话轮次的图片URL混用
正确做法：建立轮次-图片映射，确保URL对应关系正确
```

## 图片 URL 推理与多轮对话管理

### 图片 URL 识别规则
1. **最新图片优先**：总是使用当前对话轮次中最新上传或生成的图片
2. **上下文关联**：通过对话上下文判断用户指的是哪张图片
3. **明确引用**：当用户说"这张图"、"刚才的图"时，引用最近一次处理的图片
4. **避免混淆**：不要将不同对话轮次的图片 URL 混淆使用

### 多轮对话图片引用规则（增强版）
- **全新文生图**：不传 reference_images 或传空数组
- **新上传图修改**：source = user_uploaded，使用最新上传的图片 URL
- **改 AI 生成的图**：source = ai_generated，使用最近一次生成的图片 URL
- **退回原图修改**：使用用户最初上传的图，通过对话历史确认原始图片
- **多图合并**：全部放入 reference_images 数组，确保 URL 正确对应
- **跨轮次追踪**：建立图片 URL 与对话轮次的映射关系，避免串用

### URL 正确性检查清单
- [ ] 确认图片 URL 来自当前对话轮次
- [ ] 检查 URL 是否完整有效（包含协议头）
- [ ] 验证图片 source 类型与实际情况一致
- [ ] 确保多轮对话中图片引用准确无误

## 关键要点总结

### style 参数使用核心原则
1. **明确要求原则**：只有当用户明确说出**具体风格名称**时才使用 style 参数
2. **禁止猜测原则**：不要根据用户描述中的风格相关词汇自动添加 style 参数
3. **列表匹配原则**：使用的风格必须严格匹配支持风格列表中的中文关键词

### 图片 URL 推理核心原则
1. **最新优先原则**：总是使用当前对话轮次中最新的图片 URL
2. **上下文关联原则**：通过对话上下文准确判断用户指的是哪张图片
3. **避免混淆原则**：建立清晰的图片-轮次映射，防止跨轮次 URL 混用

### image_size 参数使用核心原则
1. **比例触发原则**：仅当用户明确提到比例 / 横竖屏方向时才填 `image_size`，否则不传。
2. **白名单原则**：只能从 7 个支持尺寸里选，不允许自由组合宽高（如 1920×1080、512×768 等一律不行）。
3. **方向原则**：宽>高 = 横屏，高>宽 = 竖屏，相等 = 方图，禁止把方向搞反。
4. **prompt 与 image_size 解耦**：比例信息只走 `image_size` 字段，**不要**残留在 prompt 文本里（避免模型重复理解或冲突）。

### 多轮对话管理策略
- **建立图片时间线**：记录每轮对话的图片使用情况
- **关键词识别**：通过"这张"、"刚才"、"最新"等关键词判断目标图片
- **历史清理**：及时清理不再相关的旧图片引用

### 常见陷阱规避
- ❌ 陷阱：用户说"卡通风格"就自动添加 style = "卡通"
  ✅ 规避：只有当用户明确要求"美式卡通"等具体风格时才使用

- ❌ 陷阱：在多轮对话中混淆不同轮次的图片 URL
  ✅ 规避：建立轮次-图片映射，使用最新图片 URL

- ❌ 陷阱：图生图时解析图片内容并改写用户指令
  ✅ 规避：严格原样透传用户指令，不添加任何额外描述

- ❌ 陷阱：把"比例 16:9"写进 prompt 但不填 image_size，模型不会按文本改画布
  ✅ 规避：识别比例/方向词后映射到白名单，必须通过 image_size 字段下发

- ❌ 陷阱：9:16 与 16:9 方向搞反
  ✅ 规避：宽>高=横屏(16:9=1280x768)，高>宽=竖屏(9:16=768x1280)
