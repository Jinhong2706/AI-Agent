## 概述

skillhub-publish 是一个元 skill，用于将其他 WorkBuddy skill 准备好发布到 SkillHub。其核心设计原则是"先议后行"——所有写操作前都必须经过用户确认。

## 架构决策

### ADR-001：纯指令式实现，无 Python 脚本

**状态：** 已采纳

**背景：** skillhub-publish 需要对其他 skill 的文件做读/写/编辑操作，这些操作可以通过 WorkBuddy 内置的 Read/Write/Edit/Bash 工具完成。

**决策：** 所有审计、修复、生成逻辑全部写在 SKILL.md 的步骤描述中，由 agent 在对话中直接执行。不创建 Python 脚本。

**后果：**
- 正面：零依赖，零打包，其他用户安装后直接可用
- 正面：每一步都可审计，agent 执行过程中用户可以随时干预
- 负面：SKILL.md 较长，但三级加载机制可以缓解

### ADR-002：六维审计输出为结构化报告

**状态：** 已采纳

**背景：** 审计发现问题后需要向用户清晰展示，并支持用户逐项确认修复。

**决策：** 审计报告使用固定的报告格式（P0/P1/P2 分级 + 复选框），让用户一眼看清严重程度。

**后果：**
- 正面：用户不需要理解技术细节，按优先级做决策即可
- 正面：P0 不可跳过，确保安全红线
- 负面：报告格式固定，灵活性有限（可接受）

### ADR-003：不依赖其他 skill

**状态：** 已采纳

**背景：** 本 skill 自身要发布到 SkillHub，如果依赖其他 skill（如 skill-creator-king），其他用户安装后无法使用。

**决策：** 所有审计规则、模板、生成逻辑全部自包含于本 skill 的文件和 SKILL.md 中。

**后果：**
- 正面：用户只安装一个 skill 即可用
- 正面：审计维度 6（外部依赖）可以用来检查其他 skill 是否犯同样错误
- 负面：无法复用 skill-creator-king 的验证框架（但本 skill 与 SCK 定位不同，不构成问题）

### ADR-004：发布物料在 Phase 4 集中生成

**状态：** 已采纳

**背景：** README.md 的生成需要基于修复后的 skill 状态（修正后的 description、triggers 等），因此必须在修复完成后生成。

**决策：** Phase 3 修复完成后，Phase 4 集中生成所有发布物料（README.md + 两段描述文本），确保内容准确定稿。

**后果：**
- 正面：物料内容反映最新状态
- 正面：一次确认，统一输出，减少交互轮次
- 负面：如果需要回退，物料需要重新生成（但通常不会回退）

### ADR-005：README.md 的目标用户是安装者

**状态：** 已采纳

**背景：** 被发布的 skill 的 README.md 是写给 SkillHub 上安装该 skill 的其他用户看的，不是写给当前用户（觉者）看的。

**决策：** README.md 采用标准结构（用途/核心能力/使用方式/技术架构/环境要求/安全声明），语气面向第一次接触该 skill 的用户。

**后果：**
- 正面：任何安装者都能快速上手
- 正面：与 SKILL.md（面向 agent 的执行指令）形成互补

## 流程架构

```
用户输入目标 skill 名称
        │
        ▼
┌─────────────────────────────────────────┐
│ Phase 0: 出发前确认（Pre-flight）        │
│  出示完整机票 → 确认输出位置 → 用户签字  │
└──────────────────┬──────────────────────┘
        │ 签字确认
        ▼
┌─────────────────────────────────────────┐
│ Phase 1: 六维审计（只读）               │
│  ①文件结构  ②Frontmatter  ③内容质量    │
│  ④版本一致性  ⑤安全  ⑥外部依赖        │
│  路由：零问题→Phase 4，有问题→Phase 2  │
└──────────────────┬──────────────────────┘
        │ 审计报告
        ▼  （P0/P1/P2 分级，用户确认修复清单）
┌─────────────────────────────────────────┐
│ Phase 2: 讨论确认                       │
│  展示报告 → 逐项确认 → 用户点头         │
└──────────────────┬──────────────────────┘
        │ 确认清单
        ▼
┌─────────────────────────────────────────┐
│ Phase 2.5: 创建发布副本                 │
│  rsync/cp → 排除垃圾文件 → 用户数据骨架 │
│  → 清理缓存 → 按 .skillignore 追加清理  │
└──────────────────┬──────────────────────┘
        │ 桌面副本
        ▼
┌─────────────────────────────────────────┐
│ Phase 3: 在副本上执行修复               │
│  修 Frontmatter → 补文件 → 对齐版本     │
│  清理非发布物（仅副本，可选同步原文）    │
└──────────────────┬──────────────────────┘
        │ 修复完成
        ▼
┌─────────────────────────────────────────┐
│ Phase 3.5: 发布内容脱敏（仅副本）       │
│  3.5.0 规则自举（AI扫描，首次自动生成） │
│  3.5.1 私有规则优先 → 内置规则兜底      │
│  3.5.2 find→sed 逐条替换                │
│  3.5.3 替换报告 → 3.5.4 展示确认        │
│  └─ 反馈闭环：上架后报警 → 自动补规则   │
└──────────────────┬──────────────────────┘
        │ 脱敏完成
        ▼
┌─────────────────────────────────────────┐
│ Phase 4: 生成发布物料                   │
│  README.md → SkillHub描述 → 版本说明    │
└──────────────────┬──────────────────────┘
        │ 三样产出
        ▼
┌─────────────────────────────────────────┐
│ Phase 5: 最终交付                       │
│  终检（隐藏文件/空目录）→ 输出文本      │
└─────────────────────────────────────────┘
```

## 数据流

| 阶段 | 输入 | 处理 | 输出 |
|------|------|------|------|
| Phase 0 | 用户输入 | 出示机票 + 确认 | 用户签字 |
| Phase 1 | 目标 skill 目录路径 | 逐文件读取 + 正则解析 | 审计报告（问题清单） |
| Phase 2 | 审计报告 | 用户逐项确认 | 修复清单 |
| Phase 3 | 修复清单 | 文件写/编辑操作 | 修复后的文件 |
| Phase 4 | 修复后的文件 | 内容提取 + 模板填充 | README.md / 描述文本 / 版本说明 |
| Phase 5 | 以上所有 | 目录复制 + 文本输出 | 桌面发布包 + 对话输出 |
