# 落盘决策表 (Landing Rules)

本表用于 Block 3「个性化设置和使用习惯」，将用户提出的痛点映射到具体的落盘文件、章节和写入策略。

## 使用方法

1. 用户提出一个痛点后，识别其所属的「痛点类型」
2. 在下表中找到对应行，获取推荐落盘方案
3. 如痛点不在表中，使用「Fallback 策略」进行最近邻匹配
4. 向用户确认落盘内容后，调用 `write` 工具写入

## Fallback 策略

当用户提出的痛点不在上表中时，使用以下决策流程：

1. **判断痛点本质**：属于「我怎么说话」→ SOUL.md；「我怎么做事」→ AGENTS.md；「我是谁」→ IDENTITY.md；「我服务谁」→ USER.md
2. **确定章节**：在目标文件中找到语义最接近的已有章节，或新建一个以用户语言命名的章节
3. **判断是否需要硬性 Rule**：问自己——「如果 Agent 偶尔违反这条规范，用户会不会很生气？」如果是 → 加硬性 Rule；否 → 只写规范
4. **向用户确认**：「我帮你这样记下来：[落盘文案]，你看行吗？」
5. **确认后写入**

## 章节命名原则

- 决策表中的章节名为**推荐**而非强制
- 优先使用用户自己的语言来命名章节（如用户说"你回消息太慢"，章节可以叫 `## 回复节奏` 而非 `## 消息回复规范`）
- 同一文件中避免语义重叠的章节（如已有 `## 消息格式规范` 就不要再建 `## 输出格式规范`，合并即可）
- 章节标题用中文，保持简洁

## 硬性 Rule 追加规范

当需要追加硬性 Rule 时，在 AGENTS.md 中使用以下格式：

```markdown
## [章节标题]

[具体规范内容]

> ⚠️ 硬性 Rule：在生成任何回复前，必须核对本章节内容，确保输出符合规范。
```

**何时加硬性 Rule**：
- 用户明确表示「每次都要」「千万别忘」「这是底线」
- 涉及格式/结构类约束，Agent 容易忽略的（如链接格式、进度提示）
- 涉及安全/确认类约束（如操作前先问）

**何时不加**：
- 语气风格类偏好（Agent 自然就会遵循写在 SOUL.md 里的风格，不需要 Rule 约束）
- 用户说「尽量」「偏好」「一般」等弹性表达
- 偏好本身是模糊的（如"说话自然一点"）
